## Контекст

| Параметр | Значение |
|---|---|
| Пользователей | ~200 клиентов/мес + ~20 менеджеров + 2–3 админа |
| Фреймворк | Next.js App Router (RSC + Client Components) |
| UI | shadcn/ui + Tailwind CSS v4 |
| Авторизация | Better Auth (роли: client, manager, admin) |
| Данные | Server Components → DAL (Drizzle ORM) напрямую |
| Формы | React Hook Form + zod |
| Таблицы | @tanstack/react-table (shadcn DataTable) |
| Состояние | Нет глобального стейт-менеджера — RSC + URL search params |

### Принцип: Server-first

- **Server Components** для чтения данных (списки, карточки) — вызывают DAL напрямую, без HTTP
- **Client Components** только для интерактивности (формы, модалки, фильтры)
- **Server Actions** для мутаций (создание заявки, загрузка документов)
- URL search params для фильтров и пагинации (shareable, работает без JS)

## Фаза 2: Дашборд клиента
**Статус:** [ ]
**Исполнитель:** Агент
**Зависимость:** Фаза 1

### Задачи

**2.1. Страница `(dashboard)/page.tsx` — обзор**

KPI-карточки (верхний ряд):
- Активных заявок: N
- Действующих пропусков: N
- Истекает в ближайшие 30 дней: N
- Ожидает оплату: N

Быстрые действия:
- «Новая заявка» (кнопка → `/dashboard/orders/new`)
- «Загрузить документы» (если есть pending)

Последние уведомления (3–5 шт):
- Тип + текст + время
- Ссылка на полный список

**2.2. Список заявок `(dashboard)/orders/page.tsx`**

- DataTable (shadcn): номер, зона, тип, статус (badge), дата, действия
- Фильтры: статус, зона (через URL search params)
- Поиск по номеру заявки
- Пагинация (cursor-based)
- Кнопка «Новая заявка»
- Статусы — цветные badges из дизайн-системы

**2.3. Создание заявки `(dashboard)/orders/new/page.tsx`**

Мультишаговая форма (stepper):

```
Шаг 1: Выбор зоны и типа пропуска
  → [МКАД] [ТТК] [Садовое кольцо]
  → [Годовой] [Временный] [Ночной] [Разовый]
  → Желаемые даты (от–до)

Шаг 2: Выбор ТС (из сохранённых или добавить новое)
  → Список сохранённых ТС с чекбоксами
  → «+ Добавить транспортное средство»

Шаг 3: Маршрут
  → Откуда (input с подсказками)
  → Куда (input с подсказками)
  → Промежуточные точки (опционально)

Шаг 4: Загрузка документов
  → Чеклист необходимых документов (зависит от типа)
  → Drag & drop зона для каждого документа
  → Показывать уже загруженные (зелёная галочка)

Шаг 5: Проверка и отправка
  → Сводка: зона, тип, ТС, маршрут, документы
  → Стоимость (калькулятор из API)
  → Кнопка «Отправить заявку»
```

**2.4. Карточка заявки `(dashboard)/orders/[id]/page.tsx`**

- Статус с прогресс-баром (текущий шаг в воркфлоу)
- Детали: зона, тип, ТС, маршрут, даты
- Блок документов: загруженные ✅, недостающие ❌ (с кнопкой «Загрузить»)
- Блок оплаты: статус, сумма, кнопка «Оплатить» (если payment_pending)
- Блок пропуска: номер, срок действия (если approved)
- История: timeline изменений статуса с датами
- Комментарии / чат с менеджером (опционально)

**Критерий готовности:** Клиент может создать заявку, загрузить документы, оплатить, видеть статус.

## Фаза 4: Платежи
**Статус:** [ ]
**Исполнитель:** Агент
**Зависимость:** [Бэкенд и API](2026-04-10-бэкенд-и-api.md) Фаза 6

### Задачи

**4.1. Кнопка «Оплатить» в карточке заявки**

При нажатии:
1. Вызов `POST /api/payments/create` (orderId, amount)
2. Получение `confirmation_url` от ЮKassa
3. Редирект клиента на страницу оплаты ЮKassa
4. После оплаты → возврат на `/dashboard/orders/[id]?payment=success`

**4.2. Выбор способа оплаты**

Модалка перед оплатой:
- Банковская карта (по умолчанию)
- СБП (QR-код)
- SberPay
- T-Pay
→ Передаётся как `payment_method` в API

**4.3. Страница результата оплаты**

`/dashboard/payments/result?payment_id=...`:
- Проверка статуса через API
- Успех: «Оплата прошла! ✅ Менеджер приступит к оформлению»
- Отмена: «Оплата не завершена. Попробовать снова?»

**4.4. История платежей `(dashboard)/payments/page.tsx`**

- Таблица: дата, сумма, статус, заявка, способ оплаты
- Ссылка на чек (receipt_url)

**Критерий готовности:** Клиент может оплатить заявку через ЮKassa, видеть историю платежей.
