# 2026-04-11 — Фикс регистрации + фичи кабинетов

## Контекст

Пользователь сообщил: «не получается зарегистрироваться». Параллельно
запросил:

1. Сделать регистрацию в кабинет доступной (можно в тестовом режиме —
   email/SMS подключим позже).
2. Убрать «Название компании» из формы — заполнится позже, при первых
   финансовых отношениях.
3. Сделать админку — все клиенты, аналитика по деньгам, машинам, времени,
   активности.
4. Добавить «крутые фичи» во все кабинеты (клиент / партнёр / админ),
   которые специалисты с опытом давно реализовали.

## Анализ корневой причины поломки регистрации

После исследования кода найдены **две причины**, любая из которых ломает
регистрацию (а вместе — стопроцентно):

### 1. Мёртвый редирект `/auth/login` (главная причина 404 после регистрации)

В 13 файлах `apps/web/src/app/dashboard/**` и `app/admin/layout.tsx`
вызывался `redirect("/auth/login")`. Но реальный путь формы логина —
`/login` (роут лежит в route group `(auth)`, которая в URL не пишется).

В результате после `signUp.email()` пользователя направляло на
`/dashboard` → layout проверял сессию → если что-то с куками не так,
делал `redirect("/auth/login")` → 404. Пользователь видел «не могу
зарегистрироваться».

**Фикс:** заменено `/auth/login` → `/login` во всех 13 файлах.

### 2. Регистрация клиента не сохраняет phone

Better Auth сохраняет только `email`, `password`, `name`. Поля `phone` и
`company` собирались формой, но в `authClient.signUp.email()` не
передавались — терялись. Партнёрская регистрация решает это
двухэтапно: `signUp.email()` → server action `becomePartnerAction`,
который доливает phone/company/inn в `users`.

**Фикс:** создан `apps/web/src/app/(auth)/_actions.ts` с server action
`completeClientProfileAction({ phone })`. Форма
`apps/web/src/app/(auth)/register/page.tsx` переписана под двухэтапный
паттерн (по аналогии с партнёрской), поле «Компания» удалено.

### 3. Better Auth — явная конфигурация тестового режима

Раньше `emailAndPassword: { enabled: true }` без явных опций.
Добавлены: `requireEmailVerification: false`, `autoSignIn: true`,
`minPasswordLength: 8`. Это обозначает «тестовый режим — email/SMS
подключим позже».

## Что сделано в админке

Ядро уже было: `getDashboardStats`, `getRecentOrders`, `getAdminClients`,
`getClientDetail`, `getAnalyticsData`, экспорт CSV/Excel и т. д.
Не хватало бизнес-инсайтов про клиентов — добавлено.

### Сегментация клиентов на тиры T1–T5

Новая функция `classifyClientTier(totalRevenue)` в `lib/dal/admin.ts`:

| Тир | Метка | Порог LTV |
|-----|-------|-----------|
| T5 | VIP | ≥ 500 000 ₽ |
| T4 | Постоянный | ≥ 150 000 ₽ |
| T3 | Активный | ≥ 50 000 ₽ |
| T2 | Развивающийся | ≥ 10 000 ₽ |
| T1 | Новичок | < 10 000 ₽ |

Тир рассчитывается на лету по сумме успешных платежей. Когда придёт
большая статистика, пороги легко перекрутить в одной функции.

### Расширенный `getClientDetail` с insights

В возвращаемый объект добавлен блок `insights`:

- `tier` — тир (T1–T5) с меткой и описанием
- `lastLoginAt` / `daysSinceLastLogin` — последний вход (из `sessions`)
- `lastOrderAt` / `daysSinceLastOrder`
- `successRate` — % одобренных заявок от завершённых
- `avgCycleHours` — среднее время от создания до завершения заявки
- `riskScore` (0–3) — клиенты под угрозой ухода:
  - +1 если нет заказов > 60 дней
  - +1 если не заходил > 30 дней
  - +1 если есть зависшие заявки (`documents_pending` / `payment_pending`)
- `stuckOrdersCount`

### Расширенная карточка клиента `/admin/clients/[id]`

- В шапке: badge тира + badge риска (Норма / Внимание / Под угрозой ухода)
- Кнопки быстрых действий: «Позвонить» (`tel:`) и «Написать» (`mailto:`)
- Сетка из 4 insight-карточек: LTV, последний вход, последняя заявка,
  зависшие заявки
- В сайдбаре «Статистика» добавлены: одобрено, success rate, среднее
  время цикла

### Расширенный список клиентов `/admin/clients`

- Фильтр по тиру (`?tier=T5` и т. д.)
- Колонка тира с цветным badge
- Колонки «Посл. заказ» и «Посл. вход» в относительном формате (сегодня,
  вчера, N дн., N нед., N мес.)
- Subtitle с компанией (если заполнена) под именем клиента

В сериализованных данных (`getAdminClients` → `serializedClients`) добавлены
`tier`, `tierLabel`, `lastOrderAt`, `lastLoginAt`.

## Что сделано в клиентском ЛК

### Кнопка «Повторить заявку» (re-order)

Самая частая операция транспортной компании — оформить тот же тип
пропуска для той же машины. Раньше требовалось пройти 4 шага мастера
заново.

- `/dashboard/orders/new` принимает `?type=...&vehicleId=...` через
  searchParams
- `NewOrderForm` валидирует параметры (type должен быть из `passTypes`,
  vehicleId — из доступных пользователю ТС). Если оба валидны — сразу
  открывается шаг 3 (документы)
- На дашборде в Quick Actions появилась кнопка «Повторить последнюю»
- В таблице Recent Orders у каждой строки появилась иконка
  «Повторить заявку»

## Что НЕ сделано в этой сессии (на будущее)

- **Партнёрский кабинет на реальных данных** — сейчас работает на mock
  data в `partner/(portal)/page.tsx`. Подключение реальных метрик
  (комиссии, графики, выплаты) — отдельная большая задача.
- **Уведомления / напоминания клиентам** — Telegram / email о
  истекающих пропусках. Каркас в `notifications` есть, нужны воркеры.
- **Заметки менеджера про клиента** — нет таблицы `client_notes`. Если
  нужно — добавить в schema.ts + DAL + UI блок в карточке клиента.
- **Activity timeline в карточке клиента** — хронология всех действий
  (login / order / payment / chat). Часть данных уже есть в `auditLog`,
  нужен агрегатор.
- **Cohort retention / CAC / LTV** в `/admin/analytics` — пока есть
  monthlyRevenue, monthlyOrders, zoneStats, managerStats, topClients.
  Добавить retention-когорты можно SQL window-функциями.
- **Bulk-actions в списке клиентов** — массовая рассылка, перевод
  менеджера. Структура на месте, нужен UI.
