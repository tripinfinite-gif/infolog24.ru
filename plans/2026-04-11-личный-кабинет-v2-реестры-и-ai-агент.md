## 0. Зачем v2

V1 личного кабинета описывал классическую схему «заявки на пропуск + ТС + документы + платежи». Это был MVP уровня «электронная заявка вместо звонка менеджеру».

V2 переосмысляет ЛК как **операционную систему перевозчика** — единое окно, где клиент держит и видит **всё**, что нужно для жизни парка: реестры пропусков, ТС, водителей, страховок, диагностических карт, РНИС, штрафов, ЭТрН, ГосЛог-статуса, ремонтов, эвакуаций, юридических кейсов. Сверху — встроенный голосовой AI-агент **«ИнфоПилот»**, который читает данные клиента и знает наши продукты, законы и партнёрские сервисы. Бот в **MAX-мессенджере** (см. § 9.3) остаётся как канал нотификаций и опциональный второй интерфейс, но **первичный канал — наше приложение** (web + PWA + мобильное обёртывание Capacitor/React Native).

Это превращает ЛК из «формы заявки» в **барьер переключения**: чем больше данных клиент туда занёс, тем сложнее уйти к конкуренту. И превращает каждого клиента из «оформителя пропуска» в подписчика платформы с LTV в десятки раз выше.

### Метрики v2 (через 12 месяцев после запуска)

| Метрика | Цель |
|---|---|
| Доля заявок через ЛК (а не через менеджера) | ≥ 70% |
| Средняя глубина данных в ЛК (заполненные карточки ТС/водителей/страховок) | ≥ 60% полей |
| DAU/MAU клиентского приложения | ≥ 25% |
| Доля диалогов с AI-агентом, закрытых без эскалации на менеджера | ≥ 60% |
| Доля голосовых обращений к AI-агенту от мобильных пользователей | ≥ 30% |
| Доля клиентов с ≥ 2 активированными модулями (пропуска + один из: гослог/этрн/инфопилот/антиштраф) | ≥ 35% |
| NPS ЛК | ≥ 50 |

## 1. Категории клиентов (тиры) — кто какой кабинет видит

ЛК — **один на всех**, но контент адаптируется под профиль клиента: модули включаются/выключаются в зависимости от того, какие наши продукты он использует. Это типичный pattern feature gating + progressive disclosure — клиент не пугается лишних разделов, видит только то, что относится к его подписке.

### 1.1. Профиль клиента — поля сегментации

```ts
type ClientProfile = {
  // Базовое
  id: string
  type: 'individual' | 'ip' | 'ooo'   // физик / ИП / юрлицо
  inn?: string
  fleetSize: number                    // расчётное (число активных ТС в реестре)

  // Активные продукты (определяют видимые модули)
  modules: {
    permits: boolean        // Пропуска (всегда true для активных клиентов)
    goslog: boolean         // ГосЛог Старт — зарегистрирован в реестре
    etrn: boolean           // ЭТрН Переход — подключён к ЭДО
    antiShtraf: boolean     // Антиштраф — подписка на мониторинг
    parkPro: boolean        // Парк Про — подписка на управление парком
    yurist: boolean         // Юрист-перевозчик — юридическая защита
    infoPilot: boolean      // ИнфоПилот — SOS-подписка или PAYG
  }

  // Тариф (определяет лимиты)
  plan: 'free' | 'propusk_plus' | 'tranzit_moskva' | 'flot_pro' | 'corporate'

  // Роль внутри компании
  role: 'owner' | 'dispatcher' | 'accountant' | 'driver'
}
```

### 1.2. Тиры по нашей продуктовой линейке

| Тир | Какие продукты | Какие модули в ЛК | Объём данных |
|---|---|---|---|
| **T0 — Гость** | Нет аккаунта, посетитель сайта | Чат-консультант, калькулятор стоимости, FAQ | — |
| **T1 — Пропуска** | Только пропуска (разовые заявки) | Заявки, пропуска, ТС (мини), документы (мини), уведомления, AI-консультант | 1–5 ТС |
| **T2 — Гослог / ЭТрН** | Пропуска + ГосЛог Старт и/или ЭТрН Переход | T1 + раздел «Гослог-кейс» / «ЭТрН-подключение» с КПА-сессиями, статусом подачи, реестровой выпиской, журналом подписания УКЭП | 1–10 ТС |
| **T3 — Антиштраф** | Любой из T1/T2 + подписка Антиштраф | T1/T2 + «Здоровье парка»: РНИС-телеметрия, контроль визитов, неоплаченные штрафы, алерты, ленты инцидентов, отчёт здоровья | 5–50 ТС |
| **T4 — Парк Про** | Парк Про (10+ ТС) | T3 + полные реестры (ТС, водители, страховки, ДК, ОСАГО/КАСКО, тахографы), пакетная подача, аналитика парка, роли (диспетчер/бухгалтер), API, выделенный менеджер | 10–500+ ТС |
| **T5 — Корпорат** | Парк Про Корпорат + кастомизация | T4 + SLA, white-label, SSO, кастомные интеграции с ERP/TMS, multi-branch | 50+ ТС, многокомпанийность |

> Дополнительно (ортогонально тирам): любой клиент может включить ИнфоПилот (SOS-подписка или PAYG) и Юрист-перевозчик. Они добавляют разделы «Сервисы 24/7» и «Юр.кейсы» соответственно.

### 1.3. Прогрессия клиента

```
Гость → T1 (одна заявка) → T2 (накроет волна гослог/этрн)
                              ↓
                          → T3 (мониторинг — самый дешёвый upsell)
                              ↓
                          → T4 (вырос парк / захотел всё в одном)
                              ↓
                          → T5 (корпорат)
```

ЛК должен буквально показывать клиенту прогрессию: «У вас активны 2 из 6 модулей. Подключите Антиштраф — будете видеть здоровье ваших пропусков». CTA на upsell — нативно вшиты в дашборд.

### 1.4. Роли внутри компании клиента (T4+)

В Парк Про у одной компании может быть **несколько пользователей с разными ролями**:

| Роль | Что видит/может |
|---|---|
| `owner` | Всё: модули, реквизиты, тарификация, история платежей, юр.кейсы, удаление аккаунта |
| `dispatcher` | Заявки, пропуска, ТС, водители, маршруты, ИнфоПилот, инциденты. Не видит: финансы компании, тарификацию |
| `accountant` | Платежи, счета, акты, реестры закрывающих, экспорт в 1С, ЭДО. Не видит: водителей, инциденты |
| `driver` | Только свои ТС (закреплённые), свои документы, путевые листы, ИнфоПилот для своего рейса. Не видит: финансы, других водителей |
| `manager_infolog` | Сторона Инфологистик — наш менеджер, видит клиента «как клиент» + админ-инструменты |

Реализация — Better Auth roles + RBAC через middleware и DAL guards.

## 2. Главный сдвиг архитектуры данных: от «заявок» к «реестрам»

V1 ставил в центр модели **заявку на пропуск** (`order`). V2 ставит в центр **реестры** (registries) — постоянно живущие списки сущностей, к которым прикрепляются документы и события. Заявка на пропуск становится одним из видов **событий** над ТС.

### 2.1. Реестры верхнего уровня (домены данных)

Каждый реестр = отдельный раздел ЛК + отдельный набор таблиц БД + DAL-модуль + AI-tools (для агента).

| # | Реестр | Что хранит | Кто видит | Источник заполнения |
|---|---|---|---|---|
| **R1** | **Транспортные средства** | Карточки ТС (марка, ВИН, госномер, ПТС, СТС, экокласс, грузоподъёмность, категория, год) | T1+ | OCR СТС/ПТС, ручной ввод, импорт CSV/Excel, синхронизация с реестром ФНС по ИНН |
| **R2** | **Водители** | ВУ, мед.справка, психо.освидетельствование, тахо-карта, путевые листы, история ДТП | T4+ (T3 опционально) | OCR ВУ, импорт из 1С/ERP |
| **R3** | **Документы компании** | Уставные, ЕГРЮЛ, ОКВЭД, лицензии, доверенности, договоры, ИНН/КПП/ОГРН | T1+ | OCR карточки организации, DaData, ручной ввод |
| **R4** | **Страховки** | ОСАГО, КАСКО, ОСГОП (пассажиры), ОСГОП (грузы — экспедиторская ответственность), CMR, страхование жизни водителей | T1+ | OCR полиса, API АИС ОСАГО (при наличии), ручной ввод |
| **R5** | **Диагностика и ТО** | ДК (диагностические карты), история ТО, графики ТО (моточасы/пробег), сервисные книжки | T1+ | OCR ДК, ручной ввод, импорт из СТО (через ИнфоПилот) |
| **R6** | **РНИС / телематика** | Подключение к РНИС, статус передачи данных, журнал сбоев, точки погрузки, треки, моточасы | T3+ | API РНИС-операторов (СКАУТ, GLONASSSoft, OMNICOMM, Wialon), ручная сверка |
| **R7** | **Тахографы** | Карты водителя, карты предприятия, СКЗИ, протоколы калибровки, выгрузки DDD | T4+ | Загрузка DDD-файлов, OCR карт |
| **R8** | **Пропуска** | Все выданные пропуска с историей продлений, зон, ТС, статусов | T1+ | Наш бэкенд + Дептранс (через АРМ оператора) |
| **R9** | **Заявки и кейсы** | Активные/исторические заявки на пропуска, гослог-кейсы, этрн-кейсы, юр.кейсы, инциденты ИнфоПилота | T1+ | Внутренние, генерируются ЛК |
| **R10** | **Штрафы и постановления** | Все штрафы (АМПП, МАДИ, ГИБДД), статусы оплаты, обжалования, привязка к ТС/водителю/инциденту | T3+ | API ГИС ГМП, парсинг госуслуг через КПА, OCR постановлений, ручной ввод |
| **R11** | **ЭТрН и реестр перевозок** | Электронные ТТН, статус подписания, груз, маршрут, отправитель, получатель | T2 (этрн) + | API оператора ЭДО (Контур.Логистика, СберКорус, ЭДО.Поток), импорт из ЕГИС ОТБ |
| **R12** | **ГосЛог-реестр** | Запись в реестре экспедиторов/перевозчиков, статус, контрольные мероприятия | T2 (гослог) + | АРМ Госуслуг через КПА, СМЭВ-callback |
| **R13** | **Платон и платные дороги** | Маршрутные карты Платон, баланс счёта, лицевой счёт, БУ, история проездов | T3+ | API Платон, ручная синхронизация |
| **R14** | **Топливные карты и АЗС** | Номера карт (Газпромнефть, Лукойл, Роснефть), лимиты, история транзакций | T4+ | API провайдеров, импорт CSV |
| **R15** | **Финансы** | Платежи, счета, акты, чеки, закрывающие, лицевой счёт компании, дебиторка | T1+ | YooKassa, генерация PDF, наш биллинг |
| **R16** | **Маршруты и рейсы** | Текущие и плановые рейсы (откуда/куда/что/когда), привязка к ТС/водителю/ЭТрН/пропуску/Платон | T4+ (опционально T2 для увязки с ЭТрН) | Импорт ЭТрН, ручной ввод, API ATI.SU/биржи |
| **R17** | **Сервисные кейсы (ИнфоПилот)** | Эвакуации, СТО-визиты, мойки, ремонты, страховые случаи | T1+ (если активирован ИнфоПилот) | Заявка из ИнфоПилота |
| **R18** | **Юридические кейсы** | Обжалования штрафов, аннулирований, споры с Дептрансом | T1+ (если активирован Юрист-перевозчик) | Заявка из ЛК или AI-агента |
| **R19** | **База знаний и обучение** | Прочитанные материалы, пройденные курсы по новым правилам, отметки «ознакомлен» (важно для тренинга водителей и для аудита соответствия) | T1+ | Наш контент, события чтения |
| **R20** | **Уведомления** | Лента всех событий: статусы, алерты, дедлайны, сообщения от менеджера | T1+ | Внутреннее, события из всех модулей |

> Все эти реестры объединяются графом связей: ТС ↔ Водитель ↔ Страховка ↔ ДК ↔ РНИС ↔ Пропуск ↔ Маршрут ↔ ЭТрН ↔ Штраф ↔ Платежи. Это позволяет AI-агенту отвечать на запросы вроде: «Какие ТС у меня без действующего ОСАГО?» или «Покажи штрафы по А123БВ777 за март».

### 2.2. Дополнительно — что я бы добавил сверх запроса пользователя

Помимо явно перечисленного клиентом (ДК, ОСАГО, КАСКО, РНИС, штрафы, доки на авто), я считаю критически важным заложить:

1. **Реестр водителей (R2)** — без этого нельзя продавать Парк Про корпоратам, и без этого AI не сможет помогать с путевыми листами / тахо.
2. **Реестр страховок шире, чем «КАСКО опционально»** (R4): добавить ОСГОП (при перевозке пассажиров), CMR (международные грузы), экспедиторскую ответственность (для гослог-экспедиторов — критично, регулятор требует), страхование груза. У нас целая когорта клиентов — экспедиторы, мы обязаны их закрывать.
3. **Тахографы и карты водителя (R7)** — с 2026 года ужесточение, при проверке ГИБДД «нет тахографа» = штраф 7 000 ₽ × каждая поездка. Клиенты не помнят даты выдачи карт. ЛК должен напоминать.
4. **Платон (R13)** — у любого ТС > 12 т должна быть привязка к Платон. Неоплата = блокировка проездов = простой. Мы не хотим заходить в Платон бизнесом, но статус и баланс показать в ЛК — обязаны.
5. **Топливные карты (R14)** — это **главный** документ для бухгалтера. Если мы это покажем — резко увеличиваем «уютность» кабинета для всей компании, не только владельца.
6. **Маршруты и рейсы (R16)** — критично для интеграции с ЭТрН (который привязывается к рейсам). Без рейсов невозможно делать прогноз по штрафам и нагрузке парка.
7. **Календарь дедлайнов** — отдельный сквозной модуль (см. § 3.4), который агрегирует все даты истечения из всех реестров: ОСАГО, ДК, КАСКО, тахо-карта, ВУ, мед.справка, экокласс, пропуск, лицензия, гослог-кейс. ICS-экспорт в Google/Apple Calendar. Это превращает ЛК в «второй мозг» владельца парка.
8. **VIN-сканер с автозаполнением** — клиент сканирует ВИН с лобового стекла (камера через WebRTC), мы тянем данные из ФНС/реестра ТС и предзаполняем карточку ТС полностью. Один из самых wow-эффектов первого онбординга.
9. **OBD/телематика** — встроенный коннектор для импорта с РНИС-устройств (JSON/XML), чтобы клиент не вводил пробег руками. Опционально — наш собственный OBD-донгл (далёкое будущее, но архитектурно надо предусмотреть точку входа в БД телематики).
10. **Реестр контрагентов** — отправители и получатели грузов (для ЭТрН и накладных). Часто запрашивается в обязательных полях ЭТрН.
11. **Учёт ремонтов и расходников** — TCO (cost-per-km), напоминания «замени масло через 3000 км». Это уже из мира Fleetio. Ставит ЛК в один ряд с зарубежными SaaS.
12. **Аудитный журнал** — каждое действие пользователя/системы/AI логируется (кто, когда, что изменил). Критично для T4/T5 (юрлица требуют для внутреннего комплаенса) и для разборов с Дептрансом (доказательство «мы предупреждали»).
13. **Поиск по ИНН для импорта парка** — клиент с большим парком вводит ИНН → мы через API ФНС / открытые данные тянем все ТС, зарегистрированные на компанию. Радикально снижает порог входа в Парк Про.
14. **Шеринг карточки ТС между компаниями** — если клиент сдаёт ТС в аренду/лизинг другому юрлицу, обоим нужно видеть карточку. Делаем «гостевой доступ» по ссылке/коду.
15. **Экспорт всех данных** — GDPR-style «выгрузить всё» одной кнопкой (152-ФЗ требует). Заодно решает страх клиента «а что если я уйду» — он знает, что данные забираются.
16. **Импорт-из-конкурента** — интерфейс «приехали от propusk-mkad.ru / mkadttksk.ru? Загрузите свои PDF — мы распознаем и заполним». Снижает порог миграции.

### 2.3. Дополнения по итогам research lap (мировые fleet-практики)

После исследования Samsara / Motive / Fleetio / Webfleet / Wialon / OMNICOMM / ATI.SU и UX-паттернов Revolut AIR + ChatGPT Voice — добавляю в план фичи, которые в зрелых fleet SaaS уже стандарт, но у нас (и у конкурентов по пропускам) их нет. Эти фичи становятся отдельными модулями поверх 20 реестров и закрываются в Phase S (см. § 8).

**A. DVIR / Электронный предрейсовый осмотр (Driver Vehicle Inspection Report)**

Российский аналог — обязательный предрейсовый/послерейсовый контроль ТС в путевом листе. У всех клиентов это сейчас бумажка с подписью механика. Мы делаем это **цифровым в мобильном приложении водителя**: чек-лист по узлам (тормоза, шины, фары, тахо, аптечка, огнетушитель, пломбы), фото каждого пункта, подпись водителя + механика, сохранение в R7/R5 и в путевой лист. На один tap «всё ок» если нет дефектов. При найденном дефекте → автоматически создаётся сервисный кейс (R17) и алерт диспетчеру. У Samsara/Fleetio это главная фича, которую любят водители.

**B. Predictive Fine Risk Scoring (предиктивный риск-скор штрафов)**

ML-модель на истории штрафов клиента + публичных данных (камеры, маршруты, время суток): прогнозирует вероятность штрафа за следующие 30 дней по каждому ТС/водителю. Output — «ТС А123БВ777: 70% вероятность штрафа за превышение по утрам пн-пт на МКАД». Action — алерт водителю в мобиле + предложение Антиштраф / Юрист-перевозчик. Это **продающая** фича: клиент видит, что мы знаем его данные лучше него самого, и upsell на Антиштраф становится не «купите подписку», а «вот ваш конкретный риск».

**C. Dynamic Vehicle Health Score 0–100 (не возрастной, а живой)**

Не «грузовик 2008 года = старый», а агрегат из реальных данных: код OBD, статус ДК, история ТО, моточасы, превышения, RUL (Remaining Useful Life) ключевых узлов (турбо, сцепление, тормозные колодки, аккумулятор). Виджет «Здоровье парка» (§3.3) использует именно это, а не просто «истекает ОСАГО». Источник данных — на старте ручной (механик заполняет), с интеграцией РНИС → автоматически. Долгосрочная цель — partnership с производителями OBD-донглов или ECU телематических устройств для импорта DTC в реальном времени.

**D. Driver Performance & Coaching (T4+)**

Метрики водителя без инвазивности (без видеокамер): hard braking events, harsh acceleration, превышения скорости, idle time, расход топлива относительно среднего по парку. На выходе — Driver Risk Score, который потом прокладывается в страховой пакет (страховщики дают скидку 5–15% за безопасных водителей — мы превращаем это в живой инструмент торга при перепродлении ОСАГО/КАСКО). Coaching по результатам — короткие push-уведомления водителю «вчера 3 резких торможения, в среднем 1, обрати внимание». Это копия Samsara Coach.

**E. Trip Cost Calculator (детальный TCO рейса)**

Калькулятор себестоимости рейса с разбивкой: топливо (литры × цена) + Платон + зарплата водителя × часы + амортизация ТС × км + потенциальные штрафы (из predictive risk). На выходе — рекомендованная цена для шиппера с маржой N%. Сейчас перевозчики делают это в Excel или вообще на глаз и теряют маржу. Калькулятор живёт в разделе «Маршруты и рейсы» (R16) и доступен агенту через `calculateTripCost()` tool. Источники — Яндекс.Карты для дистанции, API Платона, fuel-фиды (Газпром/Лукойл).

**F. Downtime Cost Estimator + ROI ремонта**

Сколько ты теряешь, пока грузовик стоит. Простой считается из R6 (РНИС не двигалось N часов) или из планируемых ТО. На каждый деферрный ремонт (например, «турбо подшипник, замени через 45 дней») показываем: «починить сейчас 15 000 ₽, не починить → отказ + 50 000 ₽ за срочный ремонт + 40 000 ₽ простоя = экономия 75 000 ₽». Это превращает расходы на ТО из «затраты» в «инвестиция с ROI» — другой язык продажи Парк Про корпоратам.

**G. Permit & Insurance Bundle Optimizer**

Перед оформлением нового рейса (или периодически) AI агент анализирует: какие у клиента активные пропуска / страховки / разрешения, какие нужны для типовых маршрутов клиента, и предлагает оптимальный пакет с обоснованием экономии. «Вы возите хазмат на Дальний Восток без CMR — добавьте, экономия N₽ за один инцидент против штрафа M₽». Этот же инструмент агент использует, когда клиент спрашивает «что мне нужно для рейса в Питер с грузом 20 т?».

**H. Regulatory Timeline Tracker (отдельный календарь регуляторики)**

В отличие от «календаря дедлайнов клиента» (§3.4), это **публичный календарь предстоящих изменений в законодательстве**: даты вступления в силу новых правил ГосЛог, ЭТрН, тахо, экокласса, расширения каркаса в Москве, повышения тарифов Платона. Привязка к продукту: каждое изменение помечено как «вас это касается» если у клиента активирован соответствующий модуль. Источник — наша внутренняя экспертиза (мы и так это отслеживаем для маркетинга и продаж — теперь публикуем в ЛК как value). Снижает тревожность и работает как retention-механизм.

**I. Samsara-style Intelligent Safety Inbox (T4+)**

Единый «инбокс» событий парка, где ML auto-triage по критичности: low (показать в виджете), mid (показать диспетчеру в ленте), high (push + телефонный звонок диспетчеру + SMS). События: РНИС-сбой, штраф, аннулирование, дефект из DVIR, низкое здоровье ТС, истечение документов, срыв ETrN-подписания. Это уровень корпоратов — у Samsara это главный экран диспетчера крупного парка.

**J. Inline Action Buttons in agent responses + Session Memory + Quick Quad-Menu**

UX-микро-улучшения для встроенного агента (см. § 5.5):
- В каждом ответе агента — inline-кнопки действий («Оплатить штраф», «Продлить пропуск», «Связаться с юристом», «Открыть в кабинете»). Без модалок, без переходов.
- Сессионная память контекста ТС: если клиент задал вопрос про А123БВ777, последующие 10 реплик подразумевают это же ТС, агент не переспрашивает.
- При первом нажатии на иконку агента (до того как пользователь начал говорить/печатать) — quad-menu из 4 контекстных кнопок: «Статус пропусков», «Есть штрафы?», «Кто истекает скоро?», «Сколько стоит рейс?». Снижает порог входа для нового пользователя.

**K. Steering Wheel BLE Remote (далёкая опция, T5)**

Для водителей с долгими рейсами — BLE-кнопка на руль (или адаптер существующих рулевых кнопок) → push-to-talk на нашего агента без отрыва рук. True hands-free. Это уже эксклюзив для корпоративных клиентов в дальнобое. Архитектурно учитываем сейчас, реализация — после Phase O (Capacitor + native plugins).

## 5. AI-сущность кабинета — наш встроенный агент

### 5.1. Что это и почему НЕ мессенджер как первичный канал

> **Решение от 2026-04-11:** В качестве вторичного канала используется **MAX-мессенджер** (российский национальный мессенджер от VK, обязательная предустановка на устройствах с 2026), а не Telegram. Архитектура бота остаётся прежней (webhook → нашему API → Vercel AI SDK), меняется только клиентский SDK и транспорт. Подробнее — § 9.3.

Бот в MAX остаётся как канал нотификаций и опциональный второй интерфейс — он удобен для статус-уведомлений, но **главный канал — наше приложение**. Причины (это нужно зафиксировать как принципы):

1. **Конфиденциальность.** Данные парка — реестры ТС, водителей, штрафов, маршрутов, договоров — не должны проходить через сторонний мессенджер ни при каких обстоятельствах. MAX как госмессенджер может быть удобнее с точки зрения регуляторики и комплаенса, чем Telegram, но для корпоратов из Парк Про основная часть данных всё равно должна жить в нашем приложении за нашим RBAC и аудитом.
2. **Контекст.** Когда клиент в нашем приложении, агент знает, на какой странице он находится, какие фильтры открыты, какую сущность смотрит. В мессенджере этого контекста нет — там общий чат.
3. **Голос.** В наш WebView/мобильное приложение мы можем встроить нативное распознавание речи (Web Speech API + Yandex SpeechKit, с push-to-talk и continuous listening). В мессенджер-боте голос ограничен голосовыми файлами, без low-latency UX и без барж-ин.
4. **Действия.** В нашем приложении агент может прямо открыть форму заявки, подсветить поле, вызвать сканер ВИН через камеру, открыть КПА-сессию для подписания УКЭП. В боте он может только дать ссылку «вернись в кабинет».
5. **Идентичность бренда.** Когда агент живёт внутри нашего приложения с нашей графикой и анимацией — он «наш ИнфоПилот». В мессенджере он один из множества ботов.
6. **Удержание.** Каждое обращение к агенту = заход в наше приложение = возможность увидеть алерт, апселл, новый продукт. В мессенджере пользователь не возвращается в наш UI.

> Для T0 (гости сайта) и для тех, кому удобнее писать из мессенджера, оставляем **MAX-бота `@infolog24`** как **запасной канал**: он умеет минимум — статусы заявок, нотификации, начало диалога с эскалацией на менеджера, а на сложные операции отправляет deep-link в наше приложение.

| Категория | Примеры |

### 5.5. Voice UX — паттерны

Источник вдохновения — ChatGPT Voice Mode + Perplexity Voice + автомобильные ассистенты.

| Сценарий | Поведение |
|---|---|
| **Push-to-talk (по умолчанию для desktop)** | Зажал кнопку микрофона — говорит — отпустил. STT → отображается распознанный текст → отправка. Минимизирует случайные срабатывания |
| **Hands-free (mobile, опционально)** | Кнопка «Hands-free» в виджете → включает continuous listening с VAD (voice activity detection). Для водителя за рулём (в режиме «в дороге» — см. ниже) |
| **Wake-word** | «Логист» (или другое имя) — включает слушание на N секунд. Только в hands-free. Использовать `onnxruntime-web` + лёгкая модель wake-word (Picovoice Porcupine лицензия / open source аналоги) |
| **Streaming ответ** | Модель отвечает текстом, текст параллельно идёт в TTS, аудио проигрывается. Прерывание (барж-ин): пользователь начал говорить — TTS останавливается, начинается слушание |
| **Subtitles** | Пока агент говорит — параллельно показываем текст ответа большим шрифтом. Для глухих, для шумных мест, для скриншотов |
| **Режим «в дороге»** | Клиент включает кнопкой «Я за рулём». UI упрощается до 2 кнопок (микрофон + SOS), голос — приоритетный канал. Все алерты — голосом. Action Cards требуют подтверждения голосом, не тапом |
| **Offline-fallback** | Если нет связи — STT работает на устройстве (Web Speech API native), сообщение ставится в очередь, отправляется при восстановлении. На критичных функциях (SOS-эвакуация) — fallback на телефонный звонок диспетчеру |
| **Latency targets (по этапам)** | Wake-word detection on-device ≤ 500 мс. STT streaming ≤ 1.5 с от конца фразы. LLM первый токен ≤ 1 с (Edge runtime + минимальный prompt + tool injection). TTS первый звук ≤ 500 мс. End-to-end (от конца речи до начала ответа TTS) — **≤ 1.5 с** в идеале, ≤ 2.5 с допустимо. Для этого — Yandex SpeechKit RU realtime, Edge runtime, prompt-кэширование, параллелизация tool calls с генерацией. |

### 5.6. Безопасность и privacy

1. **Scoping.** Все tool-чтения автоматически ограничены `companyId` пользователя из сессии. Нельзя случайно прочитать данные другой компании, даже если модель попытается.
2. **Confirmation для write.** Любое мутирующее действие (создание заявки, обжалование штрафа, заказ эвакуатора) — через Action Card с явным подтверждением. Никаких автоматических действий «в обход».
3. **PII redaction в логах.** ВИН, ФИО водителей, госномера — храним, но в ai_messages-логах хешируем для обучающих выборок.
4. **Аудит каждого вызова.** `ai_tool_calls` пишет: какой tool, какие аргументы, какой результат, кто запросил, когда. Доступно владельцу компании.
5. **Запрет на самоисполнение цепочек.** `maxSteps: 5` для tool calls, после — обязательная пауза с показом промежуточного результата клиенту.
6. **Прозрачность модели.** В UI показываем «Ответ сформирован моделью {name}, на основе ваших данных за {period}. Это не юридическая консультация». В чувствительных темах (юридические советы, интерпретация законов) — обязательная сноска + предложение эскалации к юристу-перевозчику.
7. **Голосовые данные.** Аудио не хранится — только текстовый транскрипт. По 152-ФЗ — согласие при первой активации голосового режима.
8. **Запрет вне-доменных запросов.** System prompt блокирует темы вне нашего домена («погода в Сочи», «расскажи анекдот»). Не как «отказ» — как «я помощник по транспорту, не отвлекайся, у нас работа».

| Источник | Что в нём | Обновление |

## 7. Мобильность — web-first, но с учётом мобильного

### 7.1. Этапы

1. **PWA (фаза 1):** ставится из браузера, работает offline (Service Worker кеширует UI и последние данные), есть push-уведомления (Web Push API). Достаточно для 80% задач.
2. **Capacitor (фаза 2):** обёртка PWA в нативную обёртку для App Store / Google Play — нужна для нативного микрофона (continuous listening), wake-word, фоновых push, нативных алертов SOS, доступа к камере (VIN-сканер), геолокации. Даёт магазины и иконку на рабочем столе.
3. **Native widgets (фаза 3, опционально):** виджет «Здоровье парка» на главном экране телефона (iOS/Android), watchOS-уведомление об истекающих документах. Это уже эксклюзив, не приоритет.

### 7.2. Mobile-specific UX

- Floating Action Button (FAB) внизу экрана — прямой вход в AI-агента (single tap → mic).
- Bottom navigation tabs: Обзор / Парк / Сервисы / Бот / Профиль.
- Все таблицы (заявки, штрафы, ТС) — карточные на mobile.
- Быстрые действия с pull-to-refresh.
- Haptic feedback на критических кнопках (SOS).
- Тёмная тема (важно для ночных рейсов).

## 9. Принятые решения (подтверждены 2026-04-11)

### 9.1. Имя и личность агента — ✅ ПРИНЯТО: «ИнфоПилот»

Один бренд **«ИнфоПилот»** для встроенного AI-агента и для продукта №6 из продуктовой линейки. Логика: продукт уже закреплён в стратегии главной v2, имеет страницу `/infopilot`, своё позиционирование и коммерческую модель. Встроенный AI-агент в ЛК — это **тот же ИнфоПилот**, только в текстово-голосовом интерфейсе.

**Что это влечёт по реализации:**
- Иконка агента в ЛК — фирменная иконка ИнфоПилота (анимированная, в брендовых цветах).
- Голос (выбирается в настройках — мужской/женский, см. § 9.2) — единый для всех сценариев: текстовый чат, голосовой режим, автозвонки партнёрам ИнфоПилота (эвакуация/СТО/мойка).
- Приветствие при первом запуске: «Привет, я ИнфоПилот. Помогу с пропусками, штрафами, документами и сервисами на трассе. Можно текстом или голосом».
- Тон коммуникации: профессиональный, без канцелярита, без излишней «дружелюбности» (B2B аудитория, доверие важнее панибратства).
- Дисклеймер при первом голосовом сообщении и при чувствительных темах (юридические советы): «Это не юридическая консультация, для официального ответа подключим юриста-перевозчика».
- Брендинговые материалы (страница `/infopilot`, лендинг, маркетинг) и встроенный агент в ЛК — **синхронизированы**: одна графика, одна анимация, один tone of voice.

### 9.2. Голос и провайдер STT/TTS — ✅ ПРИНЯТО: Yandex SpeechKit

**Стек на старте (Phase L):**
- **STT (распознавание речи):** Yandex SpeechKit Streaming Recognition API (gRPC, поддерживает русский realtime, latency ≤ 1 с).
- **TTS (синтез речи):** Yandex SpeechKit Streaming TTS, премиум-голоса (`alena`/`filipp`/`ermil` — выбор пользователя в настройках).
- **Wake-word (опционально, mobile):** Picovoice Porcupine с обученной моделью на «ИнфоПилот» (on-device, не требует серверного канала).
- **Браузерный fallback:** Web Speech API (когда есть в браузере) для T0/T1 без затрат на SpeechKit JWT.
- **Резерв:** Whisper API (OpenAI) только для случаев, когда клиент говорит не на русском (английский/китайский для международных перевозчиков). Не основной канал.

**Инфраструктурные следствия:**
- Договор и биллинг с Yandex Cloud (ЮКЛауд) — задача BizDev, нужен заранее.
- JWT-сервис на нашем backend для выдачи коротких токенов клиенту (Edge Function).
- Метрика стоимости: SpeechKit считает по символам/секундам — закладываем в unit-экономику ИнфоПилота.
- Серверный peering — Yandex Cloud Москва (`ru-central1-*`), это даёт минимальный RTT.

### 9.3. Вторичный канал — ✅ ПРИНЯТО: MAX-мессенджер вместо Telegram

**Решение:** Telegram-бот заменяется на бота в **MAX-мессенджере** (российский национальный мессенджер от VK, обязательная предустановка на устройствах с 2026 г.). Архитектура остаётся прежней, меняется транспорт.

**Почему MAX, а не Telegram:**

| Критерий | Telegram | MAX |
|---|---|---|
| Регуляторика РФ | Иностранный мессенджер, статус неопределённый | Российский, государственный, есть закон об обязательной предустановке |
| Доступность для B2B-клиентов | У большинства есть, но часть корпоратов запрещает использование | По умолчанию у всех с 2026 — нулевой порог входа |
| Комплаенс крупных клиентов | Блокер для гос./окологос/КИИ-клиентов | Не блокер |
| API-зрелость для ботов | Зрелый bot API, мини-приложения | На 2026 — стартующая платформа, но bot API/mini-apps по образу TG |
| Долгосрочные риски (блокировка/санкции) | Есть | Нет |
| Наша целевая аудитория | Часть пользуется, часть — нет | Все будут пользоваться |

**Что делает MAX-бот ИнфоПилота (рамки урезания):**

1. **Нотификации:** статусы заявок, готовность пропусков, входящие штрафы, дедлайны (за 30/14/7/1 день), смена статуса гослог-кейса, уведомления от ИнфоПилот-инцидентов («эвакуатор подъехал», «ремонт готов»).
2. **Минимальные команды:** `/start`, `/status`, `/мои_пропуска`, `/последние_штрафы`, `/связь_с_менеджером`. Всё остальное → deep-link в наше приложение.
3. **Текстовый диалог с агентом:** ограниченный — read-only вопросы по данным клиента и общие вопросы о продуктах. Любой write-action (создание заявки, обжалование штрафа, заказ эвакуатора) → deep-link «Открыть в приложении» с pre-filled параметрами.
4. **Эскалация на менеджера:** один тап → менеджер получает push с контекстом диалога.
5. **НЕ делает в MAX:** голос (только в нашем приложении), реестры (только в приложении), сложные действия (только в приложении), КПА-сессии для УКЭП (только в приложении).

**Архитектура (одна и та же для текста в нашем приложении и для MAX-бота):**

```
┌─────────────────┐    ┌─────────────────┐
│  Наш AssistantWidget │  MAX-бот        │
│  (приложение)   │    │  @infolog24     │
└────────┬────────┘    └────────┬────────┘
         │                      │
         └──────────┬───────────┘
                    │
         ┌──────────▼──────────┐
         │   /api/assistant    │
         │   (Vercel AI SDK)   │
         │   tools + RBAC      │
         └──────────┬──────────┘
                    │
         ┌──────────▼──────────┐
         │   DAL + БД + RAG    │
         └─────────────────────┘
```

Ключевое: **бэкенд один**. На фронте два транспорта (наше приложение + MAX webhook). System prompt включает channel hint (`channel: 'app' | 'max'`), чтобы агент знал, какие действия доступны (в MAX он не предлагает голос и не открывает формы — только deep-link'и).

**Технические задачи (новый план):**
- Зарегистрировать `@infolog24` в MAX-каталоге ботов (после открытия публичного API; на 2026-04 — закрытая бета, нужно подавать запрос).

- Существующий план [`2026-04-10-ai-чатбот-и-telegram-бот.md`](2026-04-10-ai-чатбот-и-telegram-бот.md) **переименовать/мигрировать** в `2026-04-11-ai-агент-и-max-бот-v2.md` (отдельная задача в обновлении мастер-плана). Каркас на grammY (Telegram SDK) удаляется, заменяется на MAX SDK / прямой webhook.

Если к моменту запуска MVP (Этап 2 дорожной карты) API не будет открыт — стартуем без вторичного канала, наше приложение справится самостоятельно. MAX-бот добавляется по готовности API без блокировки основной разработки.

### 9.4. Скоуп Этапа 1 — ✅ ПРИНЯТО: текстовый AI первым, голос потом

Подтверждённая нарезка (см. § 10):

- **Этап 1–2:** база тиров + реестры R1–R5 + базовый комплаенс + **текстовый AI-агент** (Phase K). Голоса нет.
- **Этап 3:** сервисы и юридический слой + расширенный комплаенс.
- **Этап 4:** **голосовой режим** (Phase L) + RAG база знаний (Phase M).
- **Этап 5–8:** корпоративные клиенты, мобильное, аналитика, hands-free.

Это даёт быструю проверку гипотезы «AI-агент по данным клиента закрывает > 60% обращений» на тексте, прежде чем вкладываться в голосовую инфраструктуру SpeechKit. Если конверсия плохая — пересматриваем подход без потерь на голос.

### 9.5. Где живут данные клиента — у нас или у партнёра

Часть реестров — телематика (РНИС), Платон, ЭТрН, ГосЛог — это данные **в чужих системах**. Мы только подключаемся через API и **кэшируем** локально для отображения и для AI-агента. Источник истины — у партнёра. Это правильное решение для:
- скорости (не делаем live-запросы при каждом открытии страницы)
- доступности (если партнёр лежит — мы показываем последние данные с пометкой «обновлено N часов назад»)
- AI-агента (он отвечает мгновенно по локальной БД, а не ждёт partner API)

Cron-синхронизация: каждые 15 минут для критичного (РНИС, штрафы), каждый час для среднего (Платон, ЭТрН), раз в день для медленного (ГосЛог).

## 11. Что НЕ войдёт в этот план (явно отложено)

- Полноценный TMS (Transportation Management System) уровня 1С:Логистика — мы не строим ERP, мы делаем cabinet с реестрами + AI.
- Биржа грузов (типа ATI.SU). У нас нет этой амбиции и нет ресурсов на оператора биржи.
- Бухгалтерия. Мы интегрируемся с 1С/Контуром, но не пытаемся заменить.
- Электронный документооборот (ЭДО) сами не реализуем — интеграция через Контур.Диадок / СберКорус / ЭДО.Поток.
- Карты с навигацией для водителей — у Яндекс.Грузовики и Atlas есть свои приложения, мы туда не лезем.
- Игровая механика / геймификация. Не наш сегмент (у нас B2B-владельцы парков, им не до баджей).

Эти исключения нужны как контр-список: если в обсуждении всплывает «а давайте ещё ERP добавим» — отбиваемся ссылкой на эту секцию.
