## Обзор

Личный кабинет — ключевая точка автоматизации бизнеса. Сейчас клиенты взаимодействуют через телефон и WhatsApp, менеджеры делают всё вручную. ЛК должен:

1. **Убрать рутину менеджеров** — клиент сам загружает документы, отслеживает статус, получает уведомления
2. **Ускорить оформление** — AI извлекает данные из документов, валидирует, предзаполняет формы
3. **Повысить retention** — удобный кабинет = клиент возвращается, а не ищет конкурента

**Целевая метрика:** 70% заявок подаётся через ЛК (а не через менеджера) через 6 месяцев после запуска.

## Фаза 2: Авторизация / регистрация

**Цель:** Клиент входит по телефону (SMS OTP) или email. Безопасно, быстро, без пароля на старте.

### Архитектура авторизации

```
Better Auth (server) ← настроен в packages/auth/
     ↓
API Route: /api/auth/[...all]
     ↓
Клиентский хелпер: packages/auth/client.ts
     ↓
React-компоненты: apps/web/app/(auth)/
```

### Экраны

1. **`/login`** — Вход
   - Поле телефона (маска +7 (___) ___-__-__)
   - Кнопка «Получить код»
   - Поле ввода OTP (6 цифр, автофокус, автопереход между ячейками)
   - Таймер повторной отправки (60 сек)
   - Альтернатива: «Войти по email»
   - Ссылка: «Нет аккаунта? Зарегистрируйтесь»

2. **`/register`** — Регистрация
   - Шаг 1: Телефон → SMS OTP → подтверждение
   - Шаг 2: ФИО, email, название компании (опционально), ИНН (опционально)
   - Шаг 3: Согласие на обработку персональных данных (152-ФЗ) — чекбокс + ссылка на политику
   - Результат: аккаунт создан, редирект в ЛК

3. **`/forgot-password`** — Восстановление (если будет парольная авторизация)

### SMS OTP — техническая реализация

- **Провайдер:** SMS.ru (основной) + SMSC.ru (резервный)
- **Flow:** Better Auth plugin `phoneNumber` → кастомный `sendOTP` → вызов SMS.ru API
- **Rate limiting:** max 3 SMS / 10 мин на один номер, max 10 / час
- **OTP:** 6 цифр, TTL = 5 мин, одноразовый
- **Fallback:** при 2 неудачных SMS — предложить авторизацию звонком (последние 4 цифры)

### Валидация

- Телефон: формат +7XXXXXXXXXX, проверка длины
- Email: формат + проверка MX (опционально, на сервере)
- ИНН: 10 или 12 цифр, контрольная сумма
- Все формы: React Hook Form + Zod

### Middleware защиты

```
middleware.ts:
  /dashboard/* → проверка сессии Better Auth → redirect /login если нет
  /admin/*     → проверка сессии + роль admin/manager
  /login       → redirect /dashboard если уже авторизован
```

### Критерии готовности
- [ ] Регистрация по телефону (SMS OTP) работает end-to-end
- [ ] Вход по телефону работает
- [ ] Вход по email работает
- [ ] Middleware защищает /dashboard/*
- [ ] Rate limiting SMS работает
- [ ] Согласие на обработку ПД сохраняется

| Колонка | Описание |
|---|---|

| Гос. номер | А123БВ777 |
| Масса | 18 т |
| Эко-класс | Евро-5 |
| Активных пропусков | 2 |

  - Гос. номер

  - Разрешённую массу
  - Категорию ТС

4. Confidence < 0.8 → поле подсвечивается жёлтым: «Проверьте значение»
5. Клиент корректирует если нужно → Submit
```

## Фаза 6: Загрузка документов (drag & drop, превью, валидация)

**Цель:** Максимально простая загрузка. Клиент не должен думать «в каком формате отправить».

### Компонент загрузки `FileUploader`

**Технологии:**
- `react-dropzone` — drag & drop зона
- Presigned URLs → Selectel S3 — прямая загрузка из браузера
- Claude Vision API — валидация и извлечение данных

**UX:**

```
┌─────────────────────────────────────────────┐
│                                             │
│     📄 Перетащите файлы сюда               │
│     или нажмите для выбора                  │
│                                             │
│     PDF, JPG, PNG, HEIC — до 10 МБ         │
│                                             │
└─────────────────────────────────────────────┘

Загруженные файлы:
┌──────┬──────────────────────────┬──────────┐
│ 📄   │ СТС_А123БВ777.pdf       │ ✅ Готов  │
│      │ 2.3 МБ                  │  🗑      │
├──────┼──────────────────────────┼──────────┤
│ 📷   │ IMG_2847.jpg             │ ⏳ Загр.  │
│      │ 4.1 МБ                  │  🗑      │
├──────┼──────────────────────────┼──────────┤
│ 📄   │ доверенность.pdf         │ ⚠ Ошибка │
│      │ Нечитаемый документ      │  🔄 🗑   │
└──────┴──────────────────────────┴──────────┘
```

### Flow загрузки

```
1. Клиент выбирает файл (drag&drop или клик)
2. Client-side валидация:
   - Формат: PDF, JPG, JPEG, PNG, HEIC, WEBP
   - Размер: max 10 МБ на файл, max 50 МБ на заявку
   - HEIC/WEBP → конвертация в JPEG на клиенте (browser-image-compression)
3. Запрос presigned URL: POST /api/documents/presign
4. Прямая загрузка в S3: PUT presigned_url (с прогресс-баром)
5. Подтверждение: POST /api/documents/confirm { s3_key, type }
6. Фоновая обработка:
   - Claude Vision → извлечение данных
   - Проверка читаемости
   - Определение типа документа (если не указан)
7. Результат: статус uploaded/verified/rejected
```

### Превью документов

- **PDF:** `react-pdf` — первая страница как превью, полный просмотр по клику
- **Изображения:** `<img>` с lazy loading, lightbox по клику (shadcn Dialog)
- **Миниатюры:** генерируются на клиенте (canvas resize) для списков

### Страница документов `/dashboard/documents`

- **Список всех документов** клиента
- **Фильтры:** по типу (СТС, ПТС, доверенность...), по статусу, по ТС
- **Привязка:** каждый документ привязан к ТС и/или заявке
- **Повторное использование:** при новой заявке — выбрать из уже загруженных документов

### Критерии готовности
- [ ] Drag & drop загрузка работает (PDF, JPG, PNG, HEIC)
- [ ] Прогресс-бар загрузки
- [ ] Превью загруженных файлов
- [ ] Presigned URL flow работает

## Фаза 7: Подача заявки на пропуск (пошаговая форма)

**Цель:** Пошаговый мастер. Клиент не может запутаться — каждый шаг понятен.

### Мастер заявки `/dashboard/orders/new`

**Stepper:**
```
[1. Зона] → [2. ТС] → [3. Маршрут] → [4. Документы] → [5. Проверка] → [6. Оплата]
```

#### Шаг 1: Выбор зоны и типа пропуска

```
Куда нужен пропуск?
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│    МКАД     │  │    ТТК      │  │  Садовое    │
│  (карточка) │  │  (карточка) │  │  (карточка) │
└─────────────┘  └─────────────┘  └─────────────┘

Тип пропуска:
○ Годовой — 10 000 ₽   (+ временный бесплатно)
○ Временный — 3 500 ₽
○ Ночной — 3 500 ₽

- Интеграция YooKassa:
  - `POST /api/payments/create` → создаёт платёж в YooKassa
  - Редирект на страницу оплаты YooKassa
  - Webhook `/api/yookassa/webhook` → обновляет статус заявки

- Для юрлиц: «Выставить счёт» → генерация PDF-счёта → отправка на email

### Сохранение прогресса

- Все шаги сохраняются в `localStorage` (ключ: `draft-order-{userId}`)
- При уходе со страницы — предупреждение «Заявка не сохранена»
- При возвращении — предложение продолжить
- После отправки — очистка черновика

### Валидация

Каждый шаг валидируется отдельной Zod-схемой:
- Шаг 1: `z.object({ zone: z.enum([...]), permitType: z.enum([...]), ... })`
- Шаг 2: `z.object({ vehicleId: z.string().uuid() })`
- и т.д.
- Кнопка «Далее» неактивна пока текущий шаг не валиден

### SSE — техническая реализация

```
GET /api/notifications/stream
Headers: Authorization: Bearer <session>
Content-Type: text/event-stream

data: {"type":"notification","payload":{"id":"...","title":"Пропуск готов!"}}
data: {"type":"order_status","payload":{"orderId":"...","status":"approved"}}
```

- Соединение устанавливается при загрузке layout ЛК
- Автореконнект при обрыве (EventSource делает это автоматически)
- Серверная сторона: Redis Pub/Sub для масштабирования на несколько инстансов

## Фаза 12: Тестирование, оптимизация, полировка

**Цель:** Всё работает стабильно, быстро, без багов.

### 12.1. Performance

- **Core Web Vitals:**
  - LCP < 2.5s (dashboard page)
  - FID < 100ms
  - CLS < 0.1
- **Bundle size:** проверка через `@next/bundle-analyzer`
- **Lazy loading:** все тяжёлые компоненты (charts, file preview, chat) — dynamic import
- **Оптимизация изображений:** Next.js `<Image>` для всех картинок
- **Кэширование:** React Query с правильными staleTime/gcTime

### 12.2. Тестирование

- **E2E:** Playwright — ключевые сценарии:
  - Регистрация → вход → создание заявки → оплата
  - Загрузка документа → проверка превью
  - Фильтрация списков
- **Компоненты:** Vitest + Testing Library — unit-тесты для кастомных компонентов
- **API:** Vitest — тесты для API routes (валидация, авторизация)

### 12.3. Accessibility

- Клавиатурная навигация (все интерактивные элементы focusable)
- ARIA-атрибуты (shadcn/ui обеспечивает через Radix)
- Contrast ratio ≥ 4.5:1 (WCAG AA)
- Screen reader: заголовки, labels, alt-тексты

### 12.4. Error handling

- **Error boundaries** на каждом route segment
- **Fallback UI:** скелетоны при загрузке, EmptyState при пустых данных
- **Offline:** toast «Нет соединения», retry при восстановлении
- **API errors:** единый формат ошибок, toast с понятным сообщением

### 12.5. Security

- CSRF-защита (Better Auth обеспечивает)
- Rate limiting на API routes (`@upstash/ratelimit` или кастомный через Redis)
- Input sanitization (Zod на каждом эндпоинте)
- Content Security Policy headers
- Presigned URLs с коротким TTL (5 мин)

### Критерии готовности
- [ ] Core Web Vitals в зелёной зоне
- [ ] E2E тесты проходят на ключевых сценариях
- [ ] Клавиатурная навигация работает
