### Почему Coolify, а не голый VPS

- Vercel-подобный DX: git push → автодеплой, SSL, preview deployments
- Dashboard для управления сервисами (PostgreSQL, Redis, Next.js)
- Zero-downtime deployments из коробки
- 52K+ GitHub stars, активная разработка
- Один VPS — всё на месте, без внешних зависимостей

### Почему не Vercel

- Серверы в США/ЕС — задержки для российских пользователей
- Риск блокировки/ограничения для РФ-компании
- 152-ФЗ: персональные данные клиентов должны храниться в РФ
- Pay-per-use: при росте трафика может стать дорого

## Фаза 0: Заказ и базовая настройка VPS
**Статус:** [ ]
**Исполнитель:** Вручную + агент

### Задачи

**0.1. Заказать VPS на Selectel**

Рекомендуемая конфигурация:

| Среда | CPU | RAM | Диск | Стоимость |
|---|---|---|---|---|
| **Production** | 4 core | 8 GB | 80 GB SSD | ~3 200₽/мес |
| **Staging** (опционально) | 2 core | 4 GB | 40 GB SSD | ~1 600₽/мес |

Регион: Москва (минимальная задержка для клиентов).

**0.2. Базовая безопасность VPS**

```bash
# 1. Обновить систему
apt update && apt upgrade -y

# 2. Создать пользователя (не root)
adduser deploy
usermod -aG sudo deploy

# 3. Настроить SSH-ключ
mkdir -p /home/deploy/.ssh
# Скопировать публичный ключ
chmod 700 /home/deploy/.ssh
chmod 600 /home/deploy/.ssh/authorized_keys

# 4. Отключить вход по паролю
# /etc/ssh/sshd_config:
# PasswordAuthentication no
# PermitRootLogin no
systemctl restart sshd

# 5. Настроить файервол
ufw allow 22/tcp    # SSH
ufw allow 80/tcp    # HTTP
ufw allow 443/tcp   # HTTPS
ufw allow 8000/tcp  # Coolify dashboard (временно, потом через SSH tunnel)
ufw enable

# 6. Fail2ban
apt install fail2ban -y
systemctl enable fail2ban
```

**0.3. Установить Docker + Docker Compose**

```bash
curl -fsSL https://get.docker.com | sh
usermod -aG docker deploy
```

**Критерий готовности:** VPS доступен по SSH (ключ, не пароль), файервол настроен, Docker установлен.

**Исполнитель:** Агент
**Зависимость:** Фаза 1

### Задачи

**2.1. PostgreSQL 16 (через Coolify)**

1. Coolify → Resources → New → PostgreSQL
2. Настройки:
   - Версия: 16
   - Пароль: сгенерировать надёжный (32+ символа)

   - Пароль: да
   - Persistent: AOF (для надёжности)
   - maxmemory: 256mb + allkeys-lru

**2.3. Настроить Selectel S3**

1. Selectel Console → Облачное хранилище → Создать контейнер
2. Имя: `infologistic-docs`
3. Регион: ru-1 (Москва)
4. Тип: Private (без публичного доступа)
5. Создать сервисного пользователя с доступом только к этому контейнеру
6. Сохранить Access Key + Secret Key

## Фаза 4: Деплой приложения
**Статус:** [ ]
**Исполнитель:** Агент
**Зависимость:** Фазы 1–3

### Задачи

**4.1. Настроить Coolify для Next.js**

1. Coolify → Resources → New → Application
2. Source: GitHub repository
3. Build pack: Nixpacks (автоопределение Next.js)
4. Build command: `pnpm build`
5. Start command: `pnpm start`
6. Port: 3000

**4.2. Переменные окружения**

Все `.env` переменные из [Архитектура](2026-04-10-архитектура-и-стек.md) § 6 — ввести в Coolify:
- `DATABASE_URL` → internal PostgreSQL URL
- `REDIS_URL` → internal Redis URL
- `BETTER_AUTH_SECRET` → сгенерировать
- `S3_*` → credentials из Фазы 2.3
- `TELEGRAM_BOT_TOKEN` → из BotFather
- `RESEND_API_KEY` → из Resend dashboard
- `SMS_RU_API_KEY` → из SMS.ru
- `OPENAI_API_KEY` → из OpenAI
- `YOOKASSA_*` → из ЮKassa
- `CRON_SECRET` → сгенерировать (для защиты cron-endpoints)

**4.3. Healthcheck**

Coolify → Application → Healthcheck:
- URL: `/api/health`
- Interval: 30s
- Timeout: 5s
- Retries: 3

Создать endpoint:
```typescript
// app/api/health/route.ts
export async function GET() {
  // Проверить PostgreSQL и Redis
  const db = await checkDb();
  const redis = await checkRedis();
  return Response.json({ status: 'ok', db, redis });
}
```

**4.4. Первый деплой**

1. Push в `main` → Coolify автоматически строит и деплоит
2. Проверить: `https://infolog24.ru` отвечает
3. Проверить: `https://infolog24.ru/api/health` → `{ status: 'ok' }`
4. Запустить миграции: `pnpm db:migrate` (через Coolify Execute)

**4.5. Настроить cron-задачи**

На VPS через SSH:
```bash
crontab -e

# Очередь уведомлений (каждые 5 минут)
*/5 * * * * curl -s -H "X-Cron-Secret: $CRON_SECRET" http://localhost:3000/api/cron/send-notifications >> /var/log/cron-notifications.log 2>&1

# Проверка истекающих пропусков (ежедневно 09:00 МСК)
0 9 * * * curl -s -H "X-Cron-Secret: $CRON_SECRET" http://localhost:3000/api/cron/check-expiring >> /var/log/cron-expiring.log 2>&1

# Автопереход в expired (ежедневно 01:00)
0 1 * * * curl -s -H "X-Cron-Secret: $CRON_SECRET" http://localhost:3000/api/cron/check-expired >> /var/log/cron-expired.log 2>&1

# Очистка осиротевших файлов S3 (ежедневно 03:00)
0 3 * * * curl -s -H "X-Cron-Secret: $CRON_SECRET" http://localhost:3000/api/cron/cleanup-orphans >> /var/log/cron-cleanup.log 2>&1
```

**Критерий готовности:** Сайт работает на `infolog24.ru`, healthcheck зелёный, cron-задачи настроены.

## Фаза 5: Бэкапы
**Статус:** [ ]
**Исполнитель:** Агент
**Зависимость:** Фаза 2

### Задачи

**5.1. Бэкап PostgreSQL → S3**

Скрипт `/opt/scripts/backup-db.sh`:
```bash
#!/bin/bash
TIMESTAMP=$(date +%Y-%m-%d_%H-%M)
FILENAME="infologistic_${TIMESTAMP}.sql.gz"

# Дамп + сжатие
docker exec coolify-postgresql pg_dump -U app infologistic | gzip > /tmp/$FILENAME

# Загрузка в S3
aws s3 cp /tmp/$FILENAME s3://infologistic-backups/db/$FILENAME \
  --endpoint-url https://s3.storage.selcloud.ru

# Удалить локальный файл
rm /tmp/$FILENAME

# Удалить бэкапы старше 30 дней
aws s3 ls s3://infologistic-backups/db/ --endpoint-url https://s3.storage.selcloud.ru \
  | awk '{print $4}' | head -n -30 | xargs -I {} \
  aws s3 rm s3://infologistic-backups/db/{} --endpoint-url https://s3.storage.selcloud.ru
```

Cron: ежедневно в 02:00 МСК.

**5.2. Бэкап Redis (опционально)**

Redis AOF persistence + ежедневный snapshot:
```bash
docker exec coolify-redis redis-cli BGSAVE
docker cp coolify-redis:/data/dump.rdb /tmp/redis-backup.rdb
# → S3
```

**5.3. Проверка бэкапов**

Ежемесячно:
1. Скачать последний бэкап из S3
2. Развернуть на staging
3. Проверить целостность данных

**5.4. Бэкап файлов S3 (документы клиентов)**

Selectel S3 → включить версионирование контейнера. Это встроенная защита от случайного удаления.

**Критерий готовности:** Ежедневные бэкапы в S3, ротация 30 дней, проверка восстановления пройдена.

## Фаза 7: CI/CD (GitHub Actions)
**Статус:** [ ]
**Исполнитель:** Агент
**Зависимость:** Фазы 1, 4

### Задачи

**7.1. Workflow: тесты на PR**

`.github/workflows/test.yml`:
```yaml
name: Test
on: pull_request

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 24
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm lint
      - run: pnpm typecheck
      - run: pnpm test
```

**7.2. Workflow: деплой production**

Coolify автоматически деплоит при push в `main` — дополнительный workflow не нужен.

Но добавить post-deploy проверку:
```yaml
name: Post-Deploy Check
on:
  workflow_dispatch:  # ручной запуск
  push:
    branches: [main]

jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - name: Wait for deploy
        run: sleep 120
      - name: Check health
        run: |
          STATUS=$(curl -s -o /dev/null -w "%{http_code}" https://infolog24.ru/api/health)
          if [ "$STATUS" != "200" ]; then exit 1; fi
```

**7.3. Branch protection**

GitHub → Settings → Branches → `main`:
- Require PR before merge
- Require status checks (test workflow)
- Require 1 approval (если команда > 1)
- No force push

**Критерий готовности:** Тесты запускаются на PR, main защищён, деплой автоматический.
