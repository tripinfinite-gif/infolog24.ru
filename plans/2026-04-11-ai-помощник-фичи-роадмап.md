## PHASE 1 — Foundation для second pilot (без внешних ключей)

**Цель:** превратить чат из reactive «отвечалки» в proactive ассистента, который знает контекст клиента и может действовать.

**Состав:** 5 фич, последовательно. Все используют существующую инфраструктуру.

### P1.1. Page-aware context [x] ВЫПОЛНЕНО 2026-04-11
**Размер:** S
**Зависимости:** нет
**Деплой:** коммит `08827b1`

**Что делаем.** Чат-виджет передаёт в API текущий URL и опциональный ID сущности (грузовик, заявка, документ), которую клиент сейчас просматривает. Ассистент использует это, чтобы давать контекстные ответы без переспросов.

**Файлы:**
- `apps/web/src/lib/chat/types.ts` — новый, типы для clientContext
- `apps/web/src/app/api/chat/route.ts` — принимать clientContext в body
- `apps/web/src/lib/chat/system-prompt.ts` — динамическая инъекция контекста страницы
- `apps/web/src/components/chat/ChatWidget.tsx` (или где он) — собирать context из usePathname + props

**Критерий готовности:** на странице `/dashboard/vehicles/[id]` ассистент в первом ответе ссылается на конкретный грузовик: «Вижу, вы смотрите на А123БВ777. Что подсказать по нему?»

---

### P1.2. Proactive opener [x] ВЫПОЛНЕНО 2026-04-11
**Размер:** S
**Зависимости:** P1.1 (для контекста страницы), getMyContext (уже есть)
**Деплой:** коммит `13a6117`

**Что делаем.** При открытии чата авторизованным пользователем ассистент сам пишет приветственное сообщение, сгенерированное LLM на основе getMyContext. Никаких хардкодных приветствий — всё динамическое.

**Файлы:**
- `apps/web/src/app/api/chat/welcome/route.ts` — новый endpoint, возвращает первое сообщение
- Чат-виджет: при первом mount для авторизованного user — fetch welcome
- system-prompt: отдельная инструкция «приветствие» с нюансами (имя, ближайший дедлайн, активные заявки)

**Критерий готовности:** залогинился → открыл чат → автоматически «Привет, Сергей. Я заметил, у А123БВ777 пропуск истекает через 5 дней. Помочь продлить заранее?»

---

### P1.3. Action Cards protocol [x] ВЫПОЛНЕНО 2026-04-11
**Размер:** M
**Зависимости:** P1.1, P1.2
**Реализация:** через мульти-агентную оркестрацию (3 параллельных агента + оркестратор). См. отдельный план [2026-04-11-p1.3-action-cards-orchestration.md](2026-04-11-p1.3-action-cards-orchestration.md).

**Что делаем.** Tools возвращают не только текст, но и опциональный массив `actions: ActionCard[]`. Каждая ActionCard — это UI-кнопка с действием: продлить пропуск, загрузить документ, создать заявку, связаться с менеджером. Чат-виджет рендерит их под сообщением. Клик → выполнение действия.

**Файлы:**
- `apps/web/src/lib/chat/action-cards.ts` — типы и реестр обработчиков
- Tools (`getMyPermits`, `getMyContext` и т. д.) — возвращают actions в payload
- Чат-виджет: компонент `ActionCardButton` + рендеринг
- Action handlers: вызывают существующие API (`/api/orders`, `/api/contacts`)

**Действия первой версии:**
- `extend_permit(permitId)` → редирект на форму продления с preselected ТС
- `create_order(vehicleId, zone, type)` → редирект на /dashboard/orders/new с предзаполнением
- `upload_document(orderId, type)` → открыть upload-форму
- `contact_manager(reason)` → вызвать requestCallback
- `view_permit(permitId)` → редирект на детали пропуска

**Критерий готовности:** ассистент рассказал про продление → под ответом видны 2 кнопки «Продлить сейчас» / «Напомнить через неделю». Клик «Продлить сейчас» → /dashboard/orders/new с уже выбранным ТС и зоной.

### P1.4. Sentiment-эскалация [x] ВЫПОЛНЕНО 2026-04-11
**Размер:** S
**Зависимости:** нет

**Что делаем.** Промпт инструктирует ассистента распознавать негатив, фрустрацию, ругательства, угрозы уйти. При обнаружении — автоматически инициирует requestCallback с флагом `priority='high'`, менеджер получает алерт раньше других обращений.

**Файлы:**
- `system-prompt.ts` — секция «Sentiment escalation» с триггерами
- `tools.ts` → `requestCallback` принимает `priority?: 'normal' | 'high'`
- `apps/web/src/app/api/contacts/route.ts` — пробрасывает priority
- DAL/notifications: высокий приоритет = отдельный канал/маркировка

**Критерий готовности:** клиент пишет «бесит ваш сервис, верните деньги» → ассистент: 1) извиняется, 2) собирает контекст, 3) автоматически вызывает requestCallback с priority='high', 4) сообщает: «Менеджер свяжется в ближайшие 15 минут».

### P1.5. Анти-аннуляционные алерты [x] ВЫПОЛНЕНО 2026-04-11
**Размер:** M
**Зависимости:** нет (но проверить существующий cron `expiring-permits`)

**Что делаем.** Cron-job раз в сутки проверяет активные пропуски всех клиентов: если до окончания осталось 30/14/7/3/1 день — создаёт уведомление в `notifications`. Ассистент при `getMyContext` уже возвращает `expiringSoon` (это уже работает) — теперь это будет работать и проактивно через нотификации, которые видны в кабинете и в виджете чата.

**Бонус-расширение:** заглушка для будущей интеграции с РНИС API — если данные не передаются 4+ часов, то же самое уведомление с типом `rnis_silent`.

**Файлы:**
- `apps/web/src/app/api/cron/expiring-permits/route.ts` — проверить, что уже есть; расширить
- `apps/web/src/lib/dal/notifications.ts` — функция createPermitExpirationNotice
- Возможно — миграция для нового типа notifications.kind
- Опционально: новый /api/cron/rnis-health (заглушка с TODO)

**Критерий готовности:** запустить cron вручную → у тестового клиента с пропуском, истекающим через 14 дней, появилась notification «Пропуск NNN истекает 25 апреля». Откроет чат — ассистент сам это упомянет (через P1.2 + getMyContext).

## PHASE 2 — Углубление UX и качества

**Цель:** Phase 1 уже превратила чат в second pilot, Phase 2 делает его точнее и аккуратнее.

### P2.1. Confidence scoring и safety net [x] ВЫПОЛНЕНО 2026-04-11
Ассистент возвращает оценку уверенности в ответе. Если ниже порога — не отвечает уверенно, а зовёт менеджера. Реализация через структурированный output (Zod schema) с обязательным полем `confidence`.

### P2.2. Smart suggestions / autocomplete [x] ВЫПОЛНЕНО 2026-04-11
Пока клиент печатает — фронт сразу показывает 3–5 наиболее похожих вопросов из базы. По клику — мгновенный ответ из `short` без обращения к LLM. Реализация: отдельный лёгкий endpoint `/api/chat/suggest` с тем же `searchKnowledge`, но без LLM-вызова.

### P2.3. Calendar export для дедлайнов [x] ВЫПОЛНЕНО 2026-04-11
Один клик в кабинете → скачивание `.ics` с дедлайнами всех пропусков, ОСАГО, ДК, экспертиз клиента. Импорт в Google/Apple Calendar — клиент видит сроки в своём календаре.

### P2.4. Daily/weekly digest в MAX/email [x] ВЫПОЛНЕНО 2026-04-11
Раз в неделю клиенту приходит digest: «На этой неделе у вас 2 пропуска истекают, 1 заявка отклонена, накоплено штрафов на 7 500 ₽». Реализация: новый cron + использование существующих email/MAX каналов.

### P2.5. Manual override менеджером [x] ВЫПОЛНЕНО 2026-04-11 (требует применения миграции 0001_p2_5_chat_handovers.sql)
Менеджер видит активные чаты в админке и может «перехватить» — после перехвата ассистент молчит, отвечает только человек. Возвращение ассистента — отдельная команда.

## PHASE 3 — Качество и устойчивость

### P3.1. Шаблоны с slots [x] ВЫПОЛНЕНО 2026-04-11 (мульти-агент)
Для типовых сценариев (оформление пропуска для лизингового ТС, обжалование штрафа, разъяснение отказа) — структурированные шаблоны, в которые ассистент подставляет данные клиента. Делает ответы ровнее, снижает разброс.

### P3.2. Memory across sessions [x] ВЫПОЛНЕНО 2026-04-11 (мульти-агент)
Ассистент помнит факты о клиенте между сессиями: предпочтения каналов связи, типичные вопросы, упомянутые проблемы. Реализация: новая таблица `client_facts` (key-value), tools для записи и чтения. Никаких векторов на этом этапе.

### P3.3. Golden set + eval CI [x] ВЫПОЛНЕНО 2026-04-11 (мульти-агент, 97/97 тестов зелёные)
50–100 эталонных вопросов с эталонными ответами. На каждый PR с изменениями `system-prompt.ts`, `tools.ts` или `knowledge-base.ts` — автоматический прогон через GPT-4o-mini, оценка ответов, отчёт в PR. Защита от регрессий.

## PHASE 4 — Мультимодальность и голос (нужны ключи)

### P4.1. OCR СТС/ПТС/постановлений [x] ВЫПОЛНЕНО 2026-04-11 (graceful fallback)
**Нужно от пользователя:** установить `OPENAI_VISION_ENABLED=true` в env (используется тот же `OPENAI_API_KEY`, но на gpt-4o, а не mini — ×5 дороже по input).

Клиент кидает фото СТС → ассистент извлекает марку, модель, госномер, VIN, экокласс, max weight → автоматически создаёт vehicle в кабинете. Аналогично — фото постановления о штрафе → данные нарушения → пред-заполненная форма обжалования.

### P4.2. Voice push-to-talk [x] ВЫПОЛНЕНО 2026-04-11 (graceful fallback)
**Нужно от пользователя:** `YANDEX_SPEECHKIT_API_KEY` + `YANDEX_SPEECHKIT_FOLDER_ID` в env. Без ключей кнопка микрофона не рендерится — фейлсейф.

Кнопка микрофона в виджете → push-to-talk → распознавание → текст в чат → ответ озвучивается параллельно с генерацией. Барж-ин (прервать TTS своим голосом). Описано в [плане v2 § 9.2](2026-04-11-личный-кабинет-v2-реестры-и-ai-агент.md).

## PHASE 5 — Векторный RAG (отдельный трек)

**Когда запускать:** база знаний > 300 пунктов или появятся неструктурированные источники (КонсультантПлюс, методички, внутренние гайды).

**Что:** pgvector + `text-embedding-3-small`, гибридный retrieval (BM25 наших keyword-весов + cosine на embeddings, объединение через RRF), top-k = 5–8, опционально reranker (Cohere Rerank).

**Подробности:** см. [план v2 § 5.7](2026-04-11-личный-кабинет-v2-реестры-и-ai-агент.md).
