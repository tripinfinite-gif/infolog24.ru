**Дата создания:** 2026-04-10
**Статус:** Планирование
**Цель:** Серверная часть — API для ЛК клиентов/менеджеров, интеграция с CRM и платёжной системой, хранение файлов.
**Связан с:** [Редизайн сайта](2026-04-10-редизайн-сайта-infolog24.md) (Фаза 6), [Дизайн-система](2026-04-10-дизайн-система.md) (Фаза 4)

## Контекст и масштаб задачи

**Бизнес-метрики (для понимания нагрузки):**
- 600–800 заявок/мес → ~25–35 заявок/день
- ~150–200 клиентов/мес, ~1.6 пропуска на клиента
- 30–50 сотрудников (менеджеры + админы)
- Выручка ~17 млн/мес — бизнес работает, API должен быть надёжным

**Вывод по нагрузке:** Это не highload. Пиковая нагрузка — десятки одновременных пользователей, не тысячи. Архитектура должна быть **простой, надёжной и поддерживаемой**, а не масштабируемой на миллионы запросов.

## Выбор стека (исследование проведено)

### Архитектура: Next.js Route Handlers (без отдельного бэкенда)

**Решение:** Все API-маршруты реализуются как Route Handlers внутри Next.js App Router (`app/api/...`).

**Почему не отдельный бэкенд (NestJS, Fastify, Express):**
- Один деплой, один процесс, один репозиторий — проще поддерживать
- Next.js Route Handlers используют стандартный Web API (`Request`/`Response`)
- Для 25–35 заявок/день отдельный бэкенд — оверинжиниринг
- Server Components вызывают DAL напрямую (без HTTP round-trip) — быстрее
- Официальная рекомендация Next.js (2025): Route Handlers достаточны для публичных API, вебхуков, авторизации

**Паттерн Data Access Layer (DAL):**
```
src/lib/dal/           ← Вся работа с БД и внешними сервисами
├── applications.ts    ← Заявки
├── permits.ts         ← Пропуска
├── documents.ts       ← Документы
├── users.ts           ← Пользователи
└── notifications.ts   ← Уведомления
```
Server Components вызывают DAL напрямую. Route Handlers тоже вызывают DAL. Логика — в одном месте.

---

### База данных: PostgreSQL (self-hosted на Beget VPS)

**Почему PostgreSQL:**
- Стандарт индустрии, лучшая совместимость с Drizzle ORM
- Полный контроль над данными (152-ФЗ — персональные данные в РФ)
- Нет зависимости от облачного провайдера (PlanetScale убил free tier в 2024)
- Для 200 клиентов/мес PostgreSQL на одном VPS — с запасом на годы

**Альтернатива для dev/staging:** Neon (serverless PostgreSQL) — бесплатный tier, бранчинг баз по разработчикам.

---

### ORM: Drizzle ORM

**Почему Drizzle, а не Prisma:**
- ~7.4 KB (gzip) vs ~2 MB у Prisma — быстрый холодный старт
- SQL-like API — пишешь JOIN явно, видишь что будет в SQL
- Prisma 7 (2025) убрал Rust-движок, стал быстрее, но Drizzle по-прежнему легче
- `drizzle-kit` для миграций — зрелый инструмент
- Zero runtime dependencies

**Примечание:** Prisma 7 тоже приемлема, если команда с ней знакома. Но для нового проекта Drizzle — лучший выбор в 2026.

---

### Авторизация: Auth.js v5 (NextAuth v5) + JWT + RBAC

**Почему Auth.js v5:**
- ~2.5M скачиваний/неделю — стандарт для Next.js
- Lucia Auth **закрыт** (март 2025) — не вариант
- Clerk — отличный DX, но платный (10K MAU бесплатно, потом $$$)
- Auth.js v5: бесплатный, self-hosted, полный контроль

**Роли:** `client` | `manager` | `admin`
- Роль хранится в JWT через `callbacks.jwt`
- `middleware.ts` проверяет роль и маршрут — редирект или 403
- Дополнительная проверка в каждом Route Handler (не доверяем только middleware)

---

### Хранение файлов: Beget S3 + Presigned URLs

**Паттерн (best practice 2026):**
1. Клиент запрашивает presigned URL (`POST /api/upload/presign`)
2. Клиент загружает файл напрямую в S3 — не через Next.js сервер
3. Route Handler сохраняет метаданные в PostgreSQL
4. Скачивание — через presigned GET URL (с ограничением по времени)

**Почему Beget S3:**
- S3-совместимое хранилище от Beget — данные в РФ (152-ФЗ)
- Работает с `@aws-sdk/client-s3`
- Дёшево по сравнению с AWS S3
- Альтернатива: MinIO на том же VPS (больше контроля, но больше ops)

**Безопасность документов:**
- Валидация типа файла на сервере (не только MIME от клиента)
- Ограничение размера (10 MB для документов, 5 MB для изображений)
- Антивирус на загруженные файлы (ClamAV или облачный API) — опционально на старте

---

### CRM: Битрикс24 REST API + вебхуки

**Два типа интеграции:**
- **Inbound** (мы → Битрикс24): вызываем REST API по webhook URL, создаём лиды/сделки
- **Outbound** (Битрикс24 → мы): Битрикс24 шлёт POST на наш endpoint при событиях

**Важно:** Битрикс24 отправляет `application/x-www-form-urlencoded`, не JSON.

**Rate limit:** 2 запроса/сек на метод. При массовых операциях — использовать batch-методы.

---

### Платёжная система: ЮKassa (основная) + Т-Банк (резерв)

**ЮKassa (YooKassa):**
- Самый популярный платёжный шлюз в РФ
- Поддержка: банковские карты, СБП, SberPay, T-Pay, QIWI
- Официальный Node.js SDK: `yookassa`
- 54-ФЗ фискализация встроена (с 2026 НДС 22%, коды ставок 11/12)
- Вебхуки статусов платежей на наш endpoint

**СБП (Система быстрых платежей):**
- Важна для B2B-клиентов — многие перевозчики платят через СБП
- Поддерживается нативно через ЮKassa (`payment_method_data.type = 'sbp'`)

---

### Хостинг: Beget VPS (не shared hosting)

**Shared hosting Beget НЕ подходит для Next.js:**
- Node.js через Apache mod_passenger — не нативный Node.js сервер
- Нет PM2, нет persistent-процессов, нет WebSocket
- Лимит тела запроса 4.5 MB от Apache
- Нельзя запустить `next start` как постоянный процесс

**Beget VPS (рекомендация):**
- Ubuntu 22.04 + Node.js (через nvm) + PM2 + Nginx reverse proxy
- SSL через Let's Encrypt (certbot)
- PostgreSQL локально или через Beget DB
- 4 ядра / 8 GB RAM — ~2000–3000 руб/мес, хватит с запасом
- DDoS-защита от Beget
- Дата-центр в Москве — 152-ФЗ

## Фаза 3. API-маршруты (CRUD)

> Основные endpoints для работы ЛК.

### 3.1 Заявки (`/api/applications`)

- [ ] **3.1.1** `POST /api/applications` — создать заявку
  - Клиент: создаёт от своего имени
  - Менеджер: создаёт от имени клиента (указывает user_id)
  - Валидация: зона, тип пропуска, кол-во ТС
  - После создания → push в Битрикс24 (Фаза 5)

- [ ] **3.1.2** `GET /api/applications` — список заявок
  - Клиент: только свои
  - Менеджер: свои назначенные + неназначенные
  - Админ: все
  - Фильтры: status, zone, permit_type, date_range
  - Пагинация: cursor-based (не offset — стабильнее при добавлении новых записей)
  - Сортировка: created_at desc (по умолчанию)

- [ ] **3.1.3** `GET /api/applications/[id]` — детали заявки
  - Включает: документы, платежи, пропуска, лог действий
  - Проверка доступа: своя заявка или менеджер/админ

- [ ] **3.1.4** `PATCH /api/applications/[id]` — обновить статус/данные
  - Клиент: может отменить (status → cancelled), редактировать комментарий
  - Менеджер: может менять статус, назначать себя
  - Админ: полный доступ
  - Каждое изменение статуса → запись в activity_log + уведомление клиенту

- [ ] **3.1.5** `DELETE /api/applications/[id]` — soft delete (status → cancelled)
  - Только admin

### 3.2 Пропуска (`/api/permits`)

- [ ] **3.2.1** `GET /api/permits` — список пропусков пользователя
  - Фильтры: status (active|expired), zone, type
  - Для менеджера/админа: все или по user_id

- [ ] **3.2.2** `GET /api/permits/[id]` — детали пропуска
  - Включает: номер, зона, ТС, срок действия, связанная заявка

- [ ] **3.2.3** `POST /api/permits` — создать пропуск (менеджер/админ)
  - Привязка к заявке, заполнение номера и сроков
  - Статус заявки → approved

- [ ] **3.2.4** `GET /api/permits/check` — публичный endpoint (проверка пропуска)
  - По номеру пропуска или гос. номеру ТС
  - Возвращает: статус (действует/истёк), зона, срок

- [ ] **3.2.5** Cron-job: проверка истекающих пропусков
  - За 30 дней до окончания → уведомление клиенту
  - За 7 дней → повторное уведомление
  - После истечения → status → expired

### 3.3 Документы (`/api/documents`)

- [ ] **3.3.1** `POST /api/upload/presign` — получить presigned URL для загрузки
  - Валидация: тип файла (PDF, JPG, PNG), размер (≤10 MB)
  - Генерация уникального S3-ключа: `documents/{user_id}/{application_id}/{uuid}.{ext}`
  - Возвращает: presigned PUT URL + метаданные

- [ ] **3.3.2** `POST /api/documents` — сохранить метаданные после загрузки
  - Клиент подтверждает завершение загрузки
  - Сохраняем: file_name, file_url, file_size, mime_type, application_id

- [ ] **3.3.3** `GET /api/documents/[id]/download` — presigned GET URL для скачивания
  - Проверка доступа: владелец документа или менеджер/админ
  - URL с TTL 15 минут

- [ ] **3.3.4** `DELETE /api/documents/[id]` — удалить документ
  - Удаление из S3 + удаление записи из БД
  - Только владелец или админ

### 3.4 Пользователи (`/api/users`)

- [ ] **3.4.1** `GET /api/users/me` — профиль текущего пользователя
- [ ] **3.4.2** `PATCH /api/users/me` — обновить профиль (имя, телефон, компания)
- [ ] **3.4.3** `GET /api/users` — список пользователей (admin)
  - Фильтры: role, search (имя/email/телефон)
  - Пагинация
- [ ] **3.4.4** `GET /api/users/[id]` — профиль пользователя (менеджер/админ)
- [ ] **3.4.5** `PATCH /api/users/[id]` — изменить роль, заблокировать (admin)
- [ ] **3.4.6** `PATCH /api/users/me/password` — смена пароля

### 3.5 Уведомления (`/api/notifications`)

- [ ] **3.5.1** `GET /api/notifications` — список уведомлений текущего пользователя
  - Фильтр: is_read (true/false)
  - Пагинация

- [ ] **3.5.2** `PATCH /api/notifications/[id]/read` — пометить как прочитанное

- [ ] **3.5.3** `PATCH /api/notifications/read-all` — пометить все как прочитанные

- [ ] **3.5.4** `GET /api/notifications/unread-count` — количество непрочитанных (для бейджа в UI)

- [ ] **3.5.5** Сервис отправки уведомлений (внутренний, не endpoint):
  - Email: через SMTP (Beget mail или внешний сервис типа Resend)
  - SMS: через SMS-агрегатор (SMS.ru, SMS Aero) — для критичных уведомлений
  - Telegram-бот: опционально, для тех кто подключил
  - Push в ЛК: запись в таблицу notifications

## Фаза 8. Валидация, безопасность, обработка ошибок

> API должен быть устойчив к некорректным данным и атакам.

### 8.1 Валидация входных данных

- [ ] **8.1.1** Zod для валидации всех request body и query params:
  ```ts
  const createApplicationSchema = z.object({
    zone: z.enum(['mkad', 'ttk', 'sk']),
    permit_type: z.enum(['daily', 'night', 'temporary', 'annual']),
    vehicle_count: z.number().int().min(1).max(100),
    comment: z.string().max(1000).optional(),
  });
  ```
- [ ] **8.1.2** Утилита `validateRequest(schema, body)` — единообразный ответ 400 с деталями ошибки

### 8.2 Безопасность

- [ ] **8.2.1** Rate limiting: `rate-limiter-flexible` или Nginx limit_req
  - API: 100 req/min на IP
  - Auth endpoints: 10 req/min на IP (защита от brute-force)
  - Вебхуки: whitelist IP ЮKassa и Битрикс24

- [ ] **8.2.2** CORS: разрешить только свой домен

- [ ] **8.2.3** Заголовки безопасности (через Nginx или Next.js middleware):
  - `X-Content-Type-Options: nosniff`
  - `X-Frame-Options: DENY`
  - `Strict-Transport-Security`
  - CSP (Content Security Policy)

- [ ] **8.2.4** Защита от SQL injection: Drizzle ORM параметризует запросы автоматически, но запретить raw SQL без параметров

- [ ] **8.2.5** Логирование: все мутации (создание, обновление, удаление) → activity_log

### 8.3 Обработка ошибок

- [ ] **8.3.1** Единый формат ошибок API:
  ```json
  {
    "error": {
      "code": "VALIDATION_ERROR",
      "message": "Описание ошибки",
      "details": [...]
    }
  }
  ```
- [ ] **8.3.2** HTTP-коды: 400 (валидация), 401 (не авторизован), 403 (нет прав), 404 (не найден), 500 (внутренняя)
- [ ] **8.3.3** Глобальный error handler: перехват необработанных ошибок, логирование, ответ 500 без stack trace
