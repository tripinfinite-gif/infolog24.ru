# Кабинет партнёра — MVP с приёмом архива документов

**Дата:** 2026-04-11
**Статус:** Реализовано (фазы 1–4 полностью)
**Размер:** L (архитектура, ~12 файлов, server actions, миграции не нужны)
**Автор:** Claude Code (orchestrated, 4 параллельных агента)
**Связано:** [`2026-04-10-личный-кабинет-фронтенд.md`](2026-04-10-личный-кабинет-фронтенд.md) — клиентский ЛК (отдельный поток)

---

## Цель

Дать партнёрам (сторонним привлекающим клиентов) рабочий кабинет, в котором они могут:

1. **Зарегистрироваться** как партнёр и войти.
2. **Создать заявку на пропуск**, загрузив архив с документами клиента (ZIP).
3. Система **автоматически разбирает** архив, определяет тип каждого документа (ПТС / СТС / ВУ / доверенность / договор) по имени файла, складывает по местам, проверяет полноту.
4. Если каких-то документов **не хватает** — заявка переходит в статус `documents_pending`, партнёр видит, что нужно дозагрузить.
5. Если **всё на месте** — создаётся `order` со статусом `processing`, draft-`permit` со сроком действия, `partner_referral` со **сразу рассчитанной предварительной комиссией**.
6. Партнёр видит **список своих заявок** и может открыть **карточку заявки** с предварительной комиссией, статусом и списком документов.

---

## Архитектура решения

### Стек

- Auth: **Better Auth** (уже настроен) с email+password. После signUp серверный action `becomePartnerAction` обновляет `users.role = 'partner'` и сохраняет phone/company/inn.
- Защита кабинета: **middleware** (cookie-чек, Edge runtime) + **server-component layout guard** (полная проверка роли через `getSession()`).
- ZIP-разбор: **jszip** (новая зависимость, ^3.10.1).
- Загрузка файлов: прямой `PutObjectCommand` в Selectel S3 через новую функцию `putObject` в `lib/integrations/s3.ts`.
- Расчёт комиссии: 10% от базовой цены пропуска (mkad_day=8000, mkad_night=6000, ttk=12000, sk=18000, temp=4000).
- Создание сущностей: одна транзакция `db.transaction` на создание order + permit, потом загрузка файлов + insert documents + insert partner_referral вне транзакции.

## Что НЕ вошло в MVP (отложено)

- **Дозагрузка недостающих документов** — кнопка есть, server action не реализован, показывает toast «скоро появится». Следующий шаг.
- **OCR/AI-извлечение данных из PDF** — сейчас классификация ТОЛЬКО по имени файла. AI-извлечение номера ТС, ФИО водителя, реквизитов — следующая итерация (можно через `vercel-ai-sdk` + multimodal).
- **Реферальные ссылки** — `referralCode` генерируется и хранится, но публичной странички `?ref=KX2B7` пока нет.
- **Привязка к конечному клиенту** — сейчас `order.userId = partnerId`, т.е. партнёр сам владеет заявкой. Если бизнес-модель потребует, чтобы клиент тоже видел свой пропуск — нужен `referredUserId` и более сложная RBAC-логика.
- **Выплаты комиссии** — `partner_referrals.status` сейчас всегда `pending`. Изменение статуса (`confirmed`/`paid`) и интеграция с YooKassa Payouts — отдельная фаза.
- **Идемпотентность** — повторный сабмит формы создаст дубль заявки. Защита через nonce / debounce — следующая итерация.
- **Валидация диапазона дат** — `validFrom <= validUntil` zod не проверяет.
- **VIN** — нет жёсткой проверки длины 17 / формата.

## Как это работает (E2E flow)

1. Партнёр заходит на `/partner/register` → заполняет ФИО, email, телефон, компанию, ИНН, пароль → нажимает «Стать партнёром».
2. Better Auth создаёт аккаунт; сразу после — server action `becomePartnerAction` обновляет `users.role = 'partner'` и создаёт стартовую запись в `partner_referrals` с уникальным `referralCode`.
3. Редирект на `/partner` (дашборд). Layout guard убедится, что роль == `partner`.
4. На дашборде сверху — CTA «Новая заявка» → ведёт на `/partner/passes/new`.
5. Партнёр заполняет данные ТС (брэнд, модель, госномер, VIN), выбирает тип пропуска и зону, ставит даты, перетаскивает ZIP-архив в дропзону → нажимает «Отправить заявку».
6. `submitPartnerArchive`:
   - Проверяет роль (только partner).
   - Парсит формдату через zod.
   - Распаковывает ZIP в память (`jszip.loadAsync`), классифицирует каждый файл по имени (`classifyFileName`).
   - Считает, какие из обязательных документов (pts, sts, driver_license, power_of_attorney, +contract для СК) есть, каких не хватает.
   - Транзакция: создаёт vehicle (если нет под этим партнёром с таким госномером) → order (status = `processing` если всё на месте, иначе `documents_pending`) → draft permit с указанными датами.
   - Вне транзакции: загружает каждый файл в Selectel S3 через `putObject`, создаёт запись в `documents`.
   - Создаёт запись в `partner_referrals` с `commission = round(basePrice * 0.1)`.
7. Возвращает результат: orderId, статус, список missing, число uploaded/rejected, расчёт комиссии.
8. Форма показывает зелёный/жёлтый алерт + сумму комиссии + кнопку «К списку заявок».
9. На `/partner/passes` партнёр видит таблицу со всеми своими заявками (дата, тип/зона, ТС, статус, комиссия).
10. Клик «Открыть» → `/partner/passes/[id]` → детали + список загруженных документов + если чего-то не хватает — алерт со списком.

## Что нужно для запуска на стенде

1. PostgreSQL запущен, миграция `0000_rich_captain_universe.sql` применена (`pnpm db:migrate`).
2. Заполнены env-переменные:
   - `DATABASE_URL`
   - `BETTER_AUTH_SECRET` (≥32 символа)
   - `S3_ENDPOINT`, `S3_ACCESS_KEY`, `S3_SECRET_KEY`, `S3_BUCKET` (Selectel)
   - `NEXT_PUBLIC_APP_URL`
3. `pnpm install` (jszip уже в package.json + lockfile обновлён).
4. `pnpm dev` → `/partner/register` → создать тестового партнёра → загрузить тестовый ZIP с файлами `pts.pdf`, `sts.pdf`, `voditel.pdf`, `dover.pdf`.

## Известные ограничения / технический долг

- Без unique-индекса на `partner_referrals.referral_code` — теоретическая коллизия (но шанс ничтожный, 32^8).
- Между `signUp.email` и `becomePartnerAction` есть окно: если action упадёт, аккаунт останется client. UX: показывает ошибку, при повторном входе пользователя не пустит. Нужен endpoint «доделать партнёрство».
- `requireRole` читает роль из `(session.user as any).role`, что зависит от того, что Better Auth включает кастомные поля users в session. Стоит проверить на реальном стенде.
- Если S3 недоступен (нет credentials), submitPartnerArchive создаст заявку без файлов. Возможно, лучше отклонять заявку целиком или ставить специальный статус.
