## Цель

Под каждым ответом ассистента появляются кликабельные кнопки (Action Cards) — конкретные действия: продлить пропуск, загрузить документ, открыть форму новой заявки, связаться с менеджером, посмотреть сущность. Клик переводит на нужную страницу с предзаполнением или вызывает API.

---

## Архитектура

```
[ Tool execute ]
       ↓ возвращает { ...result, actions?: ActionCard[] }
[ /api/chat/route.ts ]
       ↓ стримит ответ + tool result + actions
[ chat-widget → useChat → messages ]
       ↓ message.parts с tool result содержат actions
[ ChatMessages → ToolResultCard ]
       ↓ под результатом рендерится <ActionCardList />
[ Клик ]
       ↓
[ resolveActionUrl(action) → router.push() ИЛИ inline-handler → API call ]
```

### Phase 1 — Параллельные специализированные агенты

После Phase 0 спавним 3 агента одновременно. Каждый работает в своём файле, конфликтов нет.

#### 🤖 Агент A — Tools integration
**Subagent:** general-purpose
**Файлы (только эти):**
- `apps/web/src/lib/chat/tools.ts`

**Задача:** обновить tools так, чтобы они возвращали поле `actions: ActionCard[]` в результатах. Конкретные tools и какие actions добавить:
- `getMyContext` → если есть `expiringSoon` → action `extend_permit` для каждого + `view_permit`
- `getMyPermits` → для каждого `view_permit`, для истекающих в 30 дней — `extend_permit`
- `getMyOrders` → для каждой заявки `view_order`, для статуса `documents_pending` — `upload_document`
- `getMyVehicles` → для каждой `view_vehicle`, и общая `create_order`
- `checkPermitStatus` → если найдена — `view_order`
- `getPriceCalculation` → `create_order` с предзаполнением (если auth)
- `requestCallback` → нет actions, это финальное действие

**Контракт:** действия — это **подсказки модели для UI**, не побочные эффекты. Tools не должны выполнять навигацию.

**Правила:**
- Использовать только `ActionCard` тип из `@/lib/chat/action-cards`.
- Не менять никакие другие файлы.
- Сохранить обратную совместимость других полей в результатах (только дополнить, не убирать).
- Прогнать `pnpm exec tsc --noEmit` в `apps/web/` перед сдачей.

**Возврат:** список изменённых tools + diff-summary в 10–15 строк.

#### 🤖 Агент B — UI rendering
**Subagent:** general-purpose
**Файлы (только эти):**
- `apps/web/src/components/chat/action-card-list.tsx` (новый)
- `apps/web/src/components/chat/chat-messages.tsx` (модификация: рендерит ActionCardList под tool result когда есть actions)

**Задача:** создать компонент `ActionCardList` который:
- Принимает `actions: ActionCard[]`
- Рендерит каждую как кнопку с иконкой (`lucide-react`) и текстом из `getActionLabel`
- Кнопки `<Link>` (next/link) когда есть href, или `<Button>` с onClick для API-действий
- Стили: tailwind, соответствует существующему дизайну чата (rounded-xl, border, shadow-sm)
- Адаптивность: на мобильных — вертикальный стек, на десктопе — горизонтальный wrap
- Анимация появления — лёгкая

Интеграция в chat-messages.tsx: после блока `ToolResultCard` в каждом message с tool parts проверить, есть ли в result `actions` массив, и отрендерить `<ActionCardList actions={...} />`.

**Правила:**
- Использовать только `ActionCard` тип из `@/lib/chat/action-cards`.
- Использовать `resolveActionHref(action)` и `getActionLabel(action)` из того же модуля.
- Не трогать никакие другие файлы.
- Прогнать `pnpm exec tsc --noEmit` и `pnpm exec next lint --quiet` в `apps/web/`.

**Возврат:** список созданных/изменённых файлов + 5–10 строк описания структуры компонента.

## Принципы оркестрации

1. **Изолированность файлов.** Каждый агент трогает только свои файлы. Конфликтов нет.
2. **Самодостаточные промпты.** Агент не видит мою переписку — в его промпте есть всё нужное: контекст проекта, путь к типам, точные файлы, критерии готовности.
3. **Проверка после каждого агента.** Я читаю результат, не доверяю слепо.
4. **Идемпотентность.** Если агент сломал что-то — могу повторно спавнить fix-агента или сделать сам.
5. **Видимость через TodoWrite.** Каждый агент = todo item, статус обновляется в реальном времени.

1. **3 агента в параллель работают чисто, если файловые границы строгие.** Конфликтов не было ни одного. tools.ts (Агент A), action-card-list.tsx + chat-messages.tsx (Агент B), system-prompt.ts (Агент C) — пересечений нет.
2. **Самодостаточные промпты сработали.** Каждый агент получил абсолютные пути, типы фундамента, точные критерии готовности и список «что не трогать». Никто не выходил за рамки.
3. **Phase 0 — критически важна.** Если бы я не написал action-cards.ts заранее, агенты выдумывали бы свои типы и потом не сошлись бы. Фундамент-первым — золотое правило мульти-агентной оркестрации.
4. **Финальный typecheck — обязателен на стороне оркестратора.** Каждый агент сам прогоняет tsc у себя, но финальная проверка после интеграции — единственная гарантия, что комбинация работает.
