## 2. Обоснование ключевых решений

### 2.1. Drizzle ORM вместо Prisma

**Почему Drizzle:**
- **Размер бандла:** ~5 KB vs 40+ KB у Prisma — критично для serverless/edge
- **Производительность:** прямая генерация SQL без промежуточного Rust-движка
- **Edge Runtime:** полная совместимость (Prisma получила это только в v7)
- **SQL-first подход:** схема описывается TypeScript-объектами, максимально близко к SQL
- **Enterprise-backing:** в марте 2026 команду Drizzle приобрёл PlanetScale

**Почему не Prisma:**
- Тяжелее (40+ KB бандл, Rust engine до v7)
- Историческая проблема с cold starts на serverless
- Auth.js/NextAuth официально отказался от Prisma-адаптера в пользу Drizzle

**Миграции:** `drizzle-kit` генерирует SQL-файлы, которые можно редактировать перед применением — полный контроль.

### 2.2. Better Auth вместо NextAuth/Clerk

**Почему Better Auth:**
- **2FA, Passkeys, RBAC** — встроенные плагины, не нужно писать самому
- **Self-hosted** — данные на нашем сервере, не зависим от SaaS
- **Бесплатно** — MIT-лицензия, без лимитов на пользователей
- **Официальная рекомендация:** в сентябре 2025 команда Auth.js присоединилась к Better Auth. Auth.js → legacy, Better Auth → рекомендуемый стандарт для новых проектов
- **Полная типизация TypeScript**

**Почему не Clerk:**
- Платный ($0.02/MAU после 10K), данные на серверах Clerk (США)
- Зависимость от внешнего SaaS — риск блокировки для РФ-бизнеса

**Почему не NextAuth/Auth.js:**
- Legacy-проект, только security-патчи, не рекомендуется для новых проектов

### 2.3. Coolify + Selectel вместо Vercel

**Почему не Vercel:**
- Серверы в США/ЕС — задержки для российских пользователей
- Риск блокировки/ограничения для РФ-компании
- Ценообразование: $20/чел/мес Pro, лимиты bandwidth, function size, timeout
- Данные клиентов (документы, сканы) — необходимо хранить в РФ (152-ФЗ)

**Почему Coolify:**
- Vercel-подобный DX: git push deploy, SSL, preview deployments, dashboard
- Полный контроль: WebSocket, filesystem, long-running connections
- Фиксированная стоимость VPS вместо pay-per-use
- 52K+ GitHub stars, 325K+ пользователей
- Возможность разместить на любом российском хостинге

**Почему Selectel:**
- 6 дата-центров Tier III в Москве и СПб
- Managed PostgreSQL, S3-совместимое хранилище
- Соответствие 152-ФЗ (персональные данные в РФ)
- VPS от 800 руб/мес, масштабируемость
- Надёжная репутация, поддержка на русском

### 2.4. Монорепо (Turborepo + pnpm)

**Почему монорепо, а не отдельные репозитории:**
- Единая кодовая база для сайта, ЛК, бота, общих пакетов
- Общие типы, утилиты, конфиги — без дублирования
- Атомарные коммиты: изменение API + фронт + бот в одном PR
- Turborepo: параллельная сборка, кэширование, инкрементальные билды

**Почему Turborepo, а не Nx:**
- Проще в настройке, идеален для JS/TS-проектов
- Минимум конфигурации, нативная интеграция с pnpm
- Nx избыточен для нашего масштаба (он для polyglot-монолитов)

### 2.5. AI-чатбот: кастомный вместо SaaS-платформы

**Почему кастомный (Vercel AI SDK):**
- Интеграция в единую кодовую базу (сайт + Telegram — один бэкенд)
- Полный контроль над промптами и поведением бота
- В 3–10 раз дешевле готовых платформ при 300+ разговорах/день
- Нет зависимости от российских SaaS (SaluteBot и пр.)

**Расчёт стоимости (300 разговоров/день):**

| Модель | Стоимость/мес |
|--------|--------------|
| GPT-4o mini | ~$13 |
| Claude Haiku 4.5 | ~$97 |
| Carrot Quest (аналог) | от 3 600 руб + AI |
| Intercom + Fin | $9 000+ |

**Рекомендация:** GPT-4o mini для старта (дёшево + хорошее качество русского), Claude Haiku 4.5 для premium-качества. Vercel AI SDK позволяет переключаться между провайдерами одной строкой.

**RAG на старте не нужен:** вся база знаний (виды пропусков, цены, процесс, FAQ) помещается в system prompt (1500–2500 токенов). RAG подключим когда документов станет > 50–100.

### 2.6. YooKassa для платежей

- Единственное серьёзное решение для приёма платежей в РФ
- Рекурентные платежи (автопродление пропусков)
- TypeScript SDK: `yookassa-ts-sdk` (неофициальный, но полноценный)
- Webhooks для отслеживания статуса
- Эквайринг карт, SBP, электронные кошельки

### 2.7. grammY для Telegram-бота

**Почему grammY, а не Telegraf:**
- Легковесный, serverless/edge-ready
- Нативная поддержка webhook через адаптеры (в т.ч. Next.js)
- Современный API, отличная документация
- Для нашего объёма (уведомления + приём документов) — более чем достаточен
- Telegraf тяжелее и избыточен для наших задач

### Маршруты API (ключевые)

| Метод | Путь | Описание |
|-------|------|----------|
| `*` | `/api/auth/[...all]` | Better Auth — login, register, 2FA, sessions |
| `POST` | `/api/chat` | AI-чатбот (Vercel AI SDK, стриминг) |
| `POST` | `/api/telegram/webhook` | Входящие сообщения Telegram |
| `POST` | `/api/yookassa/webhook` | Уведомления о платежах |
| `GET/POST` | `/api/orders` | Список / создание заявок |
| `GET/PATCH` | `/api/orders/[id]` | Детали / обновление заявки |
| `GET/POST` | `/api/vehicles` | Список / добавление ТС |
| `POST` | `/api/documents` | Загрузка документов (→ S3) |
| `GET` | `/api/permits` | Список пропусков клиента |
| `POST` | `/api/payments/create` | Создание платежа (YooKassa) |
| `GET` | `/api/notifications` | Уведомления пользователя |

### 5.2. Рекомендуемая конфигурация VPS

| Среда | CPU | RAM | Диск | Стоимость |
