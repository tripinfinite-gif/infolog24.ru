## 1. Docker: оптимизация образов

### Проблема

План использует Nixpacks (автоопределение). Nixpacks — удобно для старта, но:
- Нет контроля над слоями и кэшированием
- Образы получаются тяжёлые (500+ MB вместо 150-200 MB)
- Нет multi-stage build — в production-образе остаются devDependencies
- Нет `.dockerignore` — в контекст попадает всё (node_modules, .git, .business)

### Рекомендация: custom Dockerfile с multi-stage build

```dockerfile
# ---- Stage 1: Dependencies ----
FROM node:24-alpine AS deps
RUN corepack enable && corepack prepare pnpm@latest --activate
WORKDIR /app
COPY pnpm-lock.yaml pnpm-workspace.yaml package.json ./
COPY apps/web/package.json ./apps/web/
COPY packages/*/package.json ./packages/
RUN pnpm install --frozen-lockfile --prod=false

# ---- Stage 2: Build ----
FROM node:24-alpine AS builder
RUN corepack enable && corepack prepare pnpm@latest --activate
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY --from=deps /app/apps/web/node_modules ./apps/web/node_modules
COPY . .
ENV NEXT_TELEMETRY_DISABLED=1
RUN pnpm --filter web build

# ---- Stage 3: Production ----
FROM node:24-alpine AS runner
RUN corepack enable && corepack prepare pnpm@latest --activate
WORKDIR /app
ENV NODE_ENV=production
ENV NEXT_TELEMETRY_DISABLED=1

RUN addgroup --system --gid 1001 nodejs && \
    adduser --system --uid 1001 nextjs

COPY --from=builder /app/apps/web/.next/standalone ./
COPY --from=builder /app/apps/web/.next/static ./apps/web/.next/static
COPY --from=builder /app/apps/web/public ./apps/web/public

USER nextjs
EXPOSE 3000
ENV PORT=3000
ENV HOSTNAME="0.0.0.0"

CMD ["node", "apps/web/server.js"]
```

### Обязательно: `.dockerignore`

```
.git
.business
node_modules
.next
*.md
tests
playwright-report
k6
.env*
```

### Обязательно: Next.js standalone output

В `next.config.ts` добавить:
```ts
output: 'standalone'
```

Это уменьшает production-образ с ~500 MB до ~150 MB.

### Действия

- [ ] Создать `Dockerfile` в корне монорепо с multi-stage build
- [ ] Создать `.dockerignore`
- [ ] Включить `output: 'standalone'` в next.config.ts
- [ ] В Coolify переключить build pack с Nixpacks на Dockerfile
- [ ] Замерить размер образа (целевой: < 200 MB)

## 2. Coolify: конфигурация и best practices

### Проблемы в текущем плане

1. **Порт 8000 Coolify dashboard** — план говорит "закрыть порт 8000, доступ через домен", но Cloudflare proxy с Full (strict) SSL может конфликтовать с Coolify SSL. Для `admin.infolog24.ru` выбран DNS-only — это правильно, но нужно явно указать.

2. **Нет ресурсных лимитов** — Coolify позволяет задать CPU/memory limits для контейнеров. Без них один контейнер может съесть всю память VPS.

3. **Нет стратегии rollback** — описан только автодеплой, но нет инструкции как откатить.

4. **Preview deployments** — упомянуты, но не описано как настроить wildcard SSL для `pr-{N}.infolog24.ru`.

### Рекомендации

**Ресурсные лимиты в Coolify:**

| Сервис | CPU limit | Memory limit | Memory reservation |
|--------|-----------|-------------|-------------------|
| Next.js (prod) | 2 cores | 2 GB | 512 MB |
| Next.js (staging) | 1 core | 1 GB | 256 MB |
| PostgreSQL | 1.5 cores | 3 GB | 1 GB |
| Redis | 0.5 core | 512 MB | 128 MB |
| Uptime Kuma | 0.25 core | 256 MB | 64 MB |

**Rollback-стратегия:**

1. Coolify хранит предыдущие образы — rollback через UI (кнопка "Rollback")
2. Дополнительно: перед деплоем тегировать образ `latest-stable`
3. Git tags для каждого production-релиза: `v1.0.0`, `v1.0.1`
4. Скрипт быстрого отката:
   ```bash
   # На VPS:
   docker tag app:latest app:rollback-$(date +%Y%m%d)
   ```

**Wildcard SSL для preview deployments:**
- Coolify v4 поддерживает Cloudflare DNS challenge для wildcard-сертификатов
- Настроить: Coolify → Settings → SSL → Cloudflare API Token → `*.infolog24.ru`

### Действия

- [ ] Настроить resource limits для всех контейнеров
- [ ] Документировать процедуру rollback (не только "через UI")
- [ ] Настроить wildcard SSL через Cloudflare DNS challenge
- [ ] Добавить Docker healthcheck в Dockerfile (не только в Coolify):
  ```dockerfile
  HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
    CMD wget --no-verbose --tries=1 --spider http://localhost:3000/api/health || exit 1
  ```

## 3. Zero-downtime deployments

### Проблема

План упоминает "zero-downtime из коробки" у Coolify, но не описывает конкретную стратегию. В реальности Coolify использует rolling update, который требует правильной настройки приложения.

### Что нужно для реального zero-downtime

**3.1. Graceful shutdown в Next.js:**

```typescript
// server.ts (если custom server) или через next.config.ts
process.on('SIGTERM', () => {
  console.log('SIGTERM received, shutting down gracefully...');
  // Завершить текущие запросы, закрыть соединения БД
  server.close(() => {
    process.exit(0);
  });
  // Force kill через 30s
  setTimeout(() => process.exit(1), 30000);
});
```

**3.2. Coolify deploy strategy:**

В Coolify → Application → Advanced → Deployment:
- Strategy: **Rolling Update** (не Recreate!)
- Health check grace period: 30s
- Startup probe: `/api/health`, interval 5s, failure threshold 10

**3.3. Connection draining:**

- Настроить `stop_grace_period: 30s` в Docker-конфигурации
- Coolify передаёт SIGTERM → приложение завершает текущие запросы → новый контейнер берёт трафик

**3.4. Database connection pooling:**

При rolling update оба контейнера (старый и новый) подключены к БД одновременно. Нужен connection pooling:

```typescript
// drizzle config
export const db = drizzle(pool, {
  // pool: { max: 10 } — не больше 10 соединений на контейнер
});
```

Или использовать PgBouncer:
```yaml
# docker-compose в Coolify
pgbouncer:
  image: edoburu/pgbouncer
  environment:
    DATABASE_URL: postgresql://app:***@postgres:5432/infologistic
    POOL_MODE: transaction
    MAX_CLIENT_CONN: 100
    DEFAULT_POOL_SIZE: 20
```

### Действия

- [ ] Добавить graceful shutdown handler
- [ ] Настроить rolling update в Coolify
- [ ] Добавить connection pooling (PgBouncer или настройка пула в Drizzle)
- [ ] Протестировать zero-downtime деплой на staging (нагрузка k6 во время деплоя)

## 4. Стратегия миграций базы данных

### Проблема

В плане миграции запускаются вручную через `pnpm db:migrate` "через Coolify Execute". Это:
- Не автоматизировано
- Можно забыть
- Миграции не привязаны к деплою
- Нет стратегии для breaking changes (удаление колонок, переименование таблиц)

### Рекомендация: автоматические миграции при деплое

**Вариант A: Pre-deploy hook в Coolify**

Coolify поддерживает pre-deploy commands:
```
# Coolify → Application → Advanced → Pre-deploy
cd /app && node -e "require('./migrate.js')"
```

**Вариант B: Init container / entrypoint**

```dockerfile
# entrypoint.sh
#!/bin/sh
echo "Running migrations..."
npx drizzle-kit migrate
echo "Starting application..."
exec node apps/web/server.js
```

**Правила безопасных миграций (для zero-downtime):**

| Операция | Безопасно? | Как сделать безопасно |
|----------|-----------|---------------------|
| Добавить колонку (nullable) | Да | Просто добавить |
| Добавить колонку (NOT NULL) | Нет | 1) Добавить nullable → 2) заполнить → 3) ALTER NOT NULL |
| Удалить колонку | Нет | 1) Убрать из кода → 2) деплой → 3) DROP колонку в следующем деплое |
| Переименовать колонку | Нет | 1) Создать новую → 2) скопировать данные → 3) удалить старую (через 2 деплоя) |
| Добавить индекс | Осторожно | `CREATE INDEX CONCURRENTLY` (Drizzle не всегда поддерживает — использовать raw SQL) |
| Изменить тип колонки | Нет | Через промежуточную колонку |

**Rollback миграций:**

- Для каждой миграции писать и `up`, и `down`
- Drizzle ORM не имеет встроенного down — держать отдельные rollback-скрипты:
  ```
  migrations/
    0001_create_users.sql
    0001_create_users.rollback.sql
    0002_add_phone_column.sql
    0002_add_phone_column.rollback.sql
  ```

### Действия

- [ ] Автоматизировать миграции при деплое (pre-deploy hook или entrypoint)
- [ ] Написать гайдлайн безопасных миграций для команды
- [ ] Создавать rollback-скрипты для каждой миграции
- [ ] Тестировать миграции на staging перед production
- [ ] Добавить в CI: проверку, что миграция применяется корректно (на тестовой БД)

## 5. Верификация бэкапов

### Проблема

В плане:
- Проверка бэкапов — "ежемесячно, вручную"
- Скрипт удаления старых бэкапов через `awk | head | xargs` — ненадёжный (парсит вывод `aws s3 ls`, который может меняться)
- Нет алерта при сбое бэкапа
- Нет проверки целостности дампа
- Redis бэкап "опционально" — должен быть обязательным

### Рекомендации

**5.1. Автоматическая верификация бэкапа:**

```bash
#!/bin/bash
# /opt/scripts/backup-db.sh (улучшенная версия)
set -euo pipefail

TIMESTAMP=$(date +%Y-%m-%d_%H-%M)
FILENAME="infologistic_${TIMESTAMP}.sql.gz"
BACKUP_DIR="/tmp/backups"
S3_ENDPOINT="https://s3.storage.selcloud.ru"
S3_BUCKET="s3://infologistic-backups/db"
TELEGRAM_BOT_TOKEN="${TELEGRAM_BOT_TOKEN}"
TELEGRAM_CHAT_ID="${TELEGRAM_ALERT_CHAT_ID}"

mkdir -p "$BACKUP_DIR"

send_alert() {
  curl -s -X POST "https://api.telegram.org/bot${TELEGRAM_BOT_TOKEN}/sendMessage" \
    -d "chat_id=${TELEGRAM_CHAT_ID}" \
    -d "text=$1" \
    -d "parse_mode=HTML"
}

# 1. Дамп
if ! docker exec coolify-postgresql pg_dump -U app --format=custom infologistic > "${BACKUP_DIR}/${FILENAME%.gz}"; then
  send_alert "BACKUP FAILED: pg_dump error at $(date)"
  exit 1
fi

# 2. Проверка целостности дампа
if ! docker exec coolify-postgresql pg_restore --list "${BACKUP_DIR}/${FILENAME%.gz}" > /dev/null 2>&1; then
  send_alert "BACKUP CORRUPT: pg_restore --list failed at $(date)"
  exit 1
fi

# 3. Сжатие
gzip "${BACKUP_DIR}/${FILENAME%.gz}"

# 4. Проверка размера (должен быть > 1KB, иначе пустой дамп)
FILE_SIZE=$(stat -f%z "${BACKUP_DIR}/${FILENAME}" 2>/dev/null || stat -c%s "${BACKUP_DIR}/${FILENAME}")
if [ "$FILE_SIZE" -lt 1024 ]; then
  send_alert "BACKUP SUSPICIOUS: file size ${FILE_SIZE} bytes at $(date)"
  exit 1
fi

# 5. Загрузка в S3
aws s3 cp "${BACKUP_DIR}/${FILENAME}" "${S3_BUCKET}/${FILENAME}" \
  --endpoint-url "$S3_ENDPOINT"

# 6. Проверка что файл в S3
if ! aws s3 ls "${S3_BUCKET}/${FILENAME}" --endpoint-url "$S3_ENDPOINT" > /dev/null; then
  send_alert "BACKUP S3 UPLOAD FAILED at $(date)"
  exit 1
fi

# 7. Ротация: удалить файлы старше 30 дней
aws s3 ls "${S3_BUCKET}/" --endpoint-url "$S3_ENDPOINT" \
  | awk '{print $4}' \
  | sort \
  | while read -r file; do
      file_date=$(echo "$file" | grep -oP '\d{4}-\d{2}-\d{2}')
      if [ -n "$file_date" ]; then
        days_old=$(( ($(date +%s) - $(date -d "$file_date" +%s)) / 86400 ))
        if [ "$days_old" -gt 30 ]; then
          aws s3 rm "${S3_BUCKET}/${file}" --endpoint-url "$S3_ENDPOINT"
        fi
      fi
    done

# 8. Очистка
rm -f "${BACKUP_DIR}/${FILENAME}"

# 9. Уведомление об успехе (опционально, раз в неделю)
DAY_OF_WEEK=$(date +%u)
if [ "$DAY_OF_WEEK" -eq 1 ]; then
  send_alert "Backup OK: ${FILENAME} (${FILE_SIZE} bytes)"
fi
```

**5.2. Использовать pg_dump --format=custom:**
- Поддерживает `pg_restore --list` для проверки целостности
- Сжатие встроено (не нужен отдельный gzip)
- Позволяет восстанавливать отдельные таблицы

**5.3. Еженедельная автоматическая проверка восстановления:**

```bash
#!/bin/bash
# /opt/scripts/verify-backup.sh (запускать еженедельно)
# Скачивает последний бэкап и проверяет, что он восстанавливается

LATEST=$(aws s3 ls s3://infologistic-backups/db/ --endpoint-url https://s3.storage.selcloud.ru \
  | sort | tail -1 | awk '{print $4}')

aws s3 cp "s3://infologistic-backups/db/${LATEST}" /tmp/verify-backup.sql.gz \
  --endpoint-url https://s3.storage.selcloud.ru

# Восстановить в отдельную тестовую БД
docker exec coolify-postgresql createdb -U app infologistic_verify 2>/dev/null || true
gunzip -c /tmp/verify-backup.sql.gz | docker exec -i coolify-postgresql psql -U app infologistic_verify

# Проверить количество записей в ключевых таблицах
USERS=$(docker exec coolify-postgresql psql -U app -t -c "SELECT count(*) FROM users" infologistic_verify)
ORDERS=$(docker exec coolify-postgresql psql -U app -t -c "SELECT count(*) FROM orders" infologistic_verify)

# Очистить
docker exec coolify-postgresql dropdb -U app infologistic_verify

# Алерт если таблицы пустые
if [ "$USERS" -lt 1 ] || [ "$ORDERS" -lt 1 ]; then
  # отправить алерт в Telegram
  echo "BACKUP VERIFICATION FAILED"
  exit 1
fi

rm /tmp/verify-backup.sql.gz
```

**5.4. Redis бэкап — обязательный, не опциональный:**

Redis хранит сессии и кэш. Потеря сессий = все пользователи вылетят из ЛК. Минимум: ежедневный RDB snapshot → S3.

- [ ] Сделать Redis-бэкап обязательным

## 6. Мониторинг: метрики и алерты

### Проблема

План предлагает только Uptime Kuma (HTTP-пинги) + кастомный endpoint `/api/admin/metrics`. Это минимум для первой версии, но недостаточно для production-сервиса с платежами и персональными данными.

Отсутствует:
- Мониторинг ресурсов VPS (CPU, RAM, диск, сеть)
- Мониторинг Docker-контейнеров
- Алерты по порогам (диск 80%, CPU 90%)
- Мониторинг сертификатов SSL
- Мониторинг длительности запросов (не только доступность)
- Логирование в структурированном формате
- Error tracking (Sentry упомянут во втором плане, но не в первом — разрозненность)

### Рекомендация: трёхуровневый мониторинг

**Уровень 1 — Доступность (Uptime Kuma):**
- Уже описан в плане — ОК
- Добавить: мониторинг SSL-сертификата (Uptime Kuma поддерживает)
- Добавить: мониторинг DNS resolution
- Добавить: мониторинг S3 (доступность endpoint)

**Уровень 2 — Ресурсы VPS (node_exporter + Prometheus + Grafana):**

Для VPS с бизнес-данными Prometheus + Grafana — не overkill, а необходимость. Всё ставится через Coolify:

```yaml
# Coolify → Docker Compose
services:
  prometheus:
    image: prom/prometheus:latest
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml
    ports:
      - "9090:9090"

  grafana:
    image: grafana/grafana:latest
    ports:
      - "3001:3000"
    environment:
      - GF_SECURITY_ADMIN_PASSWORD=${GRAFANA_PASSWORD}

  node-exporter:
    image: prom/node-exporter:latest
    pid: host
    volumes:
      - /proc:/host/proc:ro
      - /sys:/host/sys:ro
    command:
      - '--path.procfs=/host/proc'
      - '--path.sysfs=/host/sys'

  cadvisor:
    image: gcr.io/cadvisor/cadvisor:latest
    volumes:
      - /:/rootfs:ro
      - /var/run:/var/run:ro
      - /sys:/sys:ro
      - /var/lib/docker/:/var/lib/docker:ro
```

Если ресурсов VPS недостаточно для полного стека, альтернатива — **Beszel** (лёгкий агент мониторинга, 10 MB RAM):
```bash
# На VPS
curl -sL https://get.beszel.dev | bash
```

**Уровень 3 — Приложение (Sentry + структурированные логи):**

- Sentry — для ошибок (уже во втором плане)
- Структурированные логи через `pino`:
  ```typescript
  import pino from 'pino';
  const logger = pino({ level: 'info' });
  
  // Вместо console.log
  logger.info({ orderId, userId, action: 'order_created' }, 'New order created');
  logger.error({ err, orderId }, 'Order creation failed');
  ```

**Критические алерты (Telegram):**

| Событие | Порог | Действие |
|---------|-------|----------|
| Сайт недоступен | 2 проверки подряд | Немедленный алерт |
| CPU > 85% | 5 минут | Предупреждение |
| RAM > 90% | 1 минута | Критический алерт |
| Диск > 80% | моментально | Предупреждение + автоочистка логов |
| SSL истекает < 7 дней | ежедневно | Предупреждение |
| Бэкап не выполнен | по расписанию | Критический алерт |
| 5xx ошибки > 5/мин | 1 минута | Критический алерт |
| Время ответа p95 > 3s | 5 минут | Предупреждение |

## 7. CI pipeline: оптимизация

### Проблема

1. **E2E-тесты блокируют lint** — в `ci.yml` e2e зависит от lint-and-type-check (`needs`). Это увеличивает время CI. Lint и E2E-тесты могут запускаться параллельно.

2. **Нет кэширования Playwright-браузеров** — каждый прогон скачивает ~400 MB браузеров.

3. **Нет кэширования Next.js build** — .next/cache не сохраняется между запусками.

4. **Post-deploy check** использует `sleep 120` — хрупкий подход. Нужен polling.

5. **Два плана описывают CI/CD независимо** — дублирование и расхождения.

### Рекомендация: оптимизированный CI pipeline

```yaml
name: CI
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  # --- Параллельные быстрые проверки ---
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 24
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm lint

  typecheck:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 24
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm typecheck

  unit-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 24
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm test

  # --- E2E параллельно с lint (не зависит от него) ---
  e2e:
    runs-on: ubuntu-latest
    needs: [unit-tests]  # только от unit-tests, не от lint
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 24
          cache: pnpm
      - run: pnpm install --frozen-lockfile

      # Кэширование Playwright-браузеров
      - name: Cache Playwright browsers
        uses: actions/cache@v4
        id: playwright-cache
        with:
          path: ~/.cache/ms-playwright
          key: playwright-${{ runner.os }}-${{ hashFiles('pnpm-lock.yaml') }}

      - name: Install Playwright
        if: steps.playwright-cache.outputs.cache-hit != 'true'
        run: pnpm exec playwright install --with-deps

      - name: Install Playwright deps only
        if: steps.playwright-cache.outputs.cache-hit == 'true'
        run: pnpm exec playwright install-deps

      # Кэширование Next.js build
      - name: Cache Next.js build
        uses: actions/cache@v4
        with:
          path: apps/web/.next/cache
          key: nextjs-${{ runner.os }}-${{ hashFiles('pnpm-lock.yaml') }}-${{ hashFiles('apps/web/src/**') }}
          restore-keys: |
            nextjs-${{ runner.os }}-${{ hashFiles('pnpm-lock.yaml') }}-

      - run: pnpm --filter web build
      - run: pnpm exec playwright test
      
      - uses: actions/upload-artifact@v4
        if: failure()
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 7

  # --- Деплой (только main) ---
  post-deploy-check:
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    needs: [lint, typecheck, unit-tests, e2e]
    runs-on: ubuntu-latest
    steps:
      - name: Wait for Coolify deploy
        run: |
          for i in $(seq 1 30); do
            STATUS=$(curl -s -o /dev/null -w "%{http_code}" https://infolog24.ru/api/health)
            if [ "$STATUS" = "200" ]; then
              echo "Health check passed"
              exit 0
            fi
            echo "Attempt $i: status $STATUS, waiting 10s..."
            sleep 10
          done
          echo "Health check failed after 5 minutes"
          exit 1
```

## 10. Безопасность: укрепление

### Проблемы

**10.1. VPS hardening — описан минимально:**

- Нет автоматических обновлений безопасности
- Нет аудита входов (кто, когда, откуда)
- Fail2ban установлен, но не настроен (дефолтная конфигурация слабая)
- Нет защиты от brute-force Docker API
- Порт 8000 Coolify открыт во время настройки — легко забыть закрыть

**10.2. Secrets management:**

- Секреты хранятся в Coolify (web UI) — OK для старта, но:
  - Нет версионирования секретов
  - Нет аудита доступа к секретам
  - При компрометации Coolify — все секреты утекают

**10.3. Cron-задачи с секретом в plaintext:**

```bash
curl -H "X-Cron-Secret: $CRON_SECRET" http://localhost:3000/api/cron/...
```
`$CRON_SECRET` в crontab виден через `crontab -l` любому пользователю с доступом к VPS.

### Рекомендации

**VPS hardening — полный чеклист:**

```bash
# Автоматические обновления безопасности
apt install unattended-upgrades -y
dpkg-reconfigure -plow unattended-upgrades

# Fail2ban — усиленная конфигурация
cat > /etc/fail2ban/jail.local << 'EOF'
[DEFAULT]
bantime = 3600
findtime = 600
maxretry = 3

[sshd]
enabled = true
port = ssh
filter = sshd
maxretry = 3
bantime = 86400

[docker-auth]
enabled = true
port = 8000
filter = docker-auth
maxretry = 5
EOF

# Аудит SSH-входов
apt install auditd -y
auditctl -w /etc/ssh/sshd_config -p wa -k sshd_config

# Ограничить SSH по IP (если есть статический IP)
# ufw allow from YOUR_IP to any port 22

# Отключить IPv6 если не используется
echo "net.ipv6.conf.all.disable_ipv6 = 1" >> /etc/sysctl.conf
sysctl -p

# Kernel hardening
cat >> /etc/sysctl.conf << 'EOF'
net.ipv4.tcp_syncookies = 1
net.ipv4.conf.all.rp_filter = 1
net.ipv4.conf.all.accept_redirects = 0
net.ipv4.conf.all.send_redirects = 0
kernel.randomize_va_space = 2
EOF
sysctl -p
```

**Cron-секреты — безопасный подход:**

```bash
# Хранить секрет в файле с ограниченным доступом
echo "CRON_SECRET=ваш_секрет" > /opt/scripts/.env
chmod 600 /opt/scripts/.env
chown deploy:deploy /opt/scripts/.env

# В crontab:
*/5 * * * * /opt/scripts/run-cron.sh send-notifications
```

```bash
# /opt/scripts/run-cron.sh
#!/bin/bash
source /opt/scripts/.env
curl -s -H "X-Cron-Secret: ${CRON_SECRET}" "http://localhost:3000/api/cron/$1"
```

**Docker security:**

```bash

# (Coolify создаёт свою bridge-сеть — это ОК)
```

## 11. Оптимизация затрат

### Текущая оценка

| Статья | В плане | Комментарий |
|--------|---------|-------------|
| Production VPS 4c/8GB | 3 200₽ | Достаточно для старта с Prometheus+Grafana |
| Staging VPS 2c/4GB | 1 600₽ | Можно оптимизировать |
| S3 (документы + бэкапы) | 80₽ | Корректно |

### Рекомендации по оптимизации

**11.1. Staging — не нужен отдельный VPS на старте:**

При трафике до 1000 визитов/день staging можно запустить на том же VPS что и production, но с ресурсными лимитами:
- Production: 3 cores, 5 GB RAM
- Staging: 1 core, 2 GB RAM
- Мониторинг: 0.5 core, 1 GB RAM

Экономия: **1 600₽/мес**.

Переходить на отдельный VPS когда production-нагрузка вырастет (CPU > 60% sustained).

**11.2. Managed PostgreSQL vs Docker:**

На этапе роста (> 1000 заявок/мес) рассмотреть Selectel Managed PostgreSQL:
- Автобэкапы, point-in-time recovery
- Автоматический failover
- Обновления без простоя
- Цена: ~2 000₽/мес (но экономит время на администрирование)

**11.3. Cloudflare кэширование:**

- Настроить кэширование статических ресурсов (CSS, JS, шрифты, изображения) на Cloudflare Edge
- Cache-Control: `public, max-age=31536000, immutable` для hashed-assets (.next/static/)
- Это снизит нагрузку на VPS и ускорит загрузку

**11.4. Мониторинг затрат:**

Ежемесячно фиксировать:
- Стоимость VPS
- Стоимость S3 (объём, трафик)
- Стоимость SMS.ru (по числу SMS)
- Стоимость OpenAI API (по числу токенов)
- Стоимость Resend (по числу email)
