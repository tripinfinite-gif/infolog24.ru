## 1. Tailwind v4 CSS Variables: план vs. реальность

### Что хорошо
- В `globals.css` уже реализована корректная четырёхшаговая архитектура из скилла `tailwind-theme-builder`: CSS-переменные в `:root` / `.dark` -> `@theme inline` -> утилитарные классы.
- Используется `oklch()` вместо `hsl()` — более современное цветовое пространство с лучшей perceptual uniformity.
- `components.json` имеет `"config": ""` — корректно для v4.

### Критические расхождения

**1.1. `:root` и `.dark` внутри `@layer base` (Gotcha #1)**

В текущем `globals.css` блоки `:root` и `.dark` находятся **внутри** `@layer base` (строки 67-178). Скилл `tailwind-theme-builder` явно указывает, что это **самая частая ошибка**: Tailwind v4 может удалить CSS вне `@theme`/`@layer`, но `:root` обязан быть на корневом уровне для сохранения.

```css
/* СЕЙЧАС (неправильно): */
@layer base {
  :root { --background: oklch(1 0 0); ... }
  .dark { ... }
}

/* ДОЛЖНО БЫТЬ: */
:root { --background: oklch(1 0 0); ... }
.dark { ... }

@layer base {
  * { @apply border-border; }
  body { @apply bg-background text-foreground; }
}
```

**Приоритет: КРИТИЧЕСКИЙ.** Может привести к тому, что переменные будут удалены при сборке.

**1.2. План использует HEX-значения и `tailwind.config.js`, реализация — oklch**

В плане (раздел "Tailwind-конфиг") указана конфигурация через `tailwind.config.js` с HEX-значениями:
```js
colors: { primary: { 50: '#EFF6FF', ... } }
```
Это подход Tailwind v3. В v4 файл `tailwind.config.ts` не используется (Gotcha #4). Реализация уже правильно использует CSS-переменные, но **план нужно обновить**, чтобы не вводить в заблуждение.

**1.3. Нет полной палитры оттенков (50-900) в CSS-переменных**

План определяет `primary-50` через `primary-900`, `accent-50` через `accent-700`, `neutral-50` через `neutral-900`. Но в `globals.css` определены только семантические токены (`--primary`, `--accent`), без оттенков. Компоненты в плане ссылаются на `bg-primary-50`, `text-primary-600`, `bg-accent-600` — эти классы **не будут существовать** в текущей конфигурации.

**Рекомендация:** Добавить полную палитру оттенков как дополнительные CSS-переменные и маппинги в `@theme inline`:

```css
:root {
  --primary-50: oklch(0.97 0.01 260);
  --primary-100: oklch(0.93 0.03 260);
  --primary-500: oklch(0.55 0.18 260);
  --primary-600: oklch(0.48 0.17 260);
  --primary-800: oklch(0.35 0.15 260);
  /* ... */
}

@theme inline {
  --color-primary-50: var(--primary-50);
  --color-primary-100: var(--primary-100);
  /* ... */
}
```

Либо использовать только семантические токены (`bg-primary`, `bg-accent`) и обновить план, убрав числовые оттенки.

## 2. Порядок установки shadcn/ui компонентов

### Что хорошо
- Установлены 24 компонента — покрытие достаточное для маркетингового сайта и базового ЛК.
- Foundation-компоненты (button, input, label, card) на месте.

### Что пропущено

**2.1. Компонент `form` не установлен**

В плане описана форма заявки как ключевой компонент. Скилл `shadcn-ui` указывает, что `form` требует внешние зависимости: `react-hook-form`, `zod`, `@hookform/resolvers`. Компонент `form` отсутствует в `ui/`.

**2.2. Компонент `popover` не установлен**

Нужен для dropdown-фильтров в таблицах, комбобоксов, date-picker'ов.

**2.3. Компонент `command` не установлен**

Полезен для поиска по услугам / FAQ — быстрый поиск (Cmd+K pattern).

**2.4. Рекомендуемый порядок доустановки**

```bash
# 1. Формы (критично для конверсии)
pnpm add react-hook-form zod @hookform/resolvers
pnpm dlx shadcn@latest add form

# 2. Overlay (для фильтров, date-picker)
pnpm dlx shadcn@latest add popover

# 3. Поиск (опционально, фаза 2+)
pnpm dlx shadcn@latest add command
```

## 3. Стратегия тёмной темы

### Что хорошо
- Решение "только светлая для маркетинга, тёмная для ЛК" — прагматичное и верное.
- В `globals.css` уже определены переменные `.dark` — инфраструктура готова.

### Проблемы

**3.1. Нет ThemeProvider'а**

Скилл `tailwind-theme-builder` требует `ThemeProvider` для переключения `.dark`-класса. В проекте на Next.js 15 с App Router нужно использовать `next-themes` (не самописный provider из скилла, который заточен под Vite).

**Рекомендация для плана:**
```
Фаза 4 (ЛК):
- [ ] Установить next-themes
- [ ] Добавить ThemeProvider в root layout
- [ ] Добавить ModeToggle в хедер ЛК
- [ ] Протестировать: light / dark / system / persist after refresh
```

**3.2. Тёмная тема определена, но нигде не тестируется**

Переменные `.dark` уже в CSS, но в плане Фаза 3 не включает тест тёмной темы. Если кто-то случайно добавит `class="dark"` на `<html>`, результат непредсказуем. Стоит либо убрать `.dark` из CSS до фазы 4, либо добавить тестирование в фазу 3.

## 4. Производительность анимаций

### Что хорошо
- Длительности разумные (100-400ms).
- Поддержка `prefers-reduced-motion` упомянута.
- Принцип "функциональные, не wow" — правильный для ЦА.

### Что нужно добавить в план

**4.1. Framer Motion: стратегия lazy-loading**

Framer Motion добавляет ~30-40 KB к бандлу. Для маркетингового сайта, где Core Web Vitals критичны:

```tsx
// Вместо:
import { motion } from 'framer-motion'

// Использовать:
import { m, LazyMotion, domAnimation } from 'framer-motion'
```

`domAnimation` — минимальный пакет (~15KB вместо ~40KB). Добавить это в план.

**4.2. Intersection Observer вместо scroll-based анимаций**

Для "появления секций при скролле" лучше использовать `IntersectionObserver` (или `useInView` из framer-motion) вместо scroll listeners. Это уже подразумевается, но стоит прописать явно.

**4.3. CSS-анимации для простых случаев**

Для hover-эффектов карточек (`shadow transition`), кнопок (`transition-colors`), аккордеонов — достаточно CSS-анимаций. Framer Motion нужен только для: появление секций, счётчики, модалки, sheet. Это снизит зависимость от JS.

**4.4. prefers-reduced-motion: конкретная реализация**

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}
```

Добавить в `globals.css`. В плане упомянуто, но нет конкретики.

## 5. Адаптивные брейкпоинты

### Что хорошо
- Mobile-first подход.
- Touch targets >= 44px.
- Принцип "78% с мобайла" — хорошо обоснован.

### Что нужно улучшить

**5.1. Отсутствует брейкпоинт для container queries**

Tailwind v4 поддерживает `@container` нативно. Для карточек услуг, которые могут быть в разных контекстах (главная, страница услуг, сайдбар ЛК), container queries дадут более точную адаптивность:

```css
@theme inline {
  --container-card: 320px;
}
```

**5.2. Нет стратегии для промежуточных размеров (tablet landscape)**

Между `md` (768) и `lg` (1024) — зона iPad landscape. Сетка прыгает с 2 колонок на 3. Для bento-grid и калькулятора стоит продумать промежуточное состояние.

**5.3. max-w-7xl (1280px) может быть тесно**

Для mega-menu ("Услуги" в 3 колонки + описание + CTA) ширина 1280px может быть недостаточной. Рассмотреть `max-w-screen-xl` (1280px) для контента и `max-w-screen-2xl` (1536px) для навигации.

## 6. Доступность (a11y)

### Что хорошо
- WCAG 2.1 AA контрасты проверены.
- Бейджи дублируют цвет текстом/иконкой.
- Модалки закрываются по Escape.
- Radix-компоненты (через shadcn/ui) имеют встроенную a11y.

### Что пропущено

**6.1. Контраст `#FFFFFF` на `#EA580C` = 3.5:1 — пограничный**

Это проходит AA только для "large text" (>= 18.66px bold или >= 24px regular). В плане указано "CTA >= 16px bold" — это **не проходит** AA large text (нужно >= 18.66px bold, т.е. >= 19px). Варианты:

- Увеличить кнопки до `text-lg` (18px) с `font-bold` (фактически 18px, ещё пограничный);
- Затемнить оранжевый до `#C2410C` (4.6:1, проходит AA для обычного текста);
- Использовать тёмный текст на оранжевом фоне.

**Рекомендация:** Использовать `#C2410C` (accent-700) для CTA-кнопок вместо `#EA580C`. Разница визуально минимальна, но контраст существенно лучше.

**6.2. Нет aria-live для динамического контента**

Калькулятор стоимости и animated counter при скролле обновляют контент динамически. Для screen readers нужно:

```tsx
<div aria-live="polite" aria-atomic="true">
  Итого: {price} руб.
</div>
```

**6.3. Нет skip-navigation ссылки**

Стандартный паттерн доступности:
```tsx
<a href="#main-content" className="sr-only focus:not-sr-only">
  Перейти к основному содержанию
</a>
```

**6.4. Формы: нет связки label-input через htmlFor/id**

В плане описаны Input и Label, но не прописана обязательная связь через `htmlFor` / `id`. Это критично для screen readers.

**6.5. Фокус-индикаторы**

В плане `focus:ring-2 focus:ring-primary-500` — хорошо. Но нужно убедиться, что фокус видим на **всех** интерактивных элементах, включая карточки-ссылки и кнопки навигации.

## 7. Композиция компонентов

### Что хорошо
- Чёткая файловая структура: `ui/`, `layout/`, `sections/`, `shared/`.
- Разделение на shadcn-примитивы и бизнес-компоненты.

### Что нужно добавить

**7.1. Нет паттерна compound components**

Для сложных компонентов (калькулятор, форма заявки, stepper "Как это работает") рекомендуется compound component pattern:

```tsx
<Calculator>
  <Calculator.Field name="type" />
  <Calculator.Field name="term" />
  <Calculator.Counter name="vehicles" />
  <Calculator.Result />
  <Calculator.CTA />
</Calculator>
```

Это улучшает переиспользуемость и тестируемость.

**7.2. Нет абстракции для секций**

Все секции (Hero, Benefits, HowItWorks и т.д.) следуют одному паттерну: контейнер + заголовок + контент. Стоит добавить базовый `Section` компонент:

```tsx
<Section id="how-it-works" className="bg-neutral-50" spacing="lg">
  <Section.Header
    badge="Простой процесс"
    title="Как это работает"
    description="4 шага до получения пропуска"
  />
  <Section.Content>
    {/* ... */}
  </Section.Content>
</Section>
```

**7.3. Нет стратегии для Server Components vs Client Components**

Next.js 15 App Router по умолчанию использует Server Components. В плане не определено, какие компоненты серверные, а какие клиентские. Рекомендуемое разделение:

| Server Components | Client Components |
|---|---|
| Header (статический), Footer, секции Hero/Benefits | MobileNav (бургер), Calculator (интерактивный) |
| Cards (статические), Badge, Table | FAQ Accordion, Modal, Toast |
| Все страницы (`page.tsx`) | Формы (react-hook-form) |

## 8. Дополнительные замечания

**8.1. `baseColor` в `components.json` — "zinc", но палитра "slate"**

В `components.json` указан `"baseColor": "zinc"`, но в плане и CSS используются slate-тона (`#0F172A`, `#475569`, `#F1F5F9` — это шкала Tailwind slate). Нужно привести в соответствие: либо сменить `baseColor` на `"slate"`, либо убедиться, что это не влияет на генерацию компонентов (в v4 с CSS-переменными `baseColor` используется только при `shadcn init`).

**8.2. Нет tw-animate-css**

Скилл `tailwind-theme-builder` указывает (Gotcha #17): `tailwindcss-animate` устарел в Tailwind v4, нужно `tw-animate-css`. В `globals.css` нет импорта `tw-animate-css`. Аккордеонные анимации определены вручную (что нормально), но при добавлении shadcn-компонентов с анимациями (Dialog, Sheet, Toast) могут возникнуть проблемы.

**Рекомендация:**
```bash
pnpm add -D tw-animate-css
```
```css
/* globals.css, после @import "tailwindcss" */
@import "tw-animate-css";
```

**8.3. Шрифт Manrope не описан в CSS-переменных**

В `globals.css` определены `--font-sans` (Inter) и `--font-heading` (Manrope), но `@theme inline` не маппит `--font-heading` в утилитарный класс. Нужно проверить, что `font-heading` работает в Tailwind-классах.

**8.4. Нет стратегии для изображений**

План не упоминает `next/image`, оптимизацию изображений, placeholder blur, форматы (WebP/AVIF). Для маркетингового сайта с логотипами клиентов, иконками услуг, возможными фото — это важно для производительности.

## Сводка рекомендаций по приоритетам

### Критические (блокируют корректную работу)
1. Вынести `:root` и `.dark` из `@layer base` на корневой уровень CSS
2. Добавить палитру оттенков (primary-50..900, accent-50..700) в CSS-переменные и `@theme inline`, либо обновить план на семантические токены
3. Установить `tw-animate-css` для корректной работы shadcn-анимаций

### Важные (влияют на качество и конверсию)
4. Обновить план: убрать `tailwind.config.js`, описать v4 CSS-переменные
5. Улучшить контраст CTA-кнопок (accent-700 вместо accent-600)
6. Установить `form` компонент + зависимости (react-hook-form, zod)
7. Добавить skip-navigation и aria-live для динамического контента
8. Добавить LazyMotion стратегию для Framer Motion

### Желательные (улучшают архитектуру)
9. Добавить compound component паттерн для Calculator и Form
10. Добавить базовый Section компонент
11. Определить Server/Client Component boundary
12. Добавить container queries для карточек
13. Добавить стратегию оптимизации изображений (next/image)
14. Привести `baseColor` в `components.json` в соответствие с палитрой (slate)
