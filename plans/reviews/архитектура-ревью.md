5. **Workspace-структура `tooling/`** -- eslint-конфиг вынесен в отдельный workspace. Это правильный паттерн для монорепо.

## Конкретные проблемы и рекомендации

### 1. Drizzle-схема заперта внутри web-приложения

**Проблема:** `drizzle.config.ts` в web ссылается на `./src/lib/db/schema.ts`. Когда появится `apps/bot/` (или любой другой consumer), ему придётся импортировать из `apps/web` -- это анти-паттерн монорепо. Кроме того, root-скрипты `db:generate` и `db:migrate` жёстко привязаны к `@infolog24/web` через `--filter`.

**Рекомендация:** Вынести в `packages/db/` как в плане. Минимальная структура:

```
packages/db/
  src/
    schema.ts        # все таблицы
    index.ts         # export { db, schema }
    queries/         # типизированные запросы (DAL)
  drizzle.config.ts
  package.json       # deps: drizzle-orm, postgres
```

Root-скрипты станут:
```json
{
  "db:generate": "pnpm --filter @infolog24/db drizzle-kit generate",
  "db:migrate": "pnpm --filter @infolog24/db drizzle-kit migrate",
  "db:studio": "pnpm --filter @infolog24/db drizzle-kit studio"
}
```

**Приоритет:** Высокий. Делать до появления второго consumer-а (бот).

### 2. Отсутствует shared tsconfig

**Проблема:** Каждый пакет будет дублировать настройки TypeScript. При 5+ пакетах это станет проблемой обслуживания.

**Рекомендация:** Создать `tooling/typescript/` (или `packages/config/typescript/`):

```
tooling/typescript/
  base.json          # strict, target, module
  nextjs.json        # extends base + JSX, Next.js plugin
  library.json       # extends base + declaration, composite
  package.json
```

Пример `base.json`:
```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "compilerOptions": {
    "strict": true,
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,
    "resolveJsonModule": true,
    "isolatedModules": true,
    "noUncheckedIndexedAccess": true
  }
}
```

Потребители:
```json
// apps/web/tsconfig.json
{ "extends": "@infolog24/typescript/nextjs.json" }

// packages/db/tsconfig.json
{ "extends": "@infolog24/typescript/library.json" }
```

**Приоритет:** Средний. Можно сделать при создании второго пакета.

### 3. turbo.json -- нет `inputs`, нет remote cache, нет `--filter` в CI

**Проблема:**
- Без `inputs` Turborepo хэширует все файлы в пакете. Изменение README перезапустит build.
- Нет конфигурации remote cache (Turborepo Remote Cache или self-hosted).
- В плане CI/CD описан только "Git push -> Coolify" без `--filter` на affected.

**Рекомендация:** Улучшить `turbo.json`:

```json
{
  "$schema": "https://turbo.build/schema.json",
  "globalDependencies": [".env"],
  "globalPassThroughEnv": ["NODE_ENV", "DATABASE_URL"],
  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "inputs": ["src/**", "package.json", "tsconfig.json", "next.config.*"],
      "outputs": [".next/**", "!.next/cache/**", "dist/**"]
    },
    "dev": {
      "cache": false,
      "persistent": true
    },
    "lint": {
      "dependsOn": ["^build"],
      "inputs": ["src/**", "*.config.*"]
    },
    "type-check": {
      "dependsOn": ["^build"],
      "inputs": ["src/**", "tsconfig.json"]
    },
    "test": {
      "dependsOn": ["^build"],
      "inputs": ["src/**", "tests/**"]
    }
  }
}
```

Для CI (GitHub Actions):
```yaml
- run: pnpm turbo build --filter=...[origin/main]
```

Remote cache: если Coolify не поддерживает Turborepo remote cache нативно, можно поднять `ducktape` или `turborepo-remote-cache` (self-hosted, ~100 строк Docker) на том же VPS.

**Приоритет:** Средний. Критично, когда пакетов станет > 3.

### 4. Route groups не используются

**Проблема:** План описывает `(marketing)`, `(auth)`, `(dashboard)`, `(admin)` -- это Next.js route groups, позволяющие разделять layout-ы (маркетинг -- header+footer, dashboard -- sidebar, admin -- отдельный layout). Текущая структура плоская -- все маршруты на одном уровне, что означает один общий layout.

**Рекомендация:** Это не критичная проблема прямо сейчас (пока нет ЛК и админки), но стоит реструктурировать до начала работы над dashboard. При плоской структуре придётся добавлять conditional layout logic в `layout.tsx`, что хуже, чем route groups.

**Приоритет:** Низкий сейчас, высокий перед dashboard.

### 5. packages/shared пуст -- нужна стратегия

**Проблема:** `packages/shared` экспортирует пустой объект. При этом `apps/web` уже его импортирует (`@infolog24/shared": "workspace:*"`). Непонятно, что туда класть.

**Рекомендация:** Определить чёткие границы:

| Пакет | Содержимое | Кто использует |
|-------|-----------|----------------|
| `@infolog24/shared` | Zod-схемы валидации, константы (зоны, статусы), утилиты форматирования | web, bot, любые apps |
| `@infolog24/db` | Drizzle-схема, клиент, миграции, queries (DAL) | web, bot |
| `@infolog24/auth` | Better Auth config, клиент, permissions | web |
| `@infolog24/notifications` | Email/SMS/Telegram отправка | web, bot |

Конкретный первый шаг -- перенести Zod-схемы и enum-константы (zone, permit_type, order_status) в shared:

```typescript
// packages/shared/src/constants.ts
export const ZONES = ["mkad", "ttk", "sk"] as const;
export type Zone = (typeof ZONES)[number];

export const ORDER_STATUSES = [
  "draft", "documents_pending", "documents_review",
  "payment_pending", "in_progress", "approved",
  "rejected", "cancelled"
] as const;
export type OrderStatus = (typeof ORDER_STATUSES)[number];

// packages/shared/src/validators.ts
import { z } from "zod";
import { ZONES, ORDER_STATUSES } from "./constants";

export const createOrderSchema = z.object({
  vehicleId: z.string().uuid(),
  zone: z.enum(ZONES),
  permitType: z.enum(["annual", "temporary", "night", "one_time"]),
  desiredStart: z.coerce.date(),
  desiredEnd: z.coerce.date(),
  routeFrom: z.string().min(1),
  routeTo: z.string().min(1),
});
```

**Приоритет:** Высокий. Без этого shared-пакет -- мёртвый код.

### 6. Бот как отдельное приложение vs встроенный

**Проблема:** План описывает `apps/bot/` как отдельное приложение (grammY, свой процесс). В реальности бот реализован как webhook-эндпоинт внутри Next.js (`apps/web/src/lib/telegram/` + `api/telegram/webhook/`).

**Оценка:** Webhook-подход через Next.js API Route -- это валидно для старта. Отдельный `apps/bot/` нужен, только если:
- Бот обрабатывает тяжёлые операции (long-running conversations)
- Нужен long-polling вместо webhook
- Бот должен деплоиться независимо от сайта

Для текущего объёма (уведомления + приём документов) webhook внутри Next.js -- правильное решение.

**Рекомендация:** Обновить план, указав что `apps/bot/` -- запас на будущее, а текущая реализация через webhook в web -- осознанное решение. Вернуться к вопросу, когда бот станет сложнее.

**Приоритет:** Низкий.

### 7. Нет .env.example в репозитории

**Проблема:** План описывает `.env.example` с 15+ переменными, но в корне репозитория его нет. Новый разработчик не поймёт, какие env-переменные нужны.

**Рекомендация:** Создать `.env.example` по плану. Также добавить валидацию через `@t3-oss/env-nextjs` (уже в зависимостях web-приложения -- убедиться, что используется).

**Приоритет:** Высокий. Занимает 5 минут, экономит часы.

### 8. Нет стратегии тестирования

**Проблема:** План не упоминает тестирование вообще. В `turbo.json` нет задачи `test`. В `package.json` web-приложения нет тестового фреймворка.

**Рекомендация:** Добавить Vitest (нативная интеграция с Vite/Next.js, быстрее Jest):

```json
// apps/web/package.json -- devDependencies
"vitest": "^3",
"@testing-library/react": "^16",
"@testing-library/user-event": "^14"
```

```json
// turbo.json -- добавить задачу
"test": {
  "dependsOn": ["^build"],
  "inputs": ["src/**", "tests/**"]
}
```

На старте -- хотя бы unit-тесты для DAL-запросов и Zod-валидаторов. Не нужно 100% coverage, но критические бизнес-пути (создание заявки, расчёт цены) должны быть покрыты.

**Приоритет:** Средний. Нужно до production.

### 9. Redis в плане, но нет в зависимостях shared-уровня

**Проблема:** Архитектурная схема показывает Redis (кэш, очереди, сессии бота). В `apps/web` есть `bullmq` и `ioredis` в зависимостях. Но конфигурация Redis (connection string) не вынесена в shared -- каждый consumer будет дублировать подключение.

**Рекомендация:** При выделении бота в отдельное приложение -- вынести Redis-клиент в shared-пакет или `packages/queue/`.

**Приоритет:** Низкий (пока один consumer).

## Оптимизация затрат

1. **VPS 4 core / 8 GB за 3 200 руб -- достаточно на старте.** Но Next.js standalone build + PostgreSQL + Redis + Coolify на одной машине -- будет тесно при нагрузке. Рекомендую managed PostgreSQL от Selectel (от 1 100 руб/мес) вместо Docker-контейнера на том же VPS -- снимет нагрузку и даст автобэкапы.

2. **Resend для email -- риск.** Resend серверы в США. Для транзакционных писем российским клиентам лучше Unisender (РФ) или просто SMTP через Яндекс 360 (бесплатно до 1000 писем/день для бизнеса).

3. **GPT-4o mini для чатбота -- правильный выбор.** При 300 диалогах/день ~$13/мес -- это копейки. Не переходить на более дорогую модель без доказанной потребности.
