## 1. API Security

### 1.1 КРИТИЧНО: Rate Limiting — недостаточная детализация

**Текущее состояние (план):** "100 req/min на IP для API, 10 req/min для auth endpoints"

**Проблема:** Отсутствует rate limiting для критически важных endpoint'ов:
- `POST /api/payments/create` — без ограничения можно создать тысячи платёжных сессий в YooKassa
- `POST /api/upload/presign` — без ограничения можно заполнить S3-хранилище
- `POST /api/applications` — без ограничения можно спамить заявками
- `GET /api/permits/check` — публичный endpoint, уязвим для парсинга данных (перебор номеров)

**Рекомендация:**
```
Endpoint                        Лимит               Ключ
POST /api/auth/*                5 req/min            IP
POST /api/payments/create       3 req/min            user_id
POST /api/upload/presign        10 req/min           user_id
POST /api/applications          5 req/min            user_id
GET  /api/permits/check         10 req/min           IP
POST /api/webhooks/*            без лимита           whitelist IP
Все остальные GET               60 req/min           IP + user_id
Все остальные POST/PATCH        30 req/min           IP + user_id
```

Использовать `rate-limiter-flexible` с Redis-хранилищем (или in-memory для начала, но с пометкой миграции на Redis).

### 1.2 КРИТИЧНО: CSRF-защита не упоминается

**Проблема:** В плане бэкенда нет ни одного упоминания CSRF-токенов. Next.js Route Handlers не защищены от CSRF по умолчанию. Мутирующие операции (POST/PATCH/DELETE) уязвимы, если сессия хранится в cookie.

**Рекомендация:**
- Для Auth.js v5 с JWT-стратегией в cookies: добавить проверку `Origin`/`Referer` заголовков в middleware
- Использовать SameSite=Strict для session cookie
- Для форм с Server Actions — Next.js обеспечивает CSRF-защиту автоматически, но для Route Handlers нужна явная проверка
- Добавить отдельную задачу в Фазу 8: реализация CSRF middleware

### 1.3 СУЩЕСТВЕННО: Отсутствует валидация query parameters

**Проблема:** В плане Zod-валидация описана только для request body. Query parameters (`GET /api/applications?status=...&page=...`) не упомянуты.

**Рекомендация:** Добавить Zod-схемы для всех query params:
```typescript
const listApplicationsQuery = z.object({
  status: z.enum([...]).optional(),
  zone: z.enum([...]).optional(),
  cursor: z.string().uuid().optional(),
  limit: z.coerce.number().int().min(1).max(100).default(20),
});
```

### 1.4 СУЩЕСТВЕННО: Content-Type enforcement

**Проблема:** Не указана проверка Content-Type входящих запросов. Атакующий может отправить данные в неожиданном формате.

**Рекомендация:** В middleware проверять `Content-Type: application/json` для всех мутирующих API-маршрутов (кроме вебхуков Битрикс24, который шлёт `application/x-www-form-urlencoded`).

### 1.5 РЕКОМЕНДАЦИЯ: API Versioning

**Проблема:** Не предусмотрено версионирование API. При наличии публичного endpoint'а (`/api/permits/check`) и потенциального мобильного приложения в будущем — критично.

**Рекомендация:** Использовать префикс `/api/v1/` с первого дня. Стоимость — минимальна, а гибкость — существенна.

## 2. Безопасность базы данных

### 2.1 КРИТИЧНО: Отсутствуют индексы в схеме

**Проблема:** В описании таблиц (Фаза 1) нет ни одного упоминания индексов. При 600-800 заявках/мес таблицы будут расти, и без индексов запросы деградируют.

**Рекомендация — обязательные индексы:**

```sql
-- applications
CREATE INDEX idx_applications_user_id ON applications(user_id);
CREATE INDEX idx_applications_status ON applications(status);
CREATE INDEX idx_applications_assigned_manager ON applications(assigned_manager_id);
CREATE INDEX idx_applications_created_at ON applications(created_at DESC);
CREATE INDEX idx_applications_user_status ON applications(user_id, status);  -- composite для ЛК клиента

-- permits
CREATE INDEX idx_permits_user_id ON permits(user_id);
CREATE INDEX idx_permits_application_id ON permits(application_id);
CREATE INDEX idx_permits_status_valid_until ON permits(status, valid_until);  -- для cron проверки истечения
CREATE INDEX idx_permits_plate ON permits(vehicle_plate);  -- для публичного поиска

-- documents
CREATE INDEX idx_documents_application_id ON documents(application_id);
CREATE INDEX idx_documents_user_id ON documents(user_id);

-- payments
CREATE INDEX idx_payments_application_id ON payments(application_id);
CREATE INDEX idx_payments_user_id ON payments(user_id);
CREATE INDEX idx_payments_yookassa_id ON payments(yookassa_payment_id);  -- для идемпотентности вебхуков
CREATE UNIQUE INDEX idx_payments_yookassa_id_unique ON payments(yookassa_payment_id) WHERE yookassa_payment_id IS NOT NULL;

-- notifications
CREATE INDEX idx_notifications_user_id_read ON notifications(user_id, is_read);  -- для бейджа
CREATE INDEX idx_notifications_status_scheduled ON notifications(status, scheduled_at) WHERE status = 'pending';  -- для cron-очереди
CREATE INDEX idx_notifications_created_at ON notifications(created_at DESC);

-- activity_log
CREATE INDEX idx_activity_log_entity ON activity_log(entity_type, entity_id);
CREATE INDEX idx_activity_log_user_id ON activity_log(user_id);
CREATE INDEX idx_activity_log_created_at ON activity_log(created_at DESC);
```

### 2.2 КРИТИЧНО: Отсутствуют CHECK constraints

**Проблема:** Бизнес-правила валидируются только на уровне приложения (Zod). Если кто-то обходит API (прямой доступ к БД, баг в коде), данные могут быть невалидными.

**Рекомендация:**
```sql
-- applications
ALTER TABLE applications ADD CONSTRAINT chk_zone CHECK (zone IN ('mkad', 'ttk', 'sk'));
ALTER TABLE applications ADD CONSTRAINT chk_permit_type CHECK (permit_type IN ('daily', 'night', 'temporary', 'annual'));
ALTER TABLE applications ADD CONSTRAINT chk_status CHECK (status IN ('draft', 'documents_pending', 'documents_review', 'payment_pending', 'in_progress', 'approved', 'rejected', 'cancelled'));
ALTER TABLE applications ADD CONSTRAINT chk_vehicle_count CHECK (vehicle_count >= 1 AND vehicle_count <= 100);

-- payments
ALTER TABLE payments ADD CONSTRAINT chk_amount_positive CHECK (amount > 0);
ALTER TABLE payments ADD CONSTRAINT chk_payment_status CHECK (status IN ('pending', 'succeeded', 'cancelled', 'refunded'));

-- permits
ALTER TABLE permits ADD CONSTRAINT chk_valid_dates CHECK (valid_from < valid_until);
ALTER TABLE permits ADD CONSTRAINT chk_permit_status CHECK (status IN ('active', 'expired', 'revoked'));

-- users
ALTER TABLE users ADD CONSTRAINT chk_role CHECK (role IN ('client', 'manager', 'admin'));
```

### 2.3 СУЩЕСТВЕННО: Нет ON DELETE каскадов / политик

**Проблема:** Не определено поведение FK при удалении связанных записей. Например, что происходит с documents при удалении application?

**Рекомендация:**
```sql
-- documents -> applications: CASCADE (удаление заявки удаляет документы)
-- payments -> applications: RESTRICT (нельзя удалить заявку с платежами)
-- permits -> applications: RESTRICT (нельзя удалить заявку с пропуском)
-- notifications -> users: CASCADE
-- activity_log -> users: SET NULL (лог сохраняется, пользователь анонимизируется)
-- applications -> users: RESTRICT (нельзя удалить пользователя с заявками)
```

Плюс — рассмотреть soft delete вместо hard delete для `applications` и `users` (добавить `deleted_at TIMESTAMPTZ`).

### 2.4 СУЩЕСТВЕННО: Отсутствует `updated_at` trigger

**Проблема:** Поле `updated_at` есть в схемах, но нет механизма автоматического обновления.

**Рекомендация:** Создать PostgreSQL-триггер:
```sql
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Применить ко всем таблицам с updated_at
CREATE TRIGGER trg_applications_updated_at
  BEFORE UPDATE ON applications
  FOR EACH ROW EXECUTE FUNCTION update_updated_at();
```

### 2.5 РЕКОМЕНДАЦИЯ: Хранение денежных сумм

**Проблема:** В таблице `payments` поле `amount` не имеет указанного типа. Если использовать FLOAT — будут ошибки округления.

**Рекомендация:** Использовать `DECIMAL(12, 2)` или хранить в копейках как `BIGINT`. Для Drizzle ORM:
```typescript
amount: decimal('amount', { precision: 12, scale: 2 }).notNull(),
```

### 2.6 РЕКОМЕНДАЦИЯ: Оптимистичная блокировка

**Проблема:** Нет механизма оптимистичной блокировки. Два менеджера могут одновременно изменить статус заявки, перезаписав изменения друг друга.

**Рекомендация:** Добавить поле `version INTEGER DEFAULT 1` в таблицу `applications`. При обновлении:
```sql
UPDATE applications SET status = $1, version = version + 1
WHERE id = $2 AND version = $3;
-- Если affected rows = 0 → конфликт, вернуть 409 Conflict
```

## 3. Безопасность платежей (YooKassa)

### 3.1 КРИТИЧНО: Недостаточная верификация вебхуков

**Текущее состояние (план):** "Валидация подписи (IP-фильтрация ЮKassa или проверка через SDK)"

**Проблема:** Формулировка "или" означает, что можно выбрать только одно. Нужно **оба метода одновременно**.

**Рекомендация:**
1. **IP whitelist** (обязательно): YooKassa отправляет вебхуки с фиксированных IP. Проверять в Nginx или middleware:
   ```
   # YooKassa webhook IPs (уточнить в документации на момент реализации)
   185.71.76.0/27
   185.71.77.0/27
   77.75.153.0/25
   77.75.156.11
   77.75.156.35
   ```
2. **Проверка через API** (обязательно): После получения вебхука — запросить статус платежа напрямую через YooKassa API по `payment_id`. Не доверять данным из вебхука.
3. **Подпись** (если поддерживается SDK): Дополнительная проверка HMAC-подписи.

### 3.2 СУЩЕСТВЕННО: Нет защиты от повторной оплаты

**Проблема:** Клиент может создать несколько платежей для одной заявки, если нажмёт кнопку оплаты несколько раз.

**Рекомендация:**
- Добавить проверку: если для `application_id` уже есть платёж со статусом `pending` — не создавать новый, а вернуть существующий `confirmation_url`
- Добавить `idempotency_key` при создании платежа в YooKassa (UUID, привязанный к application_id + user_id)
- Добавить `UNIQUE` constraint: один `succeeded` платёж на заявку

### 3.3 СУЩЕСТВЕННО: Нет таймаута для pending-платежей

**Проблема:** Если клиент создал платёж, но не оплатил — запись с `status = 'pending'` висит вечно.

**Рекомендация:** Добавить cron-задачу: если платёж в статусе `pending` более 1 часа — проверить статус через YooKassa API. Если не оплачен — перевести в `expired`.

### 3.4 СУЩЕСТВЕННО: Сумма платежа формируется на клиенте

**Проблема:** Endpoint `POST /api/payments/create` принимает `amount` из запроса. Клиент может подменить сумму.

**Рекомендация:** Сумма должна вычисляться на сервере на основе `application_id`:
```typescript
// НЕ доверять amount из запроса
const application = await getApplication(applicationId);
const amount = calculatePrice(application.zone, application.permit_type, application.vehicle_count);
```

## 4. Безопасность загрузки файлов

### 4.1 СУЩЕСТВЕННО: Magic bytes проверка — хорошо, но недостаточно

**Текущее состояние:** Plan упоминает magic bytes — это отлично.

**Дополнительные рекомендации:**
1. **Переименование файлов**: Всегда генерировать UUID-имена. Никогда не использовать оригинальное имя файла в S3-ключе (path traversal, спецсимволы).
2. **Сканирование**: На первом этапе — хотя бы проверка magic bytes + размера. Позже — ClamAV.
3. **Content-Disposition**: При скачивании устанавливать `Content-Disposition: attachment` — файл не должен отображаться inline в браузере (защита от XSS через PDF/SVG).
4. **Отдельный домен для файлов**: Presigned URL должны вести на `files.infolog24.ru` или `*.s3.beget.ru`, а не на основной домен. Это изолирует cookie-scope.

### 4.2 РЕКОМЕНДАЦИЯ: Лимит файлов на заявку

**Проблема:** Нет ограничения на количество файлов в заявке. Клиент может загрузить сотни файлов.

**Рекомендация:** Максимум 20 файлов на заявку, суммарный размер не более 50 MB.

## 5. Авторизация и аутентификация

### 5.1 КРИТИЧНО: Auth.js v5 vs Better Auth — противоречие в планах

**Проблема:** В CLAUDE.md и плане архитектуры указан **Better Auth**, а в плане бэкенда — **Auth.js v5 (NextAuth)**. Это два разных решения. Необходимо определиться.

**Рекомендация:** Выбрать один вариант и обновить все планы. Если Better Auth — обновить бэкенд-план. Если Auth.js — обновить архитектуру.

**Сравнение для данного проекта:**
- **Better Auth**: более гибкий, self-hosted, plugin-система, активная разработка
- **Auth.js v5**: более зрелый, но сложнее с кастомными провайдерами (Credentials)
- Для проекта с Credentials Provider (email+пароль) Better Auth может быть удобнее

### 5.2 СУЩЕСТВЕННО: Password policy не определена

**Проблема:** Упомянут bcrypt с cost factor 12, но нет требований к паролю.

**Рекомендация:**
- Минимум 8 символов
- Проверка по словарю наиболее распространённых паролей (top-10000)
- Не ограничивать максимальную длину меньше 72 символов (лимит bcrypt)
- Не требовать "спецсимволы" — это ухудшает UX без повышения безопасности (NIST SP 800-63B)

### 5.3 СУЩЕСТВЕННО: Нет механизма блокировки аккаунта

**Проблема:** Rate limit на auth endpoints (10 req/min) — необходимый, но недостаточный уровень защиты. Нет блокировки после N неудачных попыток входа.

**Рекомендация:**
- После 5 неудачных попыток — временная блокировка на 15 минут
- После 15 неудачных попыток — блокировка на 1 час
- Уведомление пользователю по email при блокировке
- Логирование всех неудачных попыток в `activity_log`

### 5.4 РЕКОМЕНДАЦИЯ: Session invalidation

**Проблема:** При смене пароля или роли — текущие JWT-токены остаются действительными до истечения.

**Рекомендация:**
- Короткий TTL для access token (15 минут)
- Refresh token в httpOnly cookie с TTL 7 дней
- При смене пароля — инвалидация всех refresh tokens (хранить в БД или Redis)
- При смене роли — немедленная инвалидация сессии

## 6. Надёжность уведомлений

### 6.1 КРИТИЧНО: Retry-логика недостаточна

**Текущее состояние (план):** "Retry: если failed и попыток < 3 — вернуть в pending"

**Проблемы:**
1. Нет exponential backoff — при 3 retry за 15 минут (3 цикла cron по 5 мин) внешний сервис может быть всё ещё недоступен
2. Нет разделения по типам ошибок: 429 (rate limit) нужно ждать, 400 (bad request) бессмысленно ретраить
3. 3 попытки — мало для SMS (SMS.ru может быть недоступен часами)

**Рекомендация:**
```typescript
const RETRY_DELAYS = [5, 15, 60, 240, 720]; // минуты: 5мин, 15мин, 1ч, 4ч, 12ч
const MAX_RETRIES = 5;

// В таблице notifications добавить:
// next_retry_at TIMESTAMPTZ  — когда следующая попытка
// error_code    VARCHAR(50)  — категория ошибки

// В cron-задаче:
// WHERE status = 'pending' AND (next_retry_at IS NULL OR next_retry_at <= NOW())

// При ошибке:
// - 4xx (кроме 429) → статус 'failed', не ретраить
// - 429 → retry через Retry-After или RETRY_DELAYS[retry_count]
// - 5xx, timeout → retry по RETRY_DELAYS
```

### 6.2 СУЩЕСТВЕННО: Нет dead letter queue

**Проблема:** Уведомления, которые не удалось отправить после всех попыток, просто получают статус `failed`. Нет механизма ручного разбора.

**Рекомендация:**
1. Добавить статус `dead_letter` после исчерпания всех retry
2. Страница в админке: список dead_letter уведомлений с кнопкой "повторить"
3. Ежедневный алерт менеджеру в Telegram, если есть dead_letter за последние 24ч
4. Добавить поле `failure_reason` с категоризацией: `invalid_recipient`, `service_unavailable`, `rate_limited`, `unknown`

### 6.3 СУЩЕСТВЕННО: Конкурентность cron-задач

**Проблема:** Если cron-задача `send-notifications` выполняется дольше 5 минут — запустится вторая копия, и одно уведомление отправится дважды.

**Рекомендация:**
1. **Advisory lock** в PostgreSQL:
   ```sql
   SELECT pg_try_advisory_lock(hashtext('send-notifications'));
   -- Если false — другой процесс уже работает, выйти
   ```
2. Или **status locking**: при выборке уведомлений сразу менять статус на `sending`:
   ```sql
   UPDATE notifications SET status = 'sending'
   WHERE id IN (SELECT id FROM notifications WHERE status = 'pending' ORDER BY created_at LIMIT 50 FOR UPDATE SKIP LOCKED)
   RETURNING *;
   ```
   `FOR UPDATE SKIP LOCKED` — ключевой паттерн для конкурентной обработки очередей.

### 6.4 РЕКОМЕНДАЦИЯ: Idempotency для уведомлений

**Проблема:** Нет механизма предотвращения дублирования уведомлений. Если state machine вызовет `transitionOrder` дважды (баг, retry), клиент получит два одинаковых сообщения.

**Рекомендация:** Добавить уникальный ключ: `(order_id, notification_type, channel)` с дедупликацией за последние 5 минут:
```sql
-- Перед созданием уведомления проверить:
SELECT COUNT(*) FROM notifications
WHERE order_id = $1 AND type = $2 AND channel = $3
AND created_at > NOW() - INTERVAL '5 minutes';
```

## 7. State Machine (конечный автомат)

### 7.1 СУЩЕСТВЕННО: Несогласованность статусов между планами

**Проблема:** В плане бэкенда статусы заявки:
```
new | processing | submitted | approved | rejected | cancelled
```
В плане автоматизации:
```
draft | documents_pending | documents_review | payment_pending | in_progress | approved | rejected | cancelled
```

Это два разных набора. Необходимо привести к единому.

**Рекомендация:** Использовать набор из плана автоматизации — он более детальный и отражает реальный бизнес-процесс. Обновить план бэкенда (Фаза 1.1.2).

### 7.2 СУЩЕСТВЕННО: Нет авторизации переходов

**Проблема:** В state machine описаны допустимые переходы, но не указано, кто может выполнять какой переход.

**Рекомендация:**
```typescript
const TRANSITION_PERMISSIONS: Record<string, Role[]> = {
  'draft→documents_pending':      ['manager', 'admin', 'system'],
  'documents_pending→documents_review': ['client', 'manager', 'admin'],
  'documents_review→documents_pending': ['manager', 'admin'],
  'documents_review→payment_pending':   ['manager', 'admin'],
  'payment_pending→in_progress':        ['system'],  // только после оплаты
  'in_progress→approved':               ['manager', 'admin'],
  'in_progress→rejected':               ['manager', 'admin'],
  '*→cancelled':                        ['client', 'manager', 'admin'],
};
```

### 7.3 РЕКОМЕНДАЦИЯ: Хранение истории переходов

**Проблема:** В `activity_log` хранится действие, но нет явной привязки к переходу (from_status → to_status).

**Рекомендация:** В `activity_log.details` (JSONB) обязательно включать:
```json
{
  "from_status": "documents_review",
  "to_status": "payment_pending",
```

## 8. Вебхуки Битрикс24

### 8.1 СУЩЕСТВЕННО: Слабая валидация входящих вебхуков

**Текущее состояние:** "Проверка member_id или секретного токена в URL"

**Проблема:** `member_id` — не секрет, он известен всем приложениям портала. Токен в URL — лучше, но его можно перехватить из логов.

**Рекомендация:**
1. Использовать **секретный токен в URL** (минимум 32 символа): `/api/webhooks/bitrix24?token=RANDOM_SECRET`
2. Дополнительно: **IP whitelist** (если возможно, Битрикс24 Cloud использует динамические IP, но можно ограничить диапазоном)
3. **Проверка структуры данных**: Zod-валидация тела вебхука
4. **Логирование**: все входящие вебхуки в отдельную таблицу `webhook_log` для аудита и дебага

### 8.2 РЕКОМЕНДАЦИЯ: Двусторонняя синхронизация — race condition

**Проблема:** Двусторонняя синхронизация статусов (наша система <-> Битрикс24) создаёт цикл: мы меняем статус → шлём в Битрикс → Битрикс шлёт вебхук обратно → мы снова обрабатываем.

**Рекомендация:**
1. Добавить флаг `sync_source` к каждому изменению статуса: `internal` | `bitrix`
2. При обработке вебхука Битрикс24: если статус уже совпадает — игнорировать
3. В `activity_log` фиксировать источник изменения
4. Добавить дедупликацию по `bitrix_event_id` (если Битрикс предоставляет)

## 9. Cron-задачи

### 9.1 СУЩЕСТВЕННО: X-Cron-Secret — слабая защита

**Текущее состояние:** "Защита: секретный заголовок X-Cron-Secret"

**Проблема:** Заголовок передаётся в crontab в открытом виде. Любой пользователь VPS может увидеть его через `crontab -l` или `/proc`.

**Рекомендация:**
1. Хранить секрет в `.env` файле, а в crontab ссылаться на переменную:
   ```cron
   */5 * * * * . /home/app/.env && curl -s -H "X-Cron-Secret: $CRON_SECRET" http://localhost:3000/api/cron/send-notifications
   ```
2. Дополнительно: проверять, что запрос приходит с `127.0.0.1` (localhost)
3. Рассмотреть переход на BullMQ/Inngest для очередей вместо HTTP cron (более надёжно, с retry и мониторингом)

### 9.2 РЕКОМЕНДАЦИЯ: Мониторинг cron-задач

**Проблема:** В плане нет механизма мониторинга: как узнать, что cron-задача не запустилась или зависла?

**Рекомендация:**
1. Таблица `cron_executions`: `id, job_name, started_at, finished_at, status, processed_count, error`
2. Алерт в Telegram, если задача не выполнялась > 2 интервалов
3. Внешний мониторинг через healthcheck-сервис (Cronitor, Better Uptime, или собственный endpoint `/api/health`)

## 11. Персональные данные (152-ФЗ)

### 11.1 РЕКОМЕНДАЦИЯ: Шифрование чувствительных полей

**Проблема:** В БД хранятся ФИО, телефоны, email, ИНН, сканы паспортов. Plan не упоминает шифрование.

**Рекомендация:**
1. **ИНН** — шифровать на уровне приложения (AES-256-GCM), хранить в зашифрованном виде
2. **Файлы (сканы документов)** — S3 server-side encryption (SSE-S3 или SSE-KMS)
3. **Бэкапы БД** — шифровать при хранении (pgcrypto или encrypted S3 bucket)
4. **Логи** — не записывать полные ПДн (маскировать телефон: +7***456)

### 11.2 РЕКОМЕНДАЦИЯ: Право на удаление данных

**Проблема:** Нет механизма удаления данных клиента по запросу (152-ФЗ, ст. 21).

**Рекомендация:** Добавить в план:
- Endpoint `DELETE /api/users/me/data` — запрос на удаление ПДн
- Автоматическая анонимизация: замена ФИО, email, телефона на placeholder
- Удаление файлов из S3
- Сохранение финансовых записей (для бухгалтерии) в анонимизированном виде
