## 1. Server vs Client Component Boundaries

### Текущее состояние (по коду)

Уже реализованные страницы следуют правильному паттерну: page.tsx остаётся Server Component, интерактивные части выделены в отдельные `*-client.tsx` файлы (например, `faq-client.tsx`, `contact-form.tsx`, `check-status-client.tsx`). Это хорошо.

### Проблемы в планах

**P1 (критическая). RootLayout перегружен клиентскими виджетами.**

Текущий `layout.tsx` рендерит `PopupProvider`, `FloatingActions`, `StickyCTA`, `CookieConsent`, `JivoChat`, `Calltouch` на КАЖДОЙ странице, включая `/dashboard/*`. Это:
- Загружает маркетинговые скрипты в ЛК (бессмысленно).
- Увеличивает bundle для dashboard-страниц.
- `PopupProvider` оборачивает children в контекст, превращая всё дерево в client boundary.

**Рекомендация:** Разделить layout на route groups:
```
app/
  (marketing)/        <-- Header + Footer + PopupProvider + виджеты
    layout.tsx
    page.tsx           <-- /
    services/
    pricing/
    about/
    ...
  (dashboard)/        <-- DashboardShell (без маркетинговых виджетов)
    dashboard/
      layout.tsx
      page.tsx
  (auth)/             <-- Минимальный layout
    login/
    register/
  layout.tsx          <-- Root: только fonts, analytics, html/body
```

**P2 (высокая). Калькулятор на главной не нуждается в client boundary целиком.**

`calculator.tsx` импортируется напрямую в Server Component (page.tsx). Это правильно, если он `'use client'`. Но план предусматривает "общий калькулятор с главной и страницей цен" (пункт 3.2). Калькулятор должен быть единым client-компонентом с пропсами для конфигурации, а не дублироваться.

**P3 (средняя). План ЛК не описывает SC/CC границы явно.**

В плане ЛК (Фазы 4-11) не указано, какие компоненты будут Server Components, а какие Client. Для ЛК критически важно:
- **Server Components:** Layout ЛК, страницы-обёртки (page.tsx), PageHeader, серверный fetch данных.
- **Client Components:** формы (React Hook Form), DataTable (сортировка/фильтрация), FileUploader (drag & drop), Stepper, чат-виджет, колокольчик уведомлений, sidebar (collapse state).

**Рекомендация:** В каждой фазе плана добавить раздел "Границы компонентов":
```
Server: page.tsx (данные), PageHeader, сводка заявки
Client: OrderForm (RHF + Zod), DocumentChecklist (interactive), PaymentButton
```

## 2. Data Fetching Patterns

### Проблемы

**P4 (критическая). Dashboard: один API-запрос для всего экрана.**

План предлагает `GET /api/dashboard/summary` -- один запрос для KPI + заявок + алертов. Это антипаттерн:
- Если одна часть медленная (например, алерты с проверкой дат пропусков), вся страница ждёт.
- Невозможно использовать Suspense streaming.

**Рекомендация:** Разбить на параллельные fetch + Suspense:
```tsx
// app/dashboard/page.tsx (Server Component)
export default function DashboardPage() {
  return (
    <>
      <Suspense fallback={<KpiCardsSkeleton />}>
        <KpiCards />  {/* async SC, свой fetch */}
      </Suspense>
      <Suspense fallback={<OrdersTableSkeleton />}>
        <RecentOrders />  {/* async SC, свой fetch */}
      </Suspense>
      <Suspense fallback={<AlertsSkeleton />}>
        <ActionAlerts />  {/* async SC, свой fetch */}
      </Suspense>
    </>
  );
}
```
Это даёт streaming: быстрые блоки появляются мгновенно, медленные догружаются.

**P5 (высокая). "React Query с staleTime: 30s" для dashboard.**

План предлагает SSR + клиентское обновление через TanStack Query. Для dashboard это избыточно. Лучший паттерн для Next.js 15:
- **Начальная загрузка:** Server Component (RSC) с `fetch()` + `revalidate`.
- **Реактивные обновления:** SSE (уже запланированы в Фазе 10) для push-обновлений статусов.
- **TanStack Query:** оставить только для client-side фильтрации/пагинации (DataTable в Фазе 8).

Не нужно дублировать fetching на клиенте если данные обновляются через SSE.

**P6 (средняя). Отзывы: API Яндекс.Карт / 2ГИС.**

План (5.4) предлагает "живые отзывы" через API. Проблема:
- API Яндекс.Карт не даёт доступ к отзывам (только рейтинг через Places API).
- 2ГИС API сильно ограничен.
- Client-side fetch к внешним API -- медленно и ненадёжно.

**Рекомендация:** Отзывы собирать в собственную БД (ручной import или cron-скрипт). Страница отзывов -- чистый Server Component с `fetch` из своего API + ISR (revalidate: 3600).

**P7 (средняя). Блог: нет стратегии ISR/SSG.**

План блога (Фаза 9) не описывает стратегию генерации. Для 2-3 статей в неделю идеально:
```tsx
// app/blog/[slug]/page.tsx
export const revalidate = 3600; // ISR: обновлять раз в час

export async function generateStaticParams() {
  const posts = await getAllPostSlugs();
  return posts.map(slug => ({ slug }));
}
```
Без этого каждый визит статьи -- серверный рендер, что убьёт TTFB.

## 3. Routing Structure

### Проблемы

**P8 (критическая). URL сайта не совпадают с планом.**

План описывает русские URL (`/uslugi/propusk-mkad/`, `/ceny/`, `/o-kompanii/`), но реальный код использует английские (`/services/`, `/pricing/`, `/about/`). Нужно определиться:
- Русские URL лучше для SEO в Яндексе (целевой рынок).
- Английские проще технически.
- **Текущий код уже работает с английскими URL.** Менять сейчас -- двойная работа + 301 редиректы.

**Рекомендация:** Обновить план, приняв английские URL из текущего кода. Если заказчик настаивает на русских -- делать через `rewrites` в `next.config.js`, а не переименовывать папки.

**P9 (высокая). Нет route group для публичного сайта vs ЛК vs админки.**

Текущая структура:
```
app/
  layout.tsx      <-- Header + Footer + все виджеты (для ВСЕХ страниц!)
  dashboard/
  admin/
  (auth)/
```

Dashboard и admin НЕ должны рендерить публичный Header/Footer. Сейчас `dashboard/layout.tsx` оборачивает в DashboardShell, но Header и Footer из корневого layout всё равно рендерятся.

**Рекомендация:** Route groups (см. P1 выше). Это первоочередная задача.

**P10 (средняя). `/services/[slug]` -- редирект вместо детальных страниц.**

План описывает детальные страницы услуг (2.1-2.10) как ключевые SEO-посадочные. Но текущий код делает `redirect("/services")` для всех slug-ов. Это полностью противоречит плану и убивает SEO-стратегию по кластерам запросов.

**Рекомендация:** Реализовать детальные страницы услуг как приоритет. Каждая страница -- Server Component с уникальным контентом, Schema.org Service, и generateStaticParams.

**P11 (низкая). Дублирование `/partners` и `/partner`.**

В коде есть два маршрута:
- `/partners/` -- публичная страница партнёрской программы
- `/partner/` -- портал партнёра (с авторизацией)

Это может путать пользователей. Рекомендация: `/partners/` для публичной страницы, `/partner/portal/` для ЛК партнёра.

## 4. Code Splitting и Bundle Size

### Проблемы

**P12 (критическая). Маркетинговые виджеты загружаются на всех страницах.**

`JivoChat`, `Calltouch`, `PopupProvider` в root layout -- это JS, который загружается и в ЛК, и в админке, где он не нужен. Помимо route groups (P1), тяжёлые виджеты должны быть dynamic imports:

```tsx
import dynamic from 'next/dynamic';

const JivoChat = dynamic(() => import('@/components/integrations/jivochat'), {
  ssr: false,
});
const Calltouch = dynamic(() => import('@/components/integrations/calltouch'), {
  ssr: false,
});
```

**P13 (высокая). ЛК: тяжёлые компоненты без lazy loading.**

План упоминает lazy loading в Фазе 12, но это нужно закладывать сразу:
- `react-pdf` (превью PDF) -- ~500KB, обязателен dynamic import.
- `react-dropzone` -- не критичен (~10KB), но FileUploader должен быть lazy на страницах где не используется.
- TanStack Table + виртуализация -- dynamic import на страницах списков.
- AI-чат виджет -- обязательно lazy, загружать только при клике на кнопку.

**Рекомендация:** Добавить в каждую фазу раздел "Dynamic Imports":
```
Фаза 6: react-pdf -- dynamic(), react-dropzone -- direct import (маленький)
Фаза 11: Chat widget -- dynamic(), загрузка по клику
```

**P14 (средняя). Нет optimizePackageImports в next.config.**

Для lucide-react (иконки) и других barrel-export библиотек нужно:
```js
experimental: {
  optimizePackageImports: ['lucide-react', '@tanstack/react-table'],
}
```

## 5. Performance (LCP < 2.5s)

### Проблемы

**P15 (критическая). Hero-блок: "фоновое фото/видео" без стратегии оптимизации.**

План (1.1) предлагает фоновое фото/видео грузовика. Это LCP-элемент. Нужно:
- Использовать `<Image priority>` для hero-изображения.
- НЕ использовать видео как фон (CLS + LCP killer). Если нужен динамичный фон -- CSS-градиент + статичное изображение.
- `sizes="100vw"` для полноширинных hero.
- AVIF/WebP через Next.js Image (уже поддерживается).

**P16 (высокая). Яндекс.Карта на странице контактов.**

Embed карты (iframe) -- тяжёлый ресурс (~300-500KB). Должен загружаться через `loading="lazy"` или, лучше, через Intersection Observer: показывать статичную картинку, загружать карту при скролле.

**P17 (высокая). Dashboard LCP.**

Для ЛК план ставит цель LCP < 2.5s. Для этого:
- DashboardShell (sidebar + header) должен рендериться мгновенно (статический RSC).
- KPI-карточки -- первый контент, должны стримиться через Suspense (P4).
- Sidebar навигация -- Server Component (не нуждается в 'use client' для рендера, только для collapse toggle).

**Рекомендация:** Sidebar реализовать как Server Component с client-only toggle:
```tsx
// sidebar.tsx (Server Component)
export function Sidebar() {
  return (
    <aside>
      <SidebarToggle /> {/* 'use client' -- только кнопка */}
      <nav>
        <SidebarLink href="/dashboard" icon={Home} label="Обзор" />
        {/* ... статические ссылки -- Server Component */}
      </nav>
    </aside>
  );
}
```

**P18 (средняя). Нет `loading.tsx` в планах.**

Next.js App Router поддерживает `loading.tsx` для мгновенного показа скелетонов при навигации. План не упоминает их. Нужны как минимум:
- `app/dashboard/loading.tsx` -- скелетон дашборда.
- `app/dashboard/orders/loading.tsx` -- скелетон списка заявок.
- `app/dashboard/orders/[id]/loading.tsx` -- скелетон деталей заявки.
- `app/blog/[slug]/loading.tsx` -- скелетон статьи.

## 6. Form Handling

### Проблемы

**P19 (высокая). Нет стратегии Server Actions.**

Планы описывают формы через React Hook Form + Zod + client-side submit. Это работает, но Next.js 15 App Router предлагает Server Actions как предпочтительный паттерн:

```tsx
// actions/submit-order.ts
'use server';

import { z } from 'zod';

const schema = z.object({ zone: z.enum([...]), ... });

export async function submitOrder(formData: FormData) {
  const parsed = schema.safeParse(Object.fromEntries(formData));
  if (!parsed.success) return { error: parsed.error.flatten() };
  // ... сохранить в БД
  return { success: true };
}
```

**Рекомендация:** Использовать гибридный подход:
- **Простые формы** (заявка на обратный звонок, контакты) -- Server Actions + `useActionState`.
- **Сложные формы** (мульти-шаговый мастер заявки, Фаза 7) -- React Hook Form + Zod + API routes. Server Actions плохо подходят для пошаговых форм с localStorage-сохранением.

**P20 (высокая). Мульти-шаговая форма: localStorage -- ненадёжно.**

План (Фаза 7) предлагает сохранять черновик в `localStorage`. Проблемы:
- Не работает при смене устройства (начал на телефоне, продолжил на ноутбуке).
- Очищается при clear browser data.
- Нет синхронизации.

**Рекомендация:** Серверный черновик (draft):
1. При создании заявки -- сразу создавать запись в БД со статусом `draft`.
2. Автосохранение каждого шага через debounced Server Action или API call.
3. localStorage использовать только как fallback для offline-режима.

**P21 (средняя). "Единый компонент формы" (10.4) не описывает переиспользуемость.**

План говорит "единый компонент формы: имя, телефон, комментарий". Но формы на сайте разные:
- Контакты: имя + телефон + сообщение.
- CTA-блоки: только телефон.
- Калькулятор: результат + телефон.
- Партнёрство: компания + телефон + ИНН.

**Рекомендация:** Создать composable form system:
```tsx
<LeadForm variant="compact">  {/* только телефон */}
<LeadForm variant="standard"> {/* имя + телефон */}
<LeadForm variant="full">     {/* имя + телефон + комментарий */}
```
С общей логикой: маска телефона, валидация, согласие ПД, отправка в CRM, событие метрики.

## 7. State Management

### Проблемы

**P22 (высокая). Нет стратегии глобального state management.**

В плане ЛК нет упоминания о глобальном стейте. Нужен для:
- **Auth state:** текущий пользователь, роль, сессия (Better Auth предоставляет).
- **Notifications:** счётчик непрочитанных (SSE обновления).
- **Draft order:** текущий черновик заявки (между шагами).

**Рекомендация:**
- Auth: React Context через Better Auth `useSession()`.
- Notifications: `useSyncExternalStore` + EventSource (SSE).
- Draft order: Zustand (маленький, ~1KB) или React Context.
- НЕ использовать Redux -- overkill для этого проекта.

**P23 (средняя). Sidebar collapse state.**

Sidebar должен запоминать состояние (развёрнут/свёрнут) между сессиями. Простейший подход:
```tsx
// hooks/use-sidebar.ts
'use client';
export function useSidebar() {
  const [collapsed, setCollapsed] = useState(() =>
    typeof window !== 'undefined'
      ? localStorage.getItem('sidebar-collapsed') === 'true'
      : false
  );
  // ...
}
```
Или cookie-based (работает с SSR -- нет мигания при загрузке).

## 8. Reusable Layout Patterns

### Проблемы

**P24 (высокая). Дублирование layout-паттернов между ЛК, админкой и партнёрским порталом.**

В коде уже есть три shell-компонента:
- `dashboard/_components/dashboard-shell.tsx`
- `admin/_components/admin-shell.tsx`
- `partner/(portal)/_components/partner-shell.tsx`

Все три -- sidebar + header + main content. Должен быть общий `AppShell`:
```tsx
// components/shells/app-shell.tsx
export function AppShell({ sidebar, header, children }) {
  return (
    <div className="flex h-screen">
      {sidebar}
      <div className="flex-1 flex flex-col">
        {header}
        <main className="flex-1 overflow-auto p-6">{children}</main>
      </div>
    </div>
  );
}
```
Dashboard, admin и partner layout-ы используют AppShell с разными sidebar/header.

**P25 (средняя). Breadcrumbs не описаны как система.**

План упоминает breadcrumbs в header ЛК и в SEO (BreadcrumbList schema). Нужен единый компонент:
- Автоматическая генерация из URL segments.
- Возможность переопределить label на уровне page.
- Включение Schema.org BreadcrumbList.
- Работает как Server Component (не нуждается в client state).

## 9. SEO Metadata Strategy

### Проблемы

**P26 (высокая). Нет generateMetadata для динамических страниц.**

План описывает уникальные title/description для каждой страницы, но не упоминает `generateMetadata` -- ключевую функцию App Router для динамического SEO:

```tsx
// app/services/[slug]/page.tsx
export async function generateMetadata({ params }): Promise<Metadata> {
  const { slug } = await params;
  const service = getServiceBySlug(slug);
  return {
    title: service.seoTitle,
    description: service.seoDescription,
    openGraph: { ... },
  };
}
```

Без этого динамические страницы (услуги, блог) будут иметь generic metadata из parent layout.

**P27 (высокая). Schema.org: нет центральной системы.**

План перечисляет 6+ типов разметки (LocalBusiness, Service, FAQPage, Review, Article, PriceSpecification), но не описывает как их генерировать. Текущий код использует `dangerouslySetInnerHTML` для JSON-LD на главной -- это работает, но не масштабируется.

**Рекомендация:** Создать утилиту:
```tsx
// lib/schema.ts
export function generateLocalBusinessSchema(data) { ... }
export function generateServiceSchema(service) { ... }
export function generateFAQSchema(items) { ... }

// Использование в page.tsx:
<script type="application/ld+json"
  dangerouslySetInnerHTML={{ __html: JSON.stringify(generateFAQSchema(faqItems)) }}
/>
```

**P28 (средняя). Metadata template не используется для публичных страниц.**

Dashboard layout уже использует `title.template: "%s -- Инфологистик-24"`. Но root layout не использует template -- каждая публичная страница дублирует "| Инфологистик-24" в title вручную.

**Рекомендация:**
```tsx
// app/(marketing)/layout.tsx
export const metadata: Metadata = {
  title: {
    default: 'Инфологистик-24 -- Пропуска в Москву',
    template: '%s | Инфологистик-24',
  },
  // ...
};
```

**P29 (средняя). robots.txt и sitemap -- нет исключения ЛК.**

Plan не упоминает исключение `/dashboard/*`, `/admin/*`, `/partner/*` из sitemap и robots. Текущий `sitemap.ts` нужно проверить. В `robots.ts` должно быть:
```
User-agent: *
Disallow: /dashboard
Disallow: /admin
Disallow: /partner
Disallow: /api
```

## 10. Дополнительные замечания

**P30. Нет error.tsx для route segments ЛК.**

Plan (12.4) упоминает "Error boundaries на каждом route segment", но не перечисляет их. Минимальный набор:
- `app/error.tsx` (уже есть)
- `app/dashboard/error.tsx`
- `app/dashboard/orders/[id]/error.tsx`
- `app/admin/error.tsx`

**P31. Нет not-found.tsx для динамических маршрутов.**

`app/not-found.tsx` есть, но нужны также:
- `app/blog/[slug]/not-found.tsx` -- "Статья не найдена"
- `app/dashboard/orders/[id]/not-found.tsx` -- "Заявка не найдена"

**P32. Блог: CMS не определена.**

План (9.1.5) предлагает "Strapi / Payload / Sanity". Для текущего стека (Drizzle + PostgreSQL + Coolify):
- **Payload CMS** -- лучший выбор: TypeScript, встраивается в Next.js, работает с PostgreSQL.
- Strapi требует отдельный сервер.
- Sanity -- hosted, стоит денег при масштабе.
