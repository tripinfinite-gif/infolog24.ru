## КРИТИЧЕСКИЕ проблемы (исправить первыми)

### 1. Безопасность платежей
**Блок:** Бэкенд
**Проблема:** Сумма платежа принимается из клиентского запроса — можно подменить. Вебхуки YooKassa используют ИЛИ IP-фильтрацию ИЛИ проверку SDK, а не обе.
**Решение:** Сумму всегда рассчитывать на сервере. Вебхуки: IP-фильтрация + SDK-верификация + перепроверка статуса через API.

### 2. CSRF-защита отсутствует
**Блок:** Бэкенд
**Проблема:** Ни одного упоминания CSRF в плане бэкенда.
**Решение:** Добавить CSRF-токены для всех мутирующих запросов (Better Auth поддерживает из коробки).

### 3. БД без индексов
**Блок:** Бэкенд
**Проблема:** Ни одного индекса в схеме — деградация при росте данных.
**Решение:** Добавить индексы на FK, поля поиска, статусы, даты.

### 4. CSS-переменные в @layer base
**Блок:** Дизайн-система
**Проблема:** Tailwind v4 может удалить переменные при сборке. Это баг #1 по скиллу tailwind-theme-builder.
**Решение:** Вынести `:root` и `.dark` на корневой уровень CSS, вне `@layer`.

### 5. Root layout с маркетинговыми виджетами на всех страницах
**Блок:** Фронтенд
**Проблема:** JivoChat, Calltouch, PopupProvider рендерятся на ВСЕХ страницах — включая ЛК и админку.
**Решение:** Route groups: `(marketing)/`, `(dashboard)/`, `(auth)/` с отдельными layouts.

### 6. Redirect вместо SEO-посадочных
**Блок:** Фронтенд
**Проблема:** Детальные страницы услуг делают redirect — противоречит SEO-стратегии.
**Решение:** Каждая услуга = полноценная посадочная страница с уникальным контентом и мета-тегами.

### 7. Prompt injection в AI-чатботе
**Блок:** AI-чатбот
**Проблема:** Нет защиты от prompt injection — ни в системном промпте, ни на уровне валидации ввода.
**Решение:** Добавить инструкции в промпт + input sanitization + ограничение длины сообщений.

### 8. Противоречие Auth: Better Auth vs Auth.js v5
**Блок:** Бэкенд
**Проблема:** CLAUDE.md указывает Better Auth, план бэкенда — Auth.js v5.
**Решение:** Унифицировать — Better Auth (как указано в архитектурном плане и CLAUDE.md).

## ВЫСОКОПРИОРИТЕТНЫЕ проблемы

### Архитектура
- **Монорепо не используется** — вся логика в `apps/web/`, shared-пакеты пусты. Нужно вынести db, auth, shared-типы в packages.
- **Нет тестов** — ни Vitest, ни задачи test в turbo.json.
- **Нет `.env.example`** — разработчики не знают какие переменные нужны.

### Дизайн-система
- **Нет палитры оттенков** (primary-50..900) — план ссылается на классы, которые не работают.
- **Контраст CTA** — белый на #EA580C = 3.5:1, не проходит WCAG AA. Затемнить до #C2410C.
- **Нет tw-animate-css** — анимации shadcn-компонентов могут не работать.
- **Нет компонента form** — при том что форма заявки ключевой элемент.

### Фронтенд
- **URL расхождение** — план: `/uslugi/`, код: `/services/`. Нужно определиться.
- **Нет loading.tsx, error.tsx, not-found.tsx** для route segments.
- **Нет generateMetadata** для динамических страниц.
- **Hero с видео-фоном** убьёт LCP — использовать оптимизированное изображение.
- **3 дублирующихся shell-компонента** — нужен один AppShell.

### AI-чатбот
- **Нет `convertToModelMessages`** — AI SDK v5 требует конвертации.
- **Rate limiter в памяти** протекает и не работает при масштабировании — нужен Redis.
- **Telegram webhook** без верификации secret token.
- **Бюджет AI занижен** в ~3 раза ($13 vs реалистичные $25-40).

### DevOps
- **Образы ~500 MB** (Nixpacks) — custom Dockerfile уменьшит до ~150 MB.
- **Нет graceful shutdown** — zero-downtime deployment не работает.
- **Бэкапы без верификации** — нет проверки целостности, нет алертов при сбоях.
- **Мониторинг** — HTTP-пинги недостаточно, нужен мониторинг ресурсов VPS.
- **CI пайплайн** ~6 мин → можно ускорить до ~3.5 мин параллелизацией.

## Рекомендуемый порядок исправлений

**Фаза 0 — Синхронизация планов** (перед любой реализацией):
1. Унифицировать Auth → Better Auth везде
2. Унифицировать хостинг → Selectel + Coolify
3. Определить URL-паттерн (рус/англ)
4. Синхронизировать статусы заявки (единый state machine)
5. Унифицировать мониторинг и ротацию бэкапов

**Фаза 1 — Критические исправления:**
1. Безопасность платежей (серверный расчёт суммы)
2. CSRF-защита
3. Индексы в БД
4. CSS-переменные (вынести из @layer base)
5. Route groups для layout

**Фаза 2 — Высокий приоритет:**
1. Наполнить shared-пакеты монорепо
2. SEO-посадочные вместо redirect
3. Prompt injection protection
4. Custom Dockerfile
5. Тестовая инфраструктура (Vitest)
