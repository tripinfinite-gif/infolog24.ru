| Параметр | Значение |
|---|---|
| Заявок в месяц | 600–800 |
| Клиентов в месяц | 150–200 |
| Текущий штат | 30–50 человек |
| Целевой штат | 15–20 человек |

- **VPS (Selectel):** полный контроль, PM2, cron, фоновые процессы
- **Cron-задачи:** через системный crontab (VPS) — дёргает API Routes по расписанию
- **Альтернатива:** Inngest / BullMQ для очередей — если потребуется масштабирование

│  Сайт       │───▸│  Next.js API │───▸│  PostgreSQL       │

                   └──────┬───────┘
                          │
              ┌───────────┼───────────┐
              ▼           ▼           ▼
        ┌──────────┐ ┌────────┐ ┌──────────┐
        │ Telegram │ │ Email  │ │   SMS    │
        │   Bot    │ │(Resend)│ │(SMS.ru)  │

### Ключевые решения

1. **Telegram — основной канал** (бесплатно, мгновенно, 90%+ открываемость в РФ)
2. **Email — дополнительный** (Resend — 3000 писем/мес бесплатно, достаточно)
3. **SMS — критичные уведомления** (SMS.ru — от 1.5₽/SMS, только для результата и истечения)
4. **Битрикс24 — сохраняем** как CRM, синхронизируем статусы через webhook/API

### Фаза 1: Воркфлоу заявки (state machine)
**Статус:** [ ]
**Исполнитель:** Агент
**Зависимость:** Фаза 0

#### Задачи

**1.1. Создать модуль `lib/workflow/stateMachine.ts`**

Конечный автомат статусов заявки (согласован со схемой `orders` из [Архитектура](2026-04-10-архитектура-и-стек.md) § 4.3):

```
draft → documents_pending → documents_review → payment_pending → in_progress → approved
                                    ↓                                           → rejected
                           documents_pending (дослать)                     → cancelled
```

Допустимые переходы (жёсткая валидация):

```typescript
const TRANSITIONS: Record<OrderStatus, OrderStatus[]> = {
  draft:              ['documents_pending', 'cancelled'],
  documents_pending:  ['documents_review', 'cancelled'],
  documents_review:   ['documents_pending', 'payment_pending', 'cancelled'],  // pending если не хватает
  payment_pending:    ['in_progress', 'cancelled'],
  in_progress:        ['approved', 'rejected'],    // подано в Дептранс
  approved:           [],                           // терминальный (пропуск в permits)
  rejected:           [],                           // терминальный
  cancelled:          [],                           // терминальный
};
```

Функция `transitionOrder(orderId, newStatus, metadata)`:
- Проверяет допустимость перехода
- Обновляет `status` и `updated_at` в таблице `orders`
- Записывает действие в `activity_log`
- Создаёт запись в `notifications` с нужным шаблоном
- Возвращает обновлённый заказ или ошибку

**1.2. Расширить API Routes из [Бэкенд и API](2026-04-10-бэкенд-и-api.md) Фаза 3**

Дополнительные endpoints для воркфлоу (к существующим CRUD):

| Route | Метод | Описание |
|---|---|---|
| `/api/orders/[id]/transition` | POST | Сменить статус (с валидацией state machine) |
| `/api/orders/[id]/docs/check` | POST | Запустить проверку комплектности документов |
| `/api/cron/send-notifications` | POST | Очередь отправки (защита: X-Cron-Secret) |
| `/api/cron/check-expiring` | POST | Проверка истекающих пропусков |
| `/api/cron/check-expired` | POST | Автопереход permit → expired |

**1.3. Хуки при смене статуса заявки**

При каждом `transitionOrder()` автоматически:
1. Запись в `activity_log` (кто, когда, что сменил)
2. Создание `notification` для клиента (по матрице из Фазы 4)
3. Синхронизация стадии сделки в Битрикс24
4. Если `approved` → создание записи в `permits`

**Критерий готовности:** API Routes работают, заявку можно создать, получить, сменить статус. Переходы валидируются.

### Фаза 3: Каналы уведомлений
**Статус:** [ ]
**Исполнитель:** Агент
**Зависимость:** Фаза 0

#### 3.1. Telegram-бот

**Задачи:**

1. **Создать бота** через @BotFather → получить токен
2. **Создать модуль `lib/notifications/telegram.ts`**
   - Используем библиотеку `grammy` (лёгкая, TypeScript-first)
   - Бот работает в режиме webhook (не polling — совместимо с Beget)
   - Webhook URL: `https://infolog24.ru/api/telegram/webhook`
3. **Команды бота:**
   - `/start` — привязка клиента (по номеру телефона или коду из заявки)
   - `/status` — текущий статус заявки
   - `/help` — список команд
4. **Привязка клиента:**
   - Клиент нажимает `/start код12345` (код приходит в SMS при создании заявки)
   - Бот сохраняет `telegram_chat_id` в `applications`
   - С этого момента все уведомления идут в Telegram
5. **Отправка уведомлений:**
   - Функция `sendTelegram(chatId, message)` — отправка через Bot API
   - Поддержка Markdown-разметки
   - Retry 3 раза при ошибке

**Оценка:** Бесплатно. Лимит 30 сообщений/сек — более чем достаточно.

#### 3.2. Email (Resend)

**Задачи:**

1. **Зарегистрировать аккаунт** на resend.com
2. **Подтвердить домен** infolog24.ru (DNS-записи: SPF, DKIM, DMARC)
3. **Создать модуль `lib/notifications/email.ts`**
   - Используем `resend` npm-пакет
   - Шаблоны писем: React Email (JSX → HTML)
4. **Шаблоны писем:**
   - Подтверждение заявки
   - Запрос документов (с чек-листом)
   - Статус изменён
   - Пропуск готов
   - Напоминание о продлении
5. **Fallback:** Если Resend недоступен — SMTP через Beget (встроенный, бесплатный)

**Оценка:** Бесплатно до 3000 писем/мес. При 600–800 заявках — хватает с запасом.

#### 3.3. SMS (SMS.ru)

**Задачи:**

1. **Зарегистрировать аккаунт** на sms.ru
2. **Подтвердить отправителя** (имя «Infolog24»)
3. **Создать модуль `lib/notifications/sms.ts`**
   - HTTP API sms.ru (простой REST)
   - Функция `sendSMS(phone, message)`
4. **Использовать только для:**
   - Код привязки Telegram-бота (одноразовый)
   - Пропуск готов/отказ (критичный результат)
   - Пропуск истекает через 7 дней (последнее напоминание)
5. **Ограничение:** Максимум 3 SMS на заявку за весь цикл (экономия)

**Оценка:** ~1.5–2.5₽ за SMS. При 200 клиентах × 3 SMS = 600 SMS/мес ≈ 900–1500₽/мес.

#### 3.4. Единый диспетчер `lib/notifications/dispatcher.ts`

```typescript
async function notify(applicationId: number, type: NotificationType) {
  const app = await getApplication(applicationId);
  const template = await getTemplate(type);

  // Приоритет каналов:
  // 1. Telegram (если привязан) — бесплатно, мгновенно
  // 2. Email (если есть) — бесплатно, надёжно
  // 3. SMS — только для критичных (permit_ready, renewal_7, declined)

  const channels = resolveChannels(app, type);
  for (const channel of channels) {
    const message = renderTemplate(template, app);
    await sendNotification(channel, app, message);
    await logNotification(app.id, channel, type, message);
  }
}
```

**Критерий готовности:** Все три канала работают. Уведомление отправляется по приоритету. Каждая отправка логируется в `notifications`.

### Фаза 5: Cron-задачи (напоминания о продлении и очередь отправки)
**Статус:** [ ]
**Исполнитель:** Агент
**Зависимость:** Фазы 3, 4

#### Задачи

**5.1. Cron-эндпоинт: проверка истекающих пропусков**

Route: `/api/cron/check-expiring`
Защита: секретный заголовок `X-Cron-Secret` (чтобы никто извне не дёргал)
Частота: раз в день (утро, 09:00 МСК)

Логика:
```
1. Найти все applications где status = 'approved'
   AND permit_expires_at - NOW() IN (30 дней, 14 дней, 7 дней)
   AND уведомление такого типа ещё не отправлялось (проверка по notifications)
2. Для каждой — создать notification и отправить
```

**5.2. Cron-эндпоинт: обработка очереди уведомлений**

Route: `/api/cron/send-notifications`
Частота: каждые 5 минут

Логика:
```
1. Найти все notifications где status = 'pending' ORDER BY created_at LIMIT 50
2. Для каждой — отправить через нужный канал
3. Обновить status = 'sent' или 'failed' + записать ошибку
4. Retry: если failed и попыток < 3 — вернуть в pending
```

**5.3. Cron-эндпоинт: автоматический переход в expired**

Route: `/api/cron/check-expired`
Частота: раз в день

Логика:
```
1. Найти все applications где status = 'approved'
   AND permit_expires_at < NOW()
2. Перевести в статус 'expired'
```

**5.4. Настройка cron на VPS (Selectel)**

Через SSH добавить в crontab:
```cron
# Каждые 5 минут — очередь уведомлений
*/5 * * * * curl -s -H "X-Cron-Secret: $SECRET" http://localhost:3000/api/cron/send-notifications

# Каждый день в 09:00 МСК — проверка истекающих пропусков
0 9 * * * curl -s -H "X-Cron-Secret: $SECRET" http://localhost:3000/api/cron/check-expiring

# Каждый день в 01:00 МСК — автопереход в expired
0 1 * * * curl -s -H "X-Cron-Secret: $SECRET" http://localhost:3000/api/cron/check-expired
```

> Используем `localhost:3000` — cron работает на том же VPS, не нужно ходить через интернет.

**Критерий готовности:** Cron-задачи работают, уведомления отправляются по расписанию, expired-заявки обновляются автоматически.

**Статус:** [ ]

#### Задачи

**6.1. Синхронизация заявка → сделка**

При создании заявки на сайте:
1. Создать сделку в Битрикс24 через REST API
2. Сохранить `bitrix_deal_id` в `applications`
3. Маппинг полей: клиент → контакт, заявка → сделка, статус → стадия воронки

**6.2. Синхронизация статусов (двусторонняя)**

- **Сайт → Битрикс:** При смене статуса в нашей системе — обновить стадию сделки
- **Битрикс → Сайт:** Webhook из Битрикс24 при смене стадии менеджером:
  - Route: `/api/bitrix/webhook`
  - Обновить `status` в `applications`
  - Запустить уведомление клиенту

**6.3. Маппинг статусов**

| Наш статус (orders) | Стадия Битрикс24 |
|---|---|
| draft | Новая |
| documents_pending | Ожидание документов |
| documents_review | Проверка документов |
| payment_pending | Ожидание оплаты |
| in_progress | Подано в Дептранс |
| approved | Успешно завершена |

### Фаза 8: Тестирование и запуск
**Статус:** [ ]

- Unit-тесты рендера шаблонов

**8.2. Ручное end-to-end тестирование**

Сценарии:
1. ✅ Заявка с сайта → уведомление менеджеру → проверка документов → запрос недостающих → дослали → подано → одобрено → уведомление клиенту
2. ✅ Отказ → уведомление клиенту + менеджеру
3. ✅ Привязка Telegram-бота → уведомления в Telegram
4. ✅ Напоминание о продлении (30/14/7) → SMS на 7 дней
5. ✅ Автоматический переход в expired

**8.3. Нагрузочное тестирование**

- 50 заявок одновременно (стресс cron-очереди)
- Проверить, что cron-эндпоинт укладывается в 30 секунд (лимит Beget)

**8.4. Мониторинг**

- Логирование ошибок отправки в `notifications.error`
- Ежедневный отчёт в Telegram менеджеру: сколько заявок, сколько уведомлений отправлено/failed
- Алерт если failed > 10% за день

**Критерий готовности:** Все сценарии пройдены. Ошибки отправки < 1%. Cron стабильно работает.

## Риски и митигации

| Риск | Вероятность | Митигация |
|---|---|---|
| Beget не тянет cron каждые 5 мин | Средняя | Увеличить интервал до 15 мин, или перейти на VPS |
| SMS.ru блокирует отправителя | Низкая | Запасной: SMSC.ru или MTS Communicator |
| Битрикс24 API нестабилен | Средняя | Очередь повторных попыток, fallback на ручную синхронизацию |
| Менеджеры не используют новую систему | Высокая | Обучение + постепенный переход, не отключать старый процесс сразу |
| Неправильные чеклисты документов | Средняя | Валидация списков с менеджерами перед запуском |
